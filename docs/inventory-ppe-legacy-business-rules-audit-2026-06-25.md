# СИЗ: аудит старого проекта и правила переноса

Дата: 2026-06-25

## 1. Краткий вывод

Старый проект полезен как источник бизнес-правил, но не как источник кода. Главный переносимый поток:

```text
норма -> номенклатура -> факт выдачи -> печать карточки -> лист подписи -> история
```

В новом Patrol360 уже есть большинство нужных полей: нормативное имя, фактическое имя номенклатуры, бренд, модель, артикул, класс защиты, дата выдачи, история строк. Основной риск был в fallback-логике: при печати нормативное имя могло подменяться складским названием, категорией или комментарием.

Канон:

- личная карточка печатает нормы;
- лист подписи печатает фактическую выдачу;
- модель, марка, артикул и класс защиты печатаются отдельно от нормативного имени;
- категории вроде `Каски`, `Одежда`, `Обувь`, `Брюки` не могут заменять строку нормы.

## 2. Карта старого проекта

| Блок | Файл/путь | Что делает | Можно ли использовать |
| --- | --- | --- | --- |
| Сотрудники | `inventory_app/app/data/models.py` | Хранит сотрудника, подразделение, должность, статус | Использовать правила snapshot сотрудника |
| Номенклатура | `inventory_app/app/data/models.py`, migrations `20260516_0032_*` | Хранит `norm_item_name`, `actual_item_name`, `brand_name`, `model_name`, `article`, `protection_class` | Использовать разделение нормы и фактического изделия |
| Нормы | `PositionNorm`, `ppe_api.py` | Связывает должность с позицией СИЗ, количеством и сроком носки | Использовать сценарий подбора нормы по должности |
| Выдача | `PPECardLine`, `ppe_api.py`, `custody_api.py` | Фиксирует строку выдачи, дату, количество, статус | Использовать как модель факта выдачи |
| История | `PPECardLineEvent`, `history_api.py` | Хранит события по строкам СИЗ | Использовать правило: выдача/возврат/списание пишутся в историю |
| Печать карточки | `ppe_print_api.py` | Печатает личную карточку из строк нормы | Использовать правила, не копировать код |
| Лист подписи | `ppe_print_api.py`, `custody_print_api.py` | Формирует строки подписи из факта выдачи | Использовать сортировку по дате выдачи и модель отдельно |

## 3. Карта нового проекта

| Блок | Файл/путь | Что сейчас делает | Проблема |
| --- | --- | --- | --- |
| Contracts | `libs/contracts/InventoryContracts.cs` | DTO для карточек, строк, номенклатуры | Поля есть, но печатный канон нужно закрепить |
| Entities | `libs/infrastructure/Persistence/Entities/InventoryWorkflowEntities.cs` | Хранит PPE карточки, строки и события | Нет отдельного `BrandModelArticle`, пока собирается из item |
| Workflow | `libs/infrastructure/Persistence/EfInventoryWorkflowService.cs` | Создает/обновляет строки PPE | Fallback печатного имени и пункта нормы был смешан с item/comment |
| Export | `libs/infrastructure/Persistence/EfInventoryExportService.cs` | DOCX/PDF/export печатных форм | Нужна единая print-line логика и validation gate |
| Web preview | `apps/web/src/features/inventory/ppe/ppePrint.tsx` | Предпросмотр и popup печати | Должен использовать те же правила, что export |
| Web wizard | `apps/web/src/features/inventory/ppe/ppeCommon.tsx` | Собирает строки предпросмотра из wizard/detail | Fallback должен предпочитать norm item name |

## 4. Таблица сравнения

| Блок | Старый проект | Новый проект | Решение |
| --- | --- | --- | --- |
| Сотрудники | Snapshot сотрудника в карточке | Employee DTO + PPE card detail | Оставить новую модель, подтягивать snapshot в печать |
| Номенклатура | Норма и фактическое изделие разделены полями | Аналогичные поля есть в `InventoryItemDto` | Использовать `NormItemName` для нормы, brand/model/article для второй колонки |
| Нормы | Строка нормы идет от должности | Position norm DTO ограничен | Пока хранить `PrintItemName`, позже добавить норму как отдельную сущность |
| Выдача | PPECardLine фиксирует факт | InventoryPpeCardLineEntity | Оставить, усилить правила печати и истории |
| Возврат/списание | События и статусы строк | Статусы строк + events | Проверить отдельными тестами |
| Личная карточка | Печатает норму | Может печатать `PrintItemName`/item | Закрепить fallback: `PrintItemName -> Item.NormItemName -> Item.Name` |
| Лист подписи | Печатает факт выдачи | Печатает открытые выданные строки | Сортировать по `IssuedAt`, модель отдельно |
| История | PPECardLineEvent | InventoryPpeCardLineEventEntity | Использовать существующую историю |
| UI | Старые формы и стили | Новый wizard/preview | Не переносить UI, перенести сценарии |

## 5. Что переносим

| Логика | Откуда | Куда | Как переносить |
| --- | --- | --- | --- |
| Разделение нормы и факта | `Item.norm_item_name`, `actual_item_name` | `InventoryItemDto`, PPE line print | Через fallback и validation, без копирования UI |
| Модель/марка/артикул отдельно | `brand_name`, `model_name`, `article`, `protection_class` | `ModelDescription` / sheet column 2 | Собирать из полей item |
| Лист подписи по факту выдачи | `ppe_signature_print_lines` | Export + web preview | Фильтр открытых строк, сортировка по дате выдачи |
| История событий | `PPECardLineEvent` | `InventoryPpeCardLineEventEntity` | Не терять события при выдаче/возврате/списании |

## 6. Что не переносим

| Элемент | Почему не переносим |
| --- | --- |
| Старые UI-компоненты | Новая архитектура и дизайн уже другие |
| Старые стили печати без адаптации | В новом проекте есть Word-шаблоны и единый preview |
| Mojibake labels из старого проекта | Это дефект кодировки, не бизнес-правило |
| Старые прямые API-формы | В новом проекте свои REST/DTO и workflow-сервисы |

## 7. Что исправляем в новом проекте

| Ошибка | Как исправить | Приоритет |
| --- | --- | --- |
| Складское имя может попасть вместо нормы | Fallback на `Item.NormItemName` перед `Item.Name` | P0 |
| Комментарий может попасть как пункт нормы | Не использовать comment как `NormPoint` fallback | P0 |
| Модель может смешаться с первой колонкой | Валидация и отдельный `brandModelArticle` | P1 |
| Категория может стать печатным именем | Validation gate generic category names | P1 |
| Web preview и DOCX могут расходиться | Общие правила в `ppeCommon`/export | P1 |

## 8. План реализации

| Этап | Задача | Результат | Приоритет |
| --- | --- | --- | --- |
| P0 | Аудит и канон печати | Зафиксированы правила | Высокий |
| P1 | Быстрый фикс печати | Карточка/лист не смешивают нормы и факт | Высокий |
| P2 | Данные | Nullable поля для norm link/brand dictionary при необходимости | Средний |
| P3 | UI | Два уровня: норма и номенклатура | Средний |
| P4 | Импорт/сопоставление | Подсказки модели/марки с ручным подтверждением | Средний |
| P5 | Тесты | Fixture tests по карточке, листу, истории | Высокий |

## 9. Тесты

| Тест | Что проверяет |
| --- | --- |
| Каска не печатается как категория | Нормативное имя не заменено категорией |
| Костюм/спецодежда полным нормативным именем | Первая колонка карточки корректна |
| Модель/марка/артикул во второй колонке | Лист подписи отделяет факт от нормы |
| Разные даты выдачи | Лист подписи сортируется по выдаче |
| Возврат/списание | Закрытые строки не попадают в лист подписи |
| Web preview vs DOCX | Одинаковые строки и порядок |

## 10. Риски

| Риск | Как снизить |
| --- | --- |
| Старые данные без `NormItemName` | Сохранять fallback на `Item.Name`, но подсвечивать validation |
| Generic category действительно нужна как имя | Разрешать только после явного `PrintItemName`, позже добавить справочник норм |
| Нужна миграция под `BrandModelArticle` | Делать отдельным P2 проходом с nullable полем |
| Preview и export снова разойдутся | Перевести оба на один print DTO/API |
