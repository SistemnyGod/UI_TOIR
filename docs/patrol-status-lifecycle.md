# Patrol 360 Status Lifecycle

Дата: 2026-06-25

Документ фиксирует единый lifecycle для модуля "Обходы / Patrol 360": заявки, назначения, точки обхода и мобильная синхронизация.

## Принцип совместимости

`AssignmentStatusValues` сейчас хранит persisted DB-значения на русском языке. Их нельзя переименовывать как UI-текст без миграции, backfill и проверки старого APK. Поэтому:

- backend продолжает использовать текущие persisted values;
- web/mobile показывают человекочитаемые labels через map;
- новые сценарии используют существующий `/mobile/outbox` и сохраняют legacy `takePatrolRequest`.

## Assignment Statuses

| Константа | Persisted value | UI label | Значение |
| --- | --- | --- | --- |
| `Assigned` | `Назначена` | `Назначено` | Маршрут/заявка закреплены или готовы к назначению. |
| `Waiting` | `Ожидает` | `Ожидает принятия` | Назначение отправлено сотруднику, ожидается принятие. |
| `Accepted` | `Принята` | `Принято` | Сотрудник принял заявку, обход еще не начат. |
| `InProgress` | `В пути` | `В работе` | Обход начат и выполняется. |
| `Paused` | `Приостановлена` | `Приостановлено` | Обход начат, но поставлен на паузу. |
| `Completed` | `Завершено` | `Завершено` | Результат принят сервером или assignment закрыт. |
| `Cancelled` | `Отменено` | `Отменено` | Заявка/назначение отменены. |
| `NeedsDispatcherDecision` | `Требует решения диспетчера` | `Требует решения диспетчера` | Нужна ручная проверка диспетчером. |

## Mobile Local Statuses

| Mobile status | Значение | Активная работа |
| --- | --- | --- |
| `available` | Заявка доступна в списке. | Нет |
| `assigned` | Заявка назначена сотруднику. | Нет |
| `accepted` | Заявка принята, но обход не начат. | Да |
| `inProgress` | Обход выполняется. | Да |
| `paused` | Обход поставлен на паузу. | Да |
| `completedLocal` | Отчет собран локально и ждет синхронизации. | Да, пока не получено терминальное состояние |
| `syncing` | Отчет или команда отправляются. | Да |
| `completedServer` | Сервер подтвердил отчет. | Нет |
| `cancelledServer` | Сервер отменил заявку до локального прогресса. | Нет |
| `syncError` | Отправка не удалась, нужен повтор. | Да |
| `authRequired` | Нужен повторный вход, outbox сохранен. | Да |
| `needsDispatcherDecision` | Нужна ручная обработка диспетчером. | Нет для сотрудника, да для web-мониторинга |

## Point Result Statuses

| Status | UI label | Закрывает обязательную точку | Комментарий |
| --- | --- | --- | --- |
| `pending` | Не заполнено | Нет | Блокирует отправку. |
| `scanned` | Отсканировано | Нет | Нужен итоговый статус. |
| `ok` | Исправно | Да | Комментарий и вложения пока необязательны. |
| `issue` | Неисправно | Да | Должно попадать в отчет как замечание. |
| `skipped` | Метка недоступна | Да | Аварийное закрытие точки, всегда видно в web-отчете. |
| `deferred` | Отложено | Нет | Для обязательной точки блокирует обычную отправку. |

## Sync Statuses

| Status | Значение | Правило |
| --- | --- | --- |
| `pending` | Команда ждет отправки. | Данные не удалять. |
| `sending` | Команда отправляется. | Через recovery возвращать зависшие `sending` в retry. |
| `retryLater` | Временная ошибка сети/сервера. | Повторять по backoff и показывать в очереди. |
| `accepted` | Сервер принял команду. | Закрывать локальное состояние. |
| `duplicate` | Сервер уже принял тот же `clientOperationId`. | Считать успехом, дубль не создавать. |
| `conflict` | Серверное состояние конфликтует с локальным. | Показывать понятный статус и не терять данные. |
| `rejected` | Команда окончательно отклонена. | Показывать причину, не скрывать локальный след. |

## Разрешенные переходы

Нормальный мобильный lifecycle:

```text
available / assigned
-> accepted
-> inProgress
-> paused
-> inProgress
-> completedLocal
-> syncing
-> completedServer
```

Возврат до старта:

```text
accepted
-> release
-> assigned / available
```

Legacy APK:

```text
available / assigned
-> takePatrolRequest
-> inProgress
```

Пауза:

```text
inProgress
-> pause
-> paused
-> resume
-> inProgress
```

Правило: у сотрудника может быть только один `inProgress`. Остальные активные обходы должны быть `accepted`, `paused`, `completedLocal`, `syncing` или терминальными.

Передача диспетчеру:

```text
inProgress / paused
-> handoff
-> needsDispatcherDecision
```

## Offline Completed Then Server Cancelled

Бизнес-правило: если offline-обход полностью пройден, а затем после восстановления связи пришла отмена с сервера, активная задача не должна вечно висеть у сотрудника конфликтом.

Целевое поведение:

1. Мобильное приложение сохраняет локальный отчет и outbox-команду.
2. При sync сервер либо принимает отчет как поздний результат отмененной заявки, либо фиксирует отмененный/неполный результат в истории.
3. Мобильная активная задача получает терминальный статус:
   - `completedServer`, если сервер принял результат;
   - `cancelledServer`, если сервер зафиксировал отмену без принятия результата;
   - `syncError`, если нужна повторная попытка из-за сети/сервера.
4. После терминального статуса задача исчезает из активной работы сотрудника и остается доступной в истории/очереди с понятным объяснением.

Запрещено:

- молча удалять локальный отчет до серверного подтверждения;
- оставлять `needsDispatcherDecision` как вечный статус на телефоне сотрудника;
- создавать дубль результата при повторной отправке.

## Что логировать

| Событие | Обязательные поля |
| --- | --- |
| Принятие заявки | employeeId, assignmentId/requestId, clientOperationId, timestamp |
| Старт/пауза/продолжение | employeeId, assignmentId, oldStatus, newStatus, clientOperationId |
| Заполнение точки | assignmentId, pointId, status, confirmationType, hasFiles, timestamp |
| Отправка отчета | assignmentId, clientOperationId, pointCount, fileCount, syncStatus |
| Повторная отправка | clientOperationId, previousStatus, attemptCount, lastError |
| Дубль на сервере | clientOperationId, existingResultId, payloadHash/status |
| Отмена с сервера | assignmentId/requestId, localStatus, terminalStatus |
| Ошибка авторизации | accountId/userId, hasPendingOutbox, status `authRequired` |

## Acceptance

- Статусы отображаются одинаково в backend/web/mobile через label map.
- Старый APK с `takePatrolRequest` продолжает работать.
- `ok`, `issue`, `skipped` закрывают обязательную точку.
- `pending` и `deferred` блокируют отчет по обязательной точке.
- Повторная отправка одного отчета по `clientOperationId` не создает дубль.
- Offline-отчет не теряется и не остается вечным активным конфликтом после серверной отмены.
