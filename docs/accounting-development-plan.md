# План разработки модуля Бухгалтерия

Дата создания: 2026-05-30  
Статус: рабочий план для разработки  
Основной фокус: СИЗ, номенклатура, сотрудники, документы, история движения

## Цель

Сделать модуль Бухгалтерия удобным контуром учета ответственности, а не складским учетом.

Главная логика:

- номенклатура хранит предметы, оборудование и спецодежду;
- СИЗ берет предметы из номенклатуры;
- выдача фиксирует сотрудника, предмет, дату, срок, цену, сумму и статус;
- по сотруднику видно, что ему выдали, что на руках, что он вернул и на какую сумму;
- по предмету видно, кому и когда он выдавался;
- документы формируются автоматически из выбранных позиций.

## Архитектурное решение

СИЗ ведется как no-stock ledger.

Это означает:

- склад и остатки для СИЗ не обязательны;
- выдача не должна падать из-за отсутствия склада;
- возврат не должен требовать склад;
- основная единица учета - строка выдачи СИЗ сотруднику;
- история движения строится по строкам выдачи и событиям.

## Нормальная бизнес-логика СИЗ

Этот раздел фиксирует целевое поведение СИЗ перед разработкой. Если код или UI противоречит этим правилам, исправляем код и UI под эту модель.

### Главный принцип

СИЗ - это учет выдачи и ответственности сотрудника, а не склад.

Система должна отвечать на вопросы:

- кому выдали;
- что выдали;
- когда выдали;
- на какой срок;
- по какой цене;
- на какую сумму;
- что сейчас на руках;
- что вернули;
- что списали;
- где история движения.

Система не должна требовать склад для выдачи, возврата или списания СИЗ.

### Жизненный цикл строки СИЗ

Каждая строка СИЗ - это отдельная позиция, выбранная из номенклатуры и привязанная к сотруднику.

| Статус | Значение | Можно редактировать | Следующие действия |
|---|---|---|---|
| `draft` | Позиция добавлена в карточку, но еще не выдана | Да | выдать, удалить |
| `issuing` | Подготовлена к выдаче, документы можно печатать | Да | подтвердить выдачу, отменить |
| `issued` | Предмет выдан сотруднику и находится на руках | Ограниченно | вернуть, списать, продлить срок, изменить комментарий |
| `returned` | Предмет возвращен сотрудником | Нет, только комментарий/история | открыть историю, печать |
| `written_off` | Предмет списан | Нет, только комментарий/история | открыть историю, печать |
| `overdue` | Срок носки истек, предмет еще на руках | Ограниченно | вернуть, списать, продлить срок |

`overdue` может быть вычисляемым состоянием поверх `issued`, если дата окончания срока меньше текущей даты.

### Переходы статусов

Разрешенные переходы:

- `draft` -> `issuing`;
- `draft` -> `issued`;
- `issuing` -> `issued`;
- `issuing` -> `draft`;
- `issued` -> `returned`;
- `issued` -> `written_off`;
- `issued` -> `issued` при изменении срока, цены, комментария;
- `overdue` -> `returned`;
- `overdue` -> `written_off`;
- `overdue` -> `issued` при продлении срока.

Запрещенные переходы:

- `returned` -> `issued` без создания новой строки;
- `written_off` -> `issued`;
- `returned` -> `written_off`;
- удаление `issued`, `returned`, `written_off` без архивного следа;
- изменение предмета в уже выданной строке.

Если ошиблись предметом после выдачи:

1. Старую строку нужно вернуть или списать с комментарием.
2. Создать новую строку с правильным предметом.

### Цена, количество и сумма

Цена должна храниться в строке выдачи, а не только в номенклатуре.

Правила:

- цена из номенклатуры подставляется по умолчанию;
- оператор может изменить цену перед выдачей;
- после выдачи строка хранит свою цену независимо от будущих изменений номенклатуры;
- сумма строки = количество * цена строки;
- если цена равна 0, строка сохраняется, но помечается как требующая сверки;
- документы и отчеты используют цену строки, а не текущую цену предмета.

Нельзя:

- пересчитывать старые выдачи по новой цене из номенклатуры;
- молча заменять нулевую цену на случайную;
- скрывать строки с нулевой ценой из отчетов.

### Срок носки

Срок носки также должен храниться в строке выдачи.

Правила:

- срок из номенклатуры подставляется по умолчанию;
- оператор может изменить срок перед выдачей;
- дата окончания = дата выдачи + срок;
- если срок изменили после выдачи, нужно создать событие истории;
- просрочка считается по дате окончания.

Минимальный набор полей строки:

- `issued_at`;
- `due_at`;
- `life_months`;
- `unit_price_minor`;
- `quantity`;
- `amount_minor` как вычисляемое значение или расчет в запросах.

### История движения

Каждое значимое действие должно создавать событие истории.

События:

- `line_added` - строка добавлена;
- `line_updated` - изменены цена, срок, количество или комментарий;
- `issued` - предмет выдан;
- `returned` - предмет возвращен;
- `written_off` - предмет списан;
- `due_date_changed` - изменен срок;
- `price_changed` - изменена цена;
- `comment_added` - добавлен комментарий;
- `line_archived` - строка архивирована до выдачи.

В событии хранить:

- строку СИЗ;
- старый статус;
- новый статус;
- старую цену, если менялась;
- новую цену, если менялась;
- старый срок, если менялся;
- новый срок, если менялся;
- комментарий;
- автора;
- дату события.

### История по сотруднику

Карточка сотрудника должна показывать:

- активные предметы на руках;
- просроченные предметы;
- возвращенные предметы;
- списанные предметы;
- сумму активной ответственности;
- сумму всех выдач за период;
- историю событий.

Фильтры:

- период;
- статус;
- предмет;
- только просроченные;
- только без цены.

### История по предмету номенклатуры

Карточка предмета должна показывать:

- кому выдавался предмет;
- дата выдачи;
- статус;
- цена выдачи;
- сумма;
- дата возврата или списания;
- количество выдач;
- общая сумма выдач.

Это важно, чтобы быстро понять движение конкретной спецодежды или оборудования.

### Создание карточки СИЗ

Правильный сценарий:

1. Оператор выбирает сотрудника.
2. Система показывает должность, подразделение, табельный номер.
3. Оператор выбирает позиции из СИЗ-номенклатуры.
4. Система подставляет цену и срок.
5. Оператор при необходимости меняет цену, срок, количество.
6. Система показывает итоговую сумму.
7. Оператор сохраняет карточку или сразу подтверждает выдачу.
8. Система создает строки и события истории.
9. Система дает распечатать личную карточку и роспись получения.

### Редактирование карточки СИЗ

До выдачи можно:

- менять сотрудника;
- менять список предметов;
- менять количество;
- менять цену;
- менять срок;
- менять комментарий;
- удалять строки.

После выдачи нельзя:

- менять сотрудника;
- менять предмет в строке;
- удалять строку без истории.

После выдачи можно:

- вернуть;
- списать;
- продлить срок;
- исправить цену с событием истории;
- добавить комментарий.

### Возврат

Возврат должен работать без склада.

При возврате:

- статус строки становится `returned`;
- фиксируется дата возврата;
- создается событие истории;
- строка исчезает из активной ответственности сотрудника;
- строка остается в истории сотрудника и предмета.

Дополнительно можно хранить:

- состояние при возврате;
- комментарий;
- кто принял возврат.

### Списание

Списание также работает без склада.

При списании:

- статус строки становится `written_off`;
- фиксируется дата списания;
- создается событие истории;
- строка исчезает из активной ответственности;
- строка остается в истории и отчетах.

Списание требует комментарий.

### Печать документов

Документы должны печататься из состояния карточки и строк выдачи.

Личная карточка:

- включает все выбранные/выданные позиции;
- берет цену из строки;
- берет срок из строки;
- показывает сотрудника, должность, подразделение.

Роспись получения:

- включает только позиции, которые сотрудник получает;
- используется как документ под подпись;
- не должна включать старые возвращенные или списанные строки, если оператор печатает выдачу новой партии.

### Ошибки и проверки

Система должна блокировать сохранение, если:

- не выбран сотрудник;
- нет ни одной строки;
- количество меньше или равно 0;
- предмет не найден;
- предмет не относится к СИЗ, если операция именно СИЗ;
- дата выдачи некорректна;
- срок некорректен;
- попытка вернуть уже возвращенную строку;
- попытка списать уже возвращенную строку;
- попытка изменить предмет в выданной строке.

Система должна предупреждать, но не блокировать:

- цена равна 0;
- срок носки не указан;
- у сотрудника уже есть похожий активный предмет;
- документ печатается по строкам с нулевой ценой.

### Отчеты

Минимальные отчеты для первого рабочего контура:

- СИЗ на руках;
- СИЗ без цены;
- просроченные СИЗ;
- выдачи за период;
- возвраты за период;
- списания за период;
- сумма ответственности по сотрудникам;
- движение по предмету.

## Acceptance criteria

- сотрудник выбирается из реального списка;
