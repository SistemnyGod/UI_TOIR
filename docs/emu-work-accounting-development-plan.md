# План доработки ЭМУ: Учет работ

Дата: 2026-06-01  
Обновлено: 2026-06-01  
Статус: рабочий план развития  
Область: `ЭМУ -> Учет работ`, web/API/DB/worker  
Важно: мобильное приложение этим планом не затрагиваем.

## Цель

Сделать `Учет работ` рабочим инструментом для диспетчера ЭМУ: быстро отправить сотрудника в работу, видеть живую картину по активным карточкам, корректно считать время, фиксировать паузы/ожидание, закрывать результат и получать понятную историю без ручной сверки.

## Главные направления

### 1. Карточка работы

Приоритет: P1

Что нужно улучшить:

- Создание карточки через отдельную модалку, а не перегруженную форму на экране.
- Выбор сотрудников только из избранных ЭМУ по умолчанию, с быстрым поиском по общей базе для добавления в избранное.
- Ясное разделение времени:
  - `Время прихода`;
  - `Время начала работы`;
  - `Пауза / ожидание`;
  - `Время завершения`.
- Для ручной корректировки времени обязательно хранить:
  - кто изменил;
  - когда изменил;
  - старое значение;
  - новое значение;
  - комментарий причины.
- В карточке показывать live-счетчики:
  - общее время;
  - рабочее время;
  - ожидание;
  - простой;
  - количество сотрудников.

Следующий шаг:

- Переработать UI карточки и модалку создания.
- Добавить audit-событие на каждую ручную корректировку времени.

Проверка:

- Создать работу, открыть карточку, увидеть live-счетчик без перезагрузки.
- Изменить время вручную и проверить запись аудита в БД.

### 2. Паузы и ожидание

Приоритет: P1

Что нужно улучшить:

- Запретить некорректную хронологию:
  - пауза раньше прихода;
  - возврат раньше начала паузы;
  - завершение раньше прихода;
  - пауза после завершения;
  - две открытые паузы у одного сотрудника.
- Разделить типы пауз:
  - ожидание оборудования;
  - ожидание допуска;
  - ожидание материалов;
  - перерыв;
  - прочее.
- В карточке показывать не только суммарное ожидание, но и расшифровку по причинам.
- Для групповой работы разрешить паузу:
  - всей карточке;
  - отдельному сотруднику.

Следующий шаг:

- Уточнить backend-валидацию в `EfEmuService.cs`.
- Расширить UI модалку паузы выбором сотрудников и причины.

Проверка:

- DB integration: pause -> resume -> pause -> complete.
- Negative tests: pause before arrival, resume without active pause, complete during open pause.

### 3. Завершение работы

Приоритет: P1

Что нужно улучшить:

- Завершение должно требовать:
  - результат;
  - статус результата;
  - время завершения;
  - комментарий при `Не выполнено` или `Частично выполнено`.
- Статусы сделать едиными:
  - `В работе`;
  - `Ожидание`;
  - `Выполнено`;
  - `Частично выполнено`;
  - `Не выполнено`;
  - `Отменено`;
  - `Удалено`.
- Не смешивать операционный статус карточки и результат выполнения.
- После завершения карточка должна уходить из активной доски и попадать в историю.

Следующий шаг:

- Нормализовать contract/status mapping в `EmuContracts.cs`, `EfEmuService.cs`, `EmuCompletedWorkHistoryScreen.tsx`.

Проверка:

- Создать работу, завершить как `Выполнено`, проверить доску и историю.
- Завершить как `Не выполнено` без комментария: API должен вернуть validation error.

## Утвержденный порядок разработки

Этот порядок принят как основной roadmap для реализации `ЭМУ -> Учет работ`. Работы выполняем сверху вниз: сначала надежность учета времени и статусов, затем удобство оператора, затем история и только после этого планирование, уведомления и аналитика.

### Пакет 1: стабилизировать ядро

Цель: карточка работы должна корректно считать время, не принимать невозможные действия и оставлять проверяемый след изменений.

Что входит:

- Нормализовать статусы карточки и результата: `В работе`, `Ожидание`, `Завершено`, `Удалено` отдельно от результата `Выполнено`, `Частично выполнено`, `Не выполнено`, `Отменено`.
- Усилить backend-валидацию времени: запретить паузу раньше прихода, возврат раньше паузы, завершение раньше прихода, операции после завершения, две открытые паузы у одного сотрудника.
- Уточнить правило завершения при открытой паузе: либо блокировать завершение с понятной ошибкой, либо явно закрывать паузу через отдельное подтверждение.
- Хранить аудит ручных корректировок времени: пользователь, старое значение, новое значение, причина, дата изменения.
- Проверить расчет `workMinutes`, `waitingMinutes`, `otherWorkMinutes` для активной карточки, карточки на паузе и завершенной карточки.
- Добавить API-mode smoke: create -> pause -> resume -> complete -> history -> cleanup.

## Критерии готовности

- Диспетчер может создать работу за 15-30 секунд без лишних полей.
- Карточка в реальном времени показывает рабочее время и ожидание.
- Нельзя сохранить невозможную хронологию.
- Завершенная работа сразу появляется в истории.
- История и экспорт совпадают по фильтрам.
- Все ручные корректировки времени видны в аудите.
- Учет работ не зависит от mock/localStorage.
- Мобильное приложение не меняется.

## Улучшения v2: Дашборд / История / Учет работ

Главная цель v2: сделать ЭМУ не набором отдельных экранов, а рабочим центром диспетчера и руководителя. Диспетчер должен быстро видеть текущую занятость, проблемные работы и доступных сотрудников. Руководитель должен видеть потери времени, план-факт и проблемные участки.

### Дашборд ЭМУ

Оперативная сводка `что происходит сейчас`:

- В работе сейчас.
- В ожидании сейчас.
- Просроченные и забытые работы.
- Сотрудники свободны, заняты, в ожидании.
- Работы без результата или с долгим ожиданием.

Проблемные зоны:

- Топ участков по ожиданию.
- Топ причин ожидания.
- Работы дольше нормы.
- Сотрудники с перегрузкой.

### Мини-панель занятости сотрудников ЭМУ

Назначение:

- Быстро показать, кто сейчас свободен, кто в работе, кто ожидает и где есть конфликт.
- Использовать как основной быстрый выбор сотрудника для новой работы.

Где показывать:

- В `Учет работ` справа, вместо пустого пространства или как правую панель.
- Если выбрана карточка, правая панель показывает детали карточки.
- Если ничего не выбрано, правая панель показывает мини-панель занятости.
- На широком экране панель всегда видна.
- На маленьком экране открывается кнопкой `Сотрудники`.

Группы статусов:

- `Свободны`.
- `В работе`.
- `В ожидании`.
- `Конфликт`.

Что показывать по сотруднику:

- Короткое ФИО.
- Должность или участок.
- Текущая карточка, если сотрудник занят.
- Время в текущем статусе.
- Цветовой индикатор статуса.

Действия:

- Клик по свободному сотруднику открывает `Создать работу` с выбранным сотрудником.
- Клик по занятому сотруднику открывает его активную карточку.
- Клик по сотруднику в ожидании открывает карточку и причину ожидания.
- Клик по конфликту открывает список конфликтных карточек.
- Фильтр: `Все / Свободны / В работе / Ожидание / Конфликт`.
- Поиск по ФИО.
- Переключатель `Компактно / Подробно`.

Источник данных:

- Активные карточки ЭМУ.
- Избранные сотрудники ЭМУ.
- Общий справочник `employees` только для поиска и добавления в избранные.

Расчет статуса:

- Нет активной карточки -> `Свободен`.
- Есть активная карточка со статусом сотрудника `Работает` -> `В работе`.
- Есть открытая пауза или ожидание -> `В ожидании`.
- Сотрудник работает в двух карточках одновременно -> `Конфликт`.

Приоритет реализации:

1. Сначала `Учет работ`: создание, карточка, паузы, завершение.
2. Потом `История`: фильтры, детали, экспорт.
3. Потом `Дашборд`: реальные KPI, проблемные зоны, план-факт.
4. Потом уведомления и расширенная аналитика.

## ЭМУ v3: индивидуальное участие сотрудников, смены и свободное время

Статус: подтверждено к переносу в основной план. Реализацию выполнять отдельными пакетами после уже закрытых RBAC/core/operator/history/planning этапов. Мобильное приложение не трогать.

### Цель

Сделать карточку работы многофункциональной и устойчивой к реальным сменным сценариям:

- работа имеет общий жизненный цикл;
- каждый сотрудник внутри работы имеет собственный статус, интервалы работы, паузы, причины, возвраты и итог участия;
- карточка показывает фактическое время выполнения работы без суммирования параллельных сотрудников;
- история считает индивидуальную трудозатратность каждого сотрудника;
- смены, обед, свободное время, сверхурочка и переносы считаются отдельной логикой.

### Пакет v3.1: модель участия сотрудника в работе

Добавить модель интервалов участия сотрудника в рабочей карточке.

Обязательные поля:

- `workSessionId`;
- `employeeId`;
- `startedAt`;
- `endedAt`;
- `status`: `Работает`, `На паузе`, `Завершил участие`, `Частично выполнено`, `Добавлен ошибочно`;
- `reason`;
- `createdBy`;
- `createdAt`;
- audit-событие на каждое изменение.

Правила:

- сотрудника можно добавить в активную работу после старта карточки;
- время нового сотрудника начинается с фактического момента добавления;
- сотрудник может завершить участие раньше всей работы;
- сотрудник может вернуться в ту же работу с обязательной причиной;
- ошибочно добавленного сотрудника не удалять физически, а переводить в `Добавлен ошибочно`;
- время ошибочно добавленного сотрудника не попадает в трудозатраты и отчеты, но остается в audit.

### Пакет v3.2: индивидуальные и общие паузы

Переименовать пользовательский статус `В ожидании` в `На паузе`.

Типы пауз:

- общая пауза всей карточки;
- индивидуальная пауза конкретного сотрудника.

Правила:

- общая пауза создает запись паузы по каждому активному сотруднику;
- причина общей паузы может быть общей, но должна храниться отдельно по каждому сотруднику;
- индивидуальная пауза затрагивает только выбранного сотрудника;
- сотрудник может быть `На паузе` в одной работе и `Работает` в другой;
- конфликт возникает только если сотрудник одновременно `Работает` в двух активных карточках;
- если сотрудника забрали на другую работу, старая карточка ставит его `На паузе` с причиной, а новая карточка начинает новый рабочий интервал.

### Пакет v3.3: расчет времени карточки и сотрудников

Разделить три вида времени:

- фактическое время карточки;
- индивидуальное рабочее время сотрудника;
- индивидуальное время пауз сотрудника.

Правила расчета:

- карточка показывает фактическую длительность выполнения работы, а не сумму трудозатрат всех сотрудников;
- если два сотрудника работали параллельно по 10 минут, карточка показывает 10 минут;
- в истории каждый сотрудник получает свои 10 минут трудозатрат;
- если один сотрудник работал 10 минут, второй присоединился через 5 минут и работал 5 минут, карточка показывает 10 минут, а трудозатраты сотрудников: 10 и 5 минут;
- паузы показываются отдельно: где, когда, по какой причине и сколько длились;
- в истории участка показывать всех участников, включая замененных и частично завершивших участие.

### Пакет v3.4: переносы и блок `Перенесенные`

Если работа не завершена к концу смены, она переносится на следующий день без создания новой карточки.

Правила:

- номер работы сохраняется;
- карточка остается активной;
- причина переноса фиксируется в audit;
- сотрудники, которые не завершили участие, остаются `На паузе`;
- перенесенные работы показываются отдельным блоком `Перенесенные`;
- блок должен включать старые перенесенные работы и текущие работы, поставленные на паузу до следующей смены.

### Пакет v3.5: смены, обед, свободное время и сверхурочка

Добавить сменную модель сотрудника.

Типы смен:

- дневная 08:00-17:00, обед 12:00-13:00;
- 11-часовая 08:00-20:00, обед 12:00-13:00;
- ночная 20:00-08:00, обед 00:00-01:00;
- индивидуальная/скорректированная смена.

Правила:

- ночная смена считается по дате начала;
- свободное время считается только внутри фактической смены;
- свободное время не включает обед, работу и паузы;
- если сотрудник пришел раньше начала смены, смена считается с планового времени;
- если сотрудник пришел позже, смена считается с фактического входа;
- работа до начала смены учитывается в таймере работы, но отдельно помечается как `До начала смены`;
- сверхурочка начинается после окончания фактической смены;
- ручная корректировка смены требует причину и audit.

Права:

- корректировать фактическую смену могут `Оператор ЭМУ`, мастер, руководитель, admin и пользователи с отдельным правом `emu.shift.adjust`.

### Пакет v3.6: обед и спорные ситуации

Если активная работа пересекает обед:

- если сотрудник уже `На паузе`, обед исключается автоматически;
- если сотрудник работал без паузы, создать состояние `Требует решения`;
- оператор/мастер/руководитель выбирает: сотрудник работал в обед или обед нужно исключить;
- решение фиксируется в audit.

Состояние `Требует решения` использовать для:

- выхода сотрудника через Perco во время активной работы;
- пересечения обеда без подтверждения;
- ручного добавления сотрудника без фактического присутствия;
- конфликтных рабочих интервалов;
- переноса без подтвержденной причины.

Если `Требует решения` не закрыто больше 30 минут, поднимать приоритет до красного.

### Пакет v3.7: Perco-ready логика

Perco Web пока не подключать, но модель подготовить к событиям входа/выхода.

Будущие правила:

- вход до планового начала смены не сдвигает начало смены раньше плана;
- вход после планового начала сдвигает фактическое начало смены;
- выход во время активной работы автоматически ставит сотрудника на паузу;
- оператор получает уведомление и должен принять решение;
- выход в обед не считается проблемой;
- выход после окончания смены считается нормальным завершением присутствия.
