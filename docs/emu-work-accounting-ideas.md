# Идеи и проработки ЭМУ: учет работ

Дата создания: 2026-06-02  
Статус: черновой документ идей, без реализации в коде  
Важно: этот файл фиксирует предложения и UX-логику. Реализацию переносить в основной план только после подтверждения.

## Карточка работы: раздельное время по сотрудникам

Идея: в карточке работы нужно разделить общее время работы и индивидуальное время каждого сотрудника.

Сейчас в карточке под статусом `Работа` суммируется потраченное время всех сотрудников на одну работу. Это дает общий показатель по карточке, но не показывает вклад каждого сотрудника отдельно.

Как нужно:

- В карточке работы оставить общий блок по работе:
  - номер работы;
  - участок;
  - задача;
  - общий статус карточки;
  - общее рабочее время по карточке;
  - общее время пауз;
  - количество сотрудников.
- Для каждого сотрудника внутри карточки показывать отдельную строку/мини-блок:
  - ФИО или короткое имя;
  - текущий статус сотрудника на этой работе;
  - сколько он был в работе;
  - сколько он был на паузе;
  - были ли ручные изменения времени;
  - при необходимости короткий индикатор причины паузы.

## Переименование статуса сотрудника

Идея: статус сотрудника `В ожидании` переименовать в `На паузе`.

Причина: для оператора фраза `В ожидании` звучит как отдельное состояние ожидания чего-то внешнего, а по факту работа сотрудника остановлена. Статус `На паузе` понятнее и сразу объясняет, что таймер работы сотрудника остановлен.

Предлагаемая терминология:

| Текущее состояние | Новый текст в UI | Смысл |
|---|---|---|
| Работает | Работает | Сотрудник сейчас выполняет работу, идет рабочее время |
| В ожидании | На паузе | Работа сотрудника остановлена, идет время паузы |
| Завершено | Завершил | Сотрудник завершил участие в работе |
| Конфликт | Конфликт | Сотрудник одновременно числится в нескольких активных карточках |

## Мини-блок сотрудника в карточке

В компактной карточке работы рядом с именем сотрудника показывать статус и короткое время:

- `Бурухин Н.С. · На паузе · 4 мин`
- `Карандашев А.С. · Работает · 8 мин`

Цветовая логика:

- `Работает` - зеленый индикатор.
- `На паузе` - оранжевый индикатор.
- `Конфликт` - красный индикатор.
- `Завершил` - серый/нейтральный индикатор.

## Правая панель выбранной работы

Если выбрана карточка работы, в правой панели показывать не только общий статус, а расшифровку по сотрудникам.

Структура:

- Верхний блок:
  - участок;
  - список сотрудников;
  - задача;
  - общий статус карточки;
  - общее время работы;
  - общее время паузы.
- Ниже список сотрудников:
  - карточка сотрудника;
  - текущий статус;
  - рабочее время по этому сотруднику;
  - время паузы по этому сотруднику;
  - причина текущей паузы, если есть;
  - отметка ручной корректировки, если время меняли.

Пример:

```text
Бурухин Н.С.
На паузе
Работа: 4 мин
Пауза: 2 мин
Причина: ожидание допуска
```

```text
Карандашев А.С.
Работает
Работа: 8 мин
Пауза: 0 мин
```

## Правило расчета времени

Для карточки:

- `Общее время работы` - сумма рабочего времени всех сотрудников по этой карточке.
- `Общее время пауз` - сумма времени пауз всех сотрудников по этой карточке.
- `Итого по карточке` - можно показывать отдельно, если нужен руководительский показатель.

Для сотрудника:

- `Работа` - только его рабочее время в этой карточке.
- `Пауза` - только его паузы в этой карточке.
- `Изменения` - если оператор вручную изменил время, показывать отметку и ссылку на аудит.

## Что нужно продумать перед реализацией

- Есть ли в текущей БД достаточно данных для индивидуального времени по сотруднику или нужно расширять модель пауз/участия сотрудника.
- Как технически хранить общую паузу карточки и индивидуальные паузы сотрудников так, чтобы они не конфликтовали.
- Как отображать сотрудника, который завершил участие раньше всей карточки.
- Как отражать индивидуальные трудозатраты в истории и экспорте.

## Утвержденные правила модели

Дата фиксации: 2026-06-02

### 1. Пауза бывает общей и индивидуальной

Пауза может ставиться:

- на всю работу сразу, если вся бригада остановила работу или ушла на другую работу;
- на отдельного сотрудника, если только его забрали на другую работу, закончились материалы для его части работы, он не успел завершить до конца смены или есть другая индивидуальная причина.

### 2. Общий статус карточки при частичной паузе

Если хотя бы один сотрудник продолжает работать, карточка остается в статусе `В работе`.

Внутри карточки обязательно показывать состояние каждого сотрудника:

- кто сейчас `Работает`;
- кто `На паузе`;
- кто `Завершил участие`;
- где есть `Конфликт`.

Отдельный общий статус `Частично на паузе` пока не нужен как основной статус карточки. Его можно использовать только как визуальную подсказку/бейдж, если это улучшит читаемость.

### 3. Сотрудник может завершить участие раньше карточки

Разрешаем сценарий, когда один сотрудник завершил свою часть работы, а карточка продолжает выполняться другими сотрудниками.

В UI это нужно показывать как отдельный статус сотрудника `Завершил участие`, не завершая всю карточку автоматически.

### 4. Общее время карточки и трудозатраты сотрудников

В карточке работы для наглядности показываем длительность выполнения самой работы, а не сумму трудозатрат всех сотрудников.

Пример:

- 2 сотрудника работали одновременно по 10 минут;
- в карточке работы показываем `10 минут`, чтобы понимать фактическую длительность выполнения работы;
- в истории и аналитике по сотрудникам показываем индивидуальную трудозатратность каждого сотрудника: первый сотрудник `10 минут`, второй сотрудник `10 минут`.

Итоговая логика:

- карточка показывает календарную/операционную длительность работы;
- сотрудник показывает свою индивидуальную трудозатратность;
- история по участку должна уметь показывать, сколько каждый сотрудник потратил времени на этот участок.

### 5. Причина паузы индивидуальная

Причина паузы должна быть отдельной у каждого сотрудника.

Даже если оператор ставит общую паузу всей карточке, система должна записать причину паузы по каждому сотруднику, чтобы потом в истории было понятно, кто и почему был остановлен.

Примеры причин:

- забрали на другую работу;
- ожидание материалов;
- ожидание допуска;
- конец смены;
- ожидание оборудования;
- прочее.

### 6. Пауза из-за конца смены переносит работу на следующий день

Если сотрудник или вся карточка находятся на паузе по причине конца смены, работа автоматически переносится на следующий день.

Нужно продумать backend-правило:

- работа не должна потеряться как `забытая`;
- перенос должен быть виден в истории карточки;
- старый день и новый день должны фиксироваться в audit;
- на следующий день карточка должна быть доступна для продолжения;
- время сотрудника после продолжения считается отдельным новым интервалом.

### 7. В активную работу можно добавлять новых сотрудников

Оператор может добавить нового сотрудника в уже активную карточку.

Правило времени:

- у нового сотрудника время начинается не с момента создания карточки, а с момента его фактического присоединения к работе;
- если остальные сотрудники уже работали час, а новый сотрудник пришел 5 минут назад, у него показывается только 5 минут работы;
- в истории должно быть видно, когда сотрудник был добавлен в карточку.

Нужно хранить для участия сотрудника:

- `joinedAt` - когда сотрудник приступил к этой работе;
- `leftAt` - когда сотрудник завершил участие, если завершил раньше карточки;
- статус участия;
- audit события добавления/удаления/возврата сотрудника.

### 8. Сотрудника можно вернуть в карточку после завершения участия

Если сотрудник завершил участие в работе, его можно вернуть обратно в эту же карточку.

Обязательное правило:

- при возврате нужно указать причину возвращения;
- новое время работы начинается с момента возврата;
- старый интервал участия не затирается;
- в истории должны быть видны оба интервала.

Пример:

```text
Карандашев А.С.
Интервал 1: 08:00 - 09:10, работа 1 ч 10 мин
Завершил участие: 09:10
Вернулся: 10:15, причина: повторная проверка узла
Интервал 2: 10:15 - 10:40, работа 25 мин
```

### 9. Общая пауза и индивидуальные причины

При постановке общей паузы оператор может:

- указать одну общую причину для всех сотрудников;
- при необходимости переопределить причину отдельно для конкретного сотрудника.

Это нужно для случаев, когда вся работа остановлена, но причины у сотрудников разные.

Пример:

```text
Общая пауза: ожидание материалов
Бурухин Н.С.: ожидание материалов
Карандашев А.С.: забрали на другую работу
```

### 10. История участка показывает два типа времени

В истории по участку нужно показывать отдельно:

- фактическую длительность работы;
- индивидуальные трудозатраты сотрудников.

Пример:

```text
Участок Обжига
Работа ЭМУ-2026-000007
Фактическая длительность: 10 мин

Сотрудники:
Бурухин Н.С. - работа 10 мин, пауза 0 мин
Карандашев А.С. - работа 10 мин, пауза 0 мин
```

Так руководитель видит, сколько реально длилась работа, а также кто сколько времени потратил на участке.

### 11. Пауза показывается отдельно от чистого рабочего времени

По сотруднику нужно показывать:

- `Работа` - чистое время выполнения;
- `Пауза` - время остановки с причиной;
- при необходимости `Итого участия` - работа плюс пауза.

Основной показатель для трудозатрат - чистое рабочее время.

Паузы не скрываются: в истории должно быть видно, где сотрудник был на паузе, по какой причине и сколько времени.

Пример:

```text
Бурухин Н.С.
Работа: 10 мин
Пауза: 5 мин
Причина паузы: ожидание материалов
Итого участия: 15 мин
```

### 12. Аналитика должна разделять работу и паузы

Для оплаты, загрузки и аналитики нужно отдельно считать:

- чистое рабочее время сотрудника;
- время пауз;
- причины пауз;
- в каких работах и на каких участках сотрудник был на паузе;
- сколько раз сотрудника забирали на другую работу;
- сколько работ переносились из-за конца смены.

Пауза не должна пропадать как "нерабочее пустое время": она нужна для анализа потерь, причин простоев и перегрузки сотрудников.

### 13. Перенос сохраняет тот же номер работы

Если работа переносится на следующий день из-за конца смены, это остается та же карточка и тот же номер работы.

Новая карточка-продолжение не создается.

Что должно фиксироваться:

- старый рабочий день;
- новый рабочий день;
- причина переноса;
- кто перенес;
- время переноса;
- какие сотрудники остались на паузе;
- какие сотрудники продолжат работу на следующий день.

### 14. Перенесенные работы показываются отдельным блоком

На экране `Учет работ` нужен отдельный блок `Перенесенные`.

Назначение блока:

- оператор сразу видит, какие работы не закрыты с прошлой смены/дня;
- понятно, на каком участке работа;
- видно, кто из сотрудников был в работе и кто на паузе;
- можно быстро продолжить работу или заменить/добавить сотрудника.

Предлагаемое расположение:

- отдельная группа над активными карточками;
- либо отдельная вкладка/сегмент рядом с `Активные`, `На паузе`, `Завершенные`.

### 15. Добавлять сотрудника в активную работу могут роли с правом

Добавлять сотрудника в активную карточку может:

- оператор ЭМУ;
- мастер;
- руководитель;
- администратор;
- другой пользователь, которому выдано соответствующее право.

Нужно завести отдельное право действия, например:

```text
emu.work.employees.manage
```

Это право должно управлять:

- добавлением сотрудника в активную работу;
- завершением участия сотрудника;
- возвратом сотрудника в карточку;
- пометкой ошибочного добавления.

### 16. Ошибочно добавленный сотрудник не удаляется физически

Если сотрудника добавили в карточку ошибочно, его не нужно удалять из истории.

Нужно сделать отдельный статус участия:

```text
Добавлен ошибочно
```

Правила:

- такой сотрудник не учитывается в трудозатратах;
- его время не попадает в рабочее время карточки;
- запись остается в audit;
- в истории видно, кто добавил ошибочно и кто пометил ошибку.

### 17. Перевод сотрудника на другую работу выполняется последовательностью действий

Если сотрудника забрали с текущей работы на другую:

1. Оператор ставит сотрудника на паузу в текущей работе.
2. Указывает причину, например `Забрали на другую работу`.
3. Создает новую работу.
4. Выбирает этого сотрудника.
5. В новой работе время сотрудника начинается с момента его фактического начала новой работы.

Так старая карточка сохраняет правильную паузу, а новая карточка получает новый рабочий интервал.

### 18. Сотрудник может быть на паузе в одной работе и работать в другой

Разрешенный сценарий:

```text
Работа А: сотрудник На паузе, причина - забрали на другую работу
Работа Б: сотрудник Работает
```

Это не считается конфликтом, если в старой работе сотрудник именно `На паузе`.

Конфликтом считается ситуация, когда сотрудник одновременно имеет два активных статуса `Работает` в разных карточках.

### 19. История сотрудника: за смену и по участкам

История сотрудника должна иметь два представления:

- за смену одной лентой;
- по участкам.

За смену показывать:

- все работы сотрудника за смену;
- рабочее время;
- паузы;
- причины пауз;
- свободное время;
- переносы;
- ошибочные добавления, если были.

По участкам показывать:

- на каком участке сотрудник работал;
- сколько чистого рабочего времени потратил на участок;
- сколько был на паузе на этом участке;
- по каким причинам были паузы.

### 20. Расчет свободного времени сотрудника за смену

Нужно считать свободное время сотрудника за смену.

Свободное время = рабочее время смены минус:

- чистое время работы;
- время пауз;
- обеденный перерыв.

Примеры графиков:

```text
8-часовая смена:
08:00 - 17:00
обед 12:00 - 13:00
рабочее время без обеда: 8 часов
```

```text
11-часовая смена:
08:00 - 20:00
обед 12:00 - 13:00
рабочее время без обеда: 11 часов
```

В карточке/истории сотрудника показывать:

```text
Смена: 08:00 - 17:00
Обед: 12:00 - 13:00
Работал: 5 ч 20 мин
Пауза: 40 мин
Свободно: 2 ч
```

Важно: обед не считать ни работой, ни паузой, ни свободным временем.

### 21. Фиксация типа смены

Для сотрудника и работы нужно фиксировать тип смены:

- дневная;
- ночная.

Это нужно, чтобы:

- понимать, почему работа была перенесена;
- правильно считать свободное время;
- правильно показывать историю за смену;
- разделять отчеты по дневным и ночным сменам.

Минимальные поля для будущей реализации:

- `shiftType`: `day` / `night`;
- `shiftStartedAt`;
- `shiftEndedAt`;
- `lunchStartedAt`;
- `lunchEndedAt`.

### 22. Справочники смен и ручная корректировка смены

Смены должны быть фиксированными справочниками, но с возможностью ручной корректировки в крайних случаях.

Причины ручной корректировки:

- сверхурочная работа;
- опоздание;
- сотрудник пришел позже стандартного начала;
- сотрудник остался дольше стандартного окончания;
- производственная необходимость.

Примеры нестандартных смен:

```text
Стандартная смена: 08:00 - 17:00
Фактически сотрудник: 09:15 - 17:00
```

```text
Стандартная смена: 08:00 - 17:00
Фактически сотрудник: 10:00 - 20:00
```

Правило:

- базовый график берется из справочника смен;
- фактическое время можно подправить в карточке/сменной истории сотрудника;
- ручная корректировка должна попадать в audit с причиной.

### 23. Обеденный перерыв

Стандартно обед один час.

Для дневных смен:

```text
Обед: 12:00 - 13:00
```

Для ночных смен:

```text
Обед: 00:00 - 01:00
```

Но возможны исключения:

- сотрудник не ходил на обед;
- сотрудник работал во время обеда;
- сотрудник работал сверхурочно.

Правило:

- по умолчанию обед исключается из рабочего/свободного времени;
- если сотрудник реально работал в обед, это должно фиксироваться как исключение;
- исключение по обеду нужно хранить с причиной и audit.

### 24. Ночная смена

Ночная смена:

```text
20:00 - 08:00
обед 00:00 - 01:00
```

Ночная смена переходит через календарную дату, поэтому учет должен идти по смене, а не просто по календарному дню.

Правила:

- смена начинается вечером одного дня и заканчивается утром следующего дня;
- история должна показывать, что сотрудник работал именно в ночную смену;
- работы внутри ночной смены группируются по смене;
- перенос даты не должен ломать расчет времени;
- свободное время считается внутри границ ночной смены, исключая обед.

### 25. Свободное время считается только внутри фактической смены

Если смена еще не началась или уже закончилась, это не считается свободным временем.

Свободное время считается только внутри фактических границ смены сотрудника:

- начало смены;
- конец смены;
- исключенный обед;
- чистое рабочее время;
- паузы.

Если фактическое начало/окончание смены отличается от справочника, расчет идет по фактическому времени.

### 26. Перевод на другую работу и учет времени

Если сотрудника забрали на другую работу:

- в старой работе это считается паузой с причиной `Забрали на другую работу`;
- в новой работе это считается рабочим временем с момента фактического начала новой работы;
- одно и то же время не должно одновременно считаться чистой работой в старой и новой карточке.

### 27. Свободное время показываем в общей истории сотрудника

Свободное время не нужно показывать в отчете по каждому участку.

Где показывать:

- общая история сотрудника за смену;
- сменная сводка сотрудника;
- руководительская аналитика загрузки.

В отчете по участку показывать:

- чистое рабочее время сотрудника на участке;
- паузы на участке;
- причины пауз на участке.

Свободное время относится к общей загрузке сотрудника за смену, а не к конкретному участку.

### 28. Права на корректировку фактической смены

Фактическую смену сотрудника могут менять:

- оператор ЭМУ;
- мастер;
- руководитель;
- администратор;
- пользователь, которому отдельно выдали право.

Для этого нужно отдельное право, например:

```text
emu.shift.adjust
```

Это право должно разрешать:

- изменить фактическое начало смены;
- изменить фактическое окончание смены;
- отметить исключение по обеду;
- скорректировать сверхурочную работу;
- исправить ошибочно заданный график.

### 29. Причина корректировки смены обязательна

Любое ручное изменение фактической смены должно требовать причину.

Что фиксировать в audit:

- кто изменил;
- когда изменил;
- старое значение;
- новое значение;
- причина;
- сотрудник;
- смена.

### 30. Работа в обед: автоматика плюс ручное подтверждение

Работу в обед нужно определять автоматически, но оставить ручное управление.

Правило:

- если сотрудник был поставлен на паузу в период обеда, обед автоматически исключается из работы;
- если сотрудник не был на паузе и активная работа пересекла обед, система должна показать ожидание подтверждения: работал он в обед или обед нужно исключить;
- оператор может вручную подтвердить, что сотрудник работал в обед;
- оператор может вручную отметить, что обед был, даже если пауза не была поставлена вовремя.

Это нужно, чтобы не терять реальные сверхурочные и не считать обед неправильно.

### 31. Ночная смена считается по дате начала

Для ночной смены дата смены считается по дате начала смены.

Пример:

```text
Смена: 02.06 20:00 - 03.06 08:00
Дата смены: 02.06
```

Все работы внутри этого интервала должны попадать в смену `02.06`, даже если часть работы выполнена уже `03.06`.

### 32. Свободное время до интеграции Perco Web

Планируется возможная интеграция с Perco Web для фиксации входа/выхода сотрудников и автоматического определения фактического начала/окончания смены.

До интеграции Perco Web:

- если у сотрудника началась смена и нет активной работы, время можно считать свободным;
- опоздание отдельно не фиксируем как обязательную причину;
- ручная корректировка смены остается доступной через право `emu.shift.adjust`.

После интеграции Perco Web можно будет уточнить модель:

- вход сотрудника на территорию;
- выход сотрудника;
- фактическое начало смены;
- фактическое окончание смены;
- опоздания и ранние уходы.

### 33. Формат отображения свободного времени

Свободное время показывать в часах и минутах.

Пример:

```text
Свободно: 2 ч 15 мин
```

Процент загрузки можно добавить позже в аналитике, но в карточке/истории сотрудника основной формат - часы и минуты.

### 34. Perco Web: начало смены по турникету

При будущей интеграции Perco Web фактическое начало смены нужно определять по входу через турникет, но с учетом планового начала смены.

Правила:

- если сотрудник пришел раньше планового начала смены, счет смены начинается с планового времени;
- если сотрудник пришел после планового начала смены, счет смены начинается с фактического входа;
- если сотрудник пришел на 1 минуту позже, фактическое начало смены тоже сдвигается на эту минуту;
- если сотрудник пришел сильно раньше, время до начала смены не считается рабочим временем.

Примеры:

```text
Плановая смена: 08:00 - 17:00
Вход через турникет: 07:17
Фактическое начало учета смены: 08:00
```

```text
Плановая смена: 08:00 - 17:00
Вход через турникет: 08:32
Фактическое начало учета смены: 08:32
```

```text
Плановая смена: 08:00 - 17:00
Вход через турникет: 08:01
Фактическое начало учета смены: 08:01
```

### 35. Perco Web: выход с территории при активной работе

Если Perco Web показывает, что сотрудник вышел с территории, но в ЭМУ у него активная работа:

- система автоматически ставит сотрудника на паузу;
- создается уведомление оператору;
- оператор должен принять решение и исправить ситуацию при необходимости.

Пример уведомления:

```text
Сотрудник вышел с территории, но числится в активной работе.
Сотрудник: Бурухин Н.С.
Работа: ЭМУ-2026-000007
Действие: сотрудник автоматически поставлен на паузу, требуется проверка.
```

### 36. Подтверждение работы в обед

Если активная работа пересекла обед и система требует подтверждение, подтверждать могут:

- оператор ЭМУ;
- мастер;
- руководитель.

Это действие должно попадать в audit.

### 37. Сверхурочная работа

В отчетах нужно отдельно показывать сверхурочную работу.

Сверхурочная работа - это рабочее время за пределами стандартной или фактически подтвержденной смены.

Показывать:

- сколько сверхурочных часов/минут;
- на каких работах;
- на каких участках;
- кто подтвердил или изменил смену;
- причина корректировки, если она была.

### 38. Свободное время: итог и интервалы

В истории сотрудника нужно показывать:

- итоговое свободное время за смену;
- интервалы свободного времени между работами.

Пример:

```text
Свободно за смену: 2 ч 15 мин

Интервалы:
08:00 - 08:35 свободен
10:20 - 11:05 свободен
15:40 - 16:20 свободен
```

Это нужно, чтобы видеть не только общую цифру, но и когда именно сотрудник был свободен.

### 39. Блок перенесенных работ

В блоке `Перенесенные` показывать:

- работы, перенесенные с прошлой смены/дня;
- сегодняшние работы, которые поставлены на паузу до завтра;
- работы, где один или несколько сотрудников остались на паузе из-за конца смены.

Цель: оператор должен видеть все работы, которые требуют продолжения или решения, а не только старые незавершенные карточки.

### 40. Ошибочно добавленный сотрудник: audit без уведомления

Если сотрудника добавили ошибочно и пометили статусом `Добавлен ошибочно`, отдельное уведомление руководителю не требуется.

Достаточно:

- audit event;
- видимость в истории карточки;
- исключение из трудозатрат.

### 41. Perco Web: интервалы присутствия

Если сотрудник вошел, вышел и снова вошел в течение смены, это нужно считать как интервалы присутствия.

Пример:

```text
08:00 вход
11:00 выход
11:30 вход
17:00 выход
```

В истории смены должны быть видны интервалы:

```text
08:00 - 11:00 присутствовал
11:00 - 11:30 отсутствовал
11:30 - 17:00 присутствовал
```

Эти интервалы нужны для корректного расчета свободного времени, отсутствия и спорных ситуаций с активными работами.

### 42. Выход с территории без активной работы

Если сотрудник вышел с территории без активной работы, трактовка зависит от времени:

- если вышел во время обеда, это не должно считаться проблемой;
- если вышел во время смены не в обед, это отсутствие;
- если вышел по окончанию смены, это нормальное завершение присутствия, не свободное время.

### 43. Ручное добавление сотрудника без входа через турникет

Если Perco Web не показывает вход сотрудника через турникет, но оператор вручную добавляет его в работу:

- добавление разрешается;
- система показывает предупреждение;
- действие фиксируется в audit.

Пример предупреждения:

```text
По Perco Web сотрудник не отмечен на территории.
Добавить в работу все равно?
```

### 44. Работа до начала смены и сверхурочка

Если сотрудник пришел раньше планового начала смены и его отправили в работу до начала смены, это не считать сверхурочной работой в текущей модели.

Сверхурочная работа считается, если сотрудник продолжает работать после окончания смены.

Пример:

```text
Смена: 08:00 - 17:00
Начал работу: 07:45
Не считаем сверхурочкой утром.
```

```text
Смена: 08:00 - 17:00
Продолжил работу до 18:20
Сверхурочно: 1 ч 20 мин
```

### 45. Замена сотрудника в перенесенной работе

Если работа перенесена на следующий день:

- если старый сотрудник пришел и продолжает работу, его возвращаем из паузы и продолжаем учет его участия;
- если вместо него направили другого сотрудника, старому сотруднику ставим результат участия `Частично выполнено`;
- время старого сотрудника засчитывается, потому что он уже потратил время на эту работу;
- новый сотрудник начинает свой интервал с момента фактического приступления.

Важно: результат `Частично выполнено` относится к участию сотрудника, а не обязательно ко всей карточке работы.

### 46. Сверхурочка в отчетах

В отчетах по сверхурочной работе учитывать:

- чистое рабочее время сверхурочно;
- паузы сверхурочно показывать отдельно.

Не смешивать паузы со сверхурочной трудозатратой.

Пример:

```text
Сверхурочная работа: 1 ч 20 мин
Паузы сверхурочно: 15 мин
```

### 47. Статус `Требует решения`

Нужен отдельный статус/флаг `Требует решения` для карточки или сотрудника, если автоматика обнаружила спорную ситуацию и оператор еще не подтвердил решение.

Примеры ситуаций:

- Perco Web показал выход с территории при активной работе;
- работа пересекла обед без паузы и требует подтверждения;
- сотрудник добавлен в работу без входа через турникет;
- перенесенная работа требует решения: вернуть старого сотрудника или назначить нового;
- конфликт рабочих интервалов.

`Требует решения` должен:

- подсвечиваться в активной доске;
- попадать в мини-панель проблем;
- создавать уведомление оператору;
- исчезать после принятия решения и audit-записи.

### 48. Статус `Частично выполнено` у сотрудника показывается в активной карточке

Если сотрудник был заменен или завершил только свою часть работы, статус `Частично выполнено` нужно показывать прямо в активной карточке, пока работа продолжается.

Это нужно, чтобы оператор видел:

- кто уже участвовал в работе;
- кто выполнил часть работы;
- кого заменили;
- кто сейчас продолжает работу.

В истории этот статус также сохраняется.

### 49. `Требует решения` может быть у карточки и у сотрудника

Флаг `Требует решения` может быть:

- у всей карточки;
- у отдельного сотрудника;
- одновременно у карточки и у сотрудника.

Пример:

```text
Карточка требует решения: работа пересекла обед без подтверждения
Сотрудник требует решения: вышел с территории по Perco при активной работе
```

### 50. Эскалация долгих нерешенных проблем

Если статус `Требует решения` висит долго, уведомление нужно усиливать.

Предварительное правило:

- до 30 минут - предупреждение;
- после 30 минут - красный приоритет.

Это нужно, чтобы оператор не пропускал спорные ситуации.

### 51. Perco: подтвержденное завершение закрывает автоматическую паузу

Если Perco Web показал выход сотрудника с территории, система автоматически поставила паузу, а оператор потом подтвердил, что работа была завершена:

- автоматическую паузу закрыть;
- в карточке оставить audit-исключение;
- добавить примечание, что пауза была закрыта по подтверждению оператора/мастера/руководителя.

Пример audit:

```text
Автоматическая пауза по Perco закрыта.
Причина: оператор подтвердил завершение работы.
```

### 52. Работа до начала смены

Если сотрудник начал работу до планового начала смены, это время не считается сверхурочным.

Такое время относить в отдельный блок:

```text
До начала смены
```

При этом в таймере работы это время добавляется к общей длительности работы.

В карточке не нужно отдельно подсвечивать, что часть времени была до начала смены. Достаточно, чтобы в истории/аналитике это можно было увидеть как редкий интервал.

### 53. Отчет по участку показывает замененных и частично выполнивших сотрудников

В отчете по участку нужно показывать всех сотрудников, которые участвовали в работе:

- работал до конца;
- частично выполнил;
- был заменен;
- был на паузе;
- был добавлен ошибочно, если нужен audit-режим.

Для каждого сотрудника указывать:

- статус участия;
- рабочее время;
- паузы;
- причины пауз;
- кто заменил, если была замена;
- итог участия.
