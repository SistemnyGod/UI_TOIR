# Аудит модуля СИЗ: нормы, выдача, печать, история

Дата: 2026-06-25

Область проверки: backend, frontend, DTO, печать DOCX/PDF/preview и текущая бизнес-логика модуля СИЗ в Patrol360.

Ключевой бизнес-поток:

```text
Сотрудник -> Норма СИЗ -> Номенклатура -> Факт выдачи -> Печать личной карточки -> Лист подписи -> История
```

## 1. Краткий вывод

Модуль СИЗ уже содержит часть нужных данных: сотрудник, номенклатура, карточка, строка карточки, нормативное печатное имя, пункт нормы, периодичность, статусы и история событий по строке. Но текущая модель все еще смешивает три разных уровня:

| Уровень | Как должно быть | Что сейчас |
| --- | --- | --- |
| Норма СИЗ | Отдельная строка нормы по должности: `normName`, `normPoint`, `periodicity`, `quantityPeriod`, разделители | Частично хранится в `InventoryPositionNormEntity` через ссылку на складской `ItemId`, а в выдаче дублируется в `InventoryPpeCardLineEntity.PrintItemName/NormPoint/IssuePeriodText` |
| Номенклатура | Складской предмет: складское имя, бренд, модель, артикул, класс защиты, остаток | Есть в `InventoryItemEntity` и `InventoryItemDto`, но используется как fallback для печатной нормы |
| Факт выдачи | Отдельная операция выдачи с датой, количеством, подписью, возвратом, списанием | Факт выдачи совмещен со строкой карточки `InventoryPpeCardLineEntity`; отдельного `PpeIssueItem` нет |

Главная ошибка: система может сформировать документы из неполных или смешанных данных, потому что серверный validation gate слабее, чем новые UI-проверки. Личная карточка и лист подписи должны строиться из единого печатного DTO, но backend сейчас собирает строки напрямую из `InventoryPpeCardLineEntity` и `InventoryItemEntity`, с fallback на складское имя.

Самые критичные проблемы:

1. Backend export содержит mojibake в русских строках и validator категориям проверяет битые значения, а не реальные `Каски`, `Одежда`, `Обувь`.
2. Лист подписи backend может включить строки `not_issued`, `issuing`, `partial`, потому что `IsOpenPpeLineForSignature()` исключает только закрытые статусы.
3. `ValidatePpeLine()` не проверяет полное нормативное имя, пункт нормы, периодичность, дату выдачи, единицу, складской остаток и запрет категории вместо нормы.
4. Нет отдельной сущности строки нормы `PpeNormItem` и факта выдачи `PpeIssueItem`.
5. Модель/марка/артикул вычисляются из складской позиции и не сохраняются как отдельное печатное значение выдачи.
6. PPE-выдача не подтверждает складской остаток и не создает очевидное движение склада при изменении статуса на `issued`.
7. История фиксирует создание и смену статуса, но не полноценно фиксирует изменение нормы, номенклатуры, модели, цены, количества.

## 2. Главные неправильные логики

| № | Неправильная логика | Где возникает | Последствие | Как исправить | Приоритет |
| --- | --- | --- | --- | --- | --- |

| 10 | Модель/марка/артикул не сохраняются как факт выдачи | `InventoryPpeCardLineDto.ModelDescription` вычисляется из item | Если пользователь уточнил модель вручную, это не станет устойчивым фактом | Добавить `BrandModelArticle` в DTO/entity или `PpeIssueItem`; вести словарь подсказок | P1 |

| 16 | DOCX/preview могут расходиться по правилам | Backend export и web preview собирают строки разными путями | Пользователь видит одно, DOCX получает другое | Ввести единый `PpePrintLine` DTO для backend export и web preview | P0 |

| 18 | Тесты печати используют mojibake fixtures | `tests/Patrol360.Infrastructure.Tests/InventoryPpePrintDbIntegrationTests.cs` | Тест может пройти, даже если реальные русские категории не валидируются | Перевести fixture в UTF-8 и добавить негативные тесты | P0 |

## 3. Ошибки сценариев

| Сценарий | Что сейчас неправильно | Как должно быть | Доработка |
| --- | --- | --- | --- |
| Создание карточки | Карточка создается по сотруднику, но нормы подтягиваются через `positionNorms`, где норма связана со складским item | Сначала сотрудник, затем независимые нормы по должности/подразделению | Ввести `PpeNormItem` и экран выбора/подтверждения норм |
| Подбор норм | `InventoryPositionNormDto` содержит `ItemId`, `ItemName`, `Quantity`, `LifeMonths`, но не содержит `normPoint` и оригинальный текст периодичности | Норма должна иметь полное имя, пункт, периодичность, количество, разделитель, сортировку | Расширить модель нормы и импортировать стартовые нормы из DOCX |
| Выбор номенклатуры | UI уже разделяет норму/номенклатуру, но backend принимает только `ItemId` и `PrintItemName` | Номенклатура выбирается после нормы и не меняет normName | Добавить `NormItemId`, `CatalogItemId`, `BrandModelArticle` |
| Выдача строки | Можно создать строку без полноценной серверной проверки нормы/даты/склада | Факт выдачи создается только для предметной строки нормы с датой, количеством и складским item | Усилить `ValidatePpeLine()` и добавить `PpeIssueItem` |
| Не выдано | `not_issued` может попасть в backend лист подписи | Невыданное остается в норме, но не попадает в лист подписи как факт | Исправить фильтр signature sheet |
| Нет на складе | Отдельного статуса нет | Строка нормы остается незакрытой потребностью; факт выдачи не создается | Добавить status `no_stock`/`issue_later` или отдельный demand state |
| Заменено аналогом | Нет отдельного статуса и причины | Нормативное имя остается прежним, меняется только catalog item, нужна причина | Добавить status `analog_replacement`, comment/reason, history |
| Возврат | Статус `returned` есть, но нет отдельного возвратного количества/даты/подписи | Возврат должен попасть в историю и лист подписи, склад увеличивается если предмет возвращаемый | Добавить return fields/events и stock move |
| Списание | Статус `written_off` есть, но акт списания в листе подписи не заполнен | Списание должно иметь акт/дату/номер и событие | Добавить write-off metadata |
| Печать | Backend тексты и validator частично mojibake | Документы должны быть читаемым UTF-8 и валидироваться до генерации | Исправить строки и тесты |

## 7. Ошибки статусов

Текущие backend-статусы PPE строки:

| Статус | Что означает сейчас | Проблема | Исправление |
| --- | --- | --- | --- |
| `not_issued` | Не выдано | Может попасть в backend лист подписи; смешивает норму и факт | Исключить из листа подписи; трактовать как состояние нормы/потребности |
| `issuing` | Готовится к выдаче | Не является фактом выдачи, но может печататься в листе подписи | Исключить из листа подписи |
| `partial` | Частично | Неясно, частично выдано или частично закрыта потребность | Разделить `partiallyIssued` и demand state |
| `issued` | Выдано | Нет обязательной проверки склада/warehouse и сохраненной модели | Добавить validation, stock move, issue fact |
| `returned` | Возвращено | Нет отдельной даты/количества возврата в DTO печати | Добавить return metadata |
| `written_off` | Списано | Нет акта списания в печати | Добавить write-off act fields |
| `lost` | Утеряно | Нет отдельного сценария взыскания/акта | Описать business rule |
| `reissued` | Переоформлено | Нормализуется, но transition rules явно не дают обычный путь в этот статус | Уточнить lifecycle или удалить из доступных |

| Статус | Что означает | Кто ставит | Что дальше | Что логировать |

## 8. Ошибки истории

| Действие | Логируется сейчас | Что должно логироваться | Исправление |
| --- | --- | --- | --- |
| Создание карточки | Да, system log/card created | Номер, сотрудник, должность, actor | Оставить и расширить details |
