# ТЗ: мобильное приложение Patrol360

Дата: 25.05.2026  
Статус: финальная версия v1.0: готово к реализации MVP  
Область: мобильное приложение для сотрудников, обходов и учета работ

## 1. Цель

Нужно разработать мобильное приложение Patrol360, которое стабильно работает с текущим сайтом и backend API, принимает уведомления и заявки на проведение обходов, позволяет сотруднику выполнять обходы с телефона, фиксировать выполненные задачи и получать задачи из модуля ЭМУ / "Учет работ".

Приложение должно работать:

- онлайн через backend Patrol360;
- локально в сети предприятия;
- оффлайн без связи с сервером;
- с последующей синхронизацией накопленных действий.

Главный принцип: сотрудник не должен терять рабочие задания и внесенные факты из-за плохой связи.

## 2. Рекомендуемый технологический стек

Мобильное приложение:

- React Native + Expo.
- TypeScript.
- Expo Router для навигации.
- SQLite на устройстве через `expo-sqlite`.
- Secure storage для токенов и чувствительных данных.
- Push notifications через FCM/APNs с backend-регистрацией устройства.
- Background sync через нативные фоновые задачи, где это разрешает ОС.
- `@react-native-community/netinfo` для определения состояния сети.
- `expo-file-system` для локального хранения фото до подтвержденной отправки.
- `expo-camera` или совместимая камера для QR/фотофиксации.
- `react-native-nfc-manager` для чтения NFC-меток, если NFC включается в релиз.
- `expo-secure-store` для access/refresh token и локального ключа устройства.
- `zod` для валидации DTO и payload перед записью в outbox.
- TanStack Query только для online-запросов и кеширования справочников; критичные offline-данные хранить в SQLite, а не только в query cache.
- Локальные миграции SQLite с версией схемы приложения.

Backend:

- текущий ASP.NET Core API Patrol360;
- отдельный mobile/sync API поверх существующих модулей;
- PostgreSQL как источник истины;
- idempotency keys для всех мобильных команд;
- audit log для всех действий с телефона.

Почему не WebView: приложению нужны оффлайн-данные, локальная очередь действий, камера/QR/NFC, стабильные push-уведомления, фоновые синхронизации и нормальная работа при плохой сети. WebView для этого будет ненадежным.

## 3. Роли пользователей

### 3.1. Сотрудник обхода

Получает заявки и назначения на обход, видит маршрут и точки, проходит маршрут, фиксирует факт выполнения, замечания, фото и комментарии.

### 3.2. Сотрудник ЭМУ

Получает задачи из "Учет работ", видит активные работы, может принять задачу, начать выполнение, поставить на паузу, продолжить, завершить и описать результат.

### 3.3. Руководитель / оператор

В первой версии может использовать web-панель как основной рабочий инструмент. Мобильные функции руководителя допускаются позже: просмотр статусов, подтверждения, контроль просрочек.

## 4. Основные модули мобильного приложения

### 4.1. Вход и профиль

Требования:

- вход по текущему backend auth;
- хранение session token в защищенном хранилище;
- отображение текущего пользователя, ролей и прав;
- logout с очисткой локальной сессии;
- режим "нет связи" после успешного входа хотя бы один раз;
- запрет первого входа без сети, если нет локальной авторизованной сессии.

Окно авторизации:

- поле логина;
- поле пароля;
- выбор адреса сервера: основной, локальная сеть предприятия, тестовый;
- индикатор доступности сервера до входа;
- кнопка "Войти";
- ссылка/кнопка "Продолжить оффлайн" только если пользователь уже входил ранее и локальная сессия действительна;
- отображение версии приложения и текущего API endpoint;
- понятные ошибки: нет сети, сервер недоступен, неверный логин/пароль, истекла сессия, сертификат недоверенный;
- после успешного входа приложение выполняет `bootstrap` и сохраняет минимум данных для оффлайн-старта.

После первого успешного входа можно включить быстрый вход:

- PIN-код приложения;
- биометрия, если устройство поддерживает;
- автоматическое обновление токена при наличии сети;
- запрет оффлайн-входа, если пользователь был заблокирован сервером при последней синхронизации.

### 4.2. Уведомления

Приложение должно стабильно получать уведомления о:

- новой заявке на обход;
- назначении сотрудника на маршрут;
- изменении или отмене назначения;
- просрочке обхода;
- новой задаче ЭМУ;
- изменении задачи ЭМУ;
- напоминании о незавершенной работе;
- конфликте синхронизации, если действие с телефона не принято сервером.

Требования:

- устройство регистрирует push token на backend;
- backend хранит привязку `deviceId`, `pushToken`, `userId`, `employeeId`, platform, appVersion;
- уведомление должно открывать нужный экран и конкретную сущность;
- уведомления не должны дублироваться при повторной регистрации устройства;
- если push не доставлен, задача все равно должна появиться после sync;
- пользователь видит локальный журнал уведомлений внутри приложения.

### 4.3. Заявки и назначения на обход

Сотрудник должен видеть:

- доску доступных заявок на обход;
- список назначенных обходов;
- статус каждого обхода;
- дату, время, смену;
- плановое время начала;
- маршрут;
- ФИО сотрудника, если заявка уже закреплена;
- точки маршрута;
- обязательность фото;
- комментарии оператора;
- историю изменений назначения.

Основные действия:

- взять свободную заявку с доски, если это разрешено правами;
- принять назначение;
- начать обход;
- открыть маршрут;
- отметить точку как пройденную;
- добавить комментарий;
- добавить фото;
- зафиксировать проблему;
- завершить обход;
- отправить результат на сервер.

Доска заявок:

- показывает доступные заявки на обход;
- по каждой заявке показывает ФИО закрепленного сотрудника или статус "свободная";
- показывает название маршрута;
- показывает плановое время начала;
- показывает статус: свободная, закреплена, принята, в работе, завершена, просрочена, отменена;
- сотрудник может взять только разрешенную ему заявку;
- если два сотрудника одновременно пытаются взять одну заявку, сервер должен закрепить ее только за одним, второй получает понятное сообщение;
- после взятия заявки приложение сразу загружает маршрут и точки для оффлайн-прохождения.

Оффлайн:

- назначенные обходы должны быть доступны без сети после синхронизации;
- прохождение точек пишется локально;
- фото сохраняются локально до отправки;
- завершение обхода ставится в очередь отправки;
- после восстановления связи все действия отправляются в правильном порядке.

### 4.4. Маршрут и точки контроля

На телефоне нужно показать:

- название маршрута;
- список точек маршрута;
- рекомендуемый порядок прохождения, если он задан;
- тип точки: NFC, QR, ручная отметка;
- обязательное фото или комментарий;
- статус точки: ожидает, выполнена, пропущена, проблема, не синхронизирована;
- время плановое и фактическое;
- отклонение от плана.

Порядок прохождения:

- сотрудники не обязаны идти строго по порядку точек;
- обход можно начать с любой точки маршрута;
- точку можно открыть и выполнить в любом месте маршрута, если она принадлежит текущему назначению;
- приложение должно показывать, какие точки уже выполнены, какие остались и какие требуют внимания;
- если для конкретного маршрута нужен строгий порядок, это должно быть отдельной настройкой маршрута, а не общим правилом системы;
- завершение обхода разрешено не по порядку, но только после выполнения всех обязательных условий по точкам, фото и комментариям.

Требования к фиксации точки:

- NFC/QR должны валидироваться по коду точки;
- ручная отметка требует причины, если точка должна подтверждаться NFC/QR;
- дата и время факта фиксируются на устройстве и подтверждаются сервером при sync;
- сервер должен хранить признак ручной корректировки и источник события.

Фотофиксация с камеры телефона:

- сотрудник должен фиксировать исправное состояние точки/оборудования через камеру телефона, если это требуется правилами маршрута;
- сотрудник может фиксировать неисправность через камеру телефона, но по умолчанию фото для "проблема" или "неисправность" не обязательно;
- обязательность фото для исправности или неисправности включается только настройкой маршрута/точки;
- фото должно привязываться к конкретной точке маршрута, заявке, сотруднику, устройству и времени съемки;
- для неисправности сотрудник должен указать тип неисправности, комментарий и при необходимости критичность;
- для исправного состояния достаточно статуса "исправно", если маршрут не требует комментарий или фото;
- приложение должно запрещать завершение точки, если обязательное фото не сделано;
- нельзя выбирать старое фото из галереи для обязательной фиксации, если бизнес-правило требует именно камеру;
- фото сохраняется локально до подтвержденной загрузки сервером;
- после синхронизации оператор в web-панели должен видеть фото исправности/неисправности в карточке точки и в отчете обхода.

### 4.5. Учет работ / ЭМУ

Мобильное приложение должно получать задачи из модуля ЭМУ / "Учет работ".

Сотрудник должен видеть:

- активные работы, назначенные ему;
- новые задачи, ожидающие принятия;
- участок;
- тип работы;
- описание задачи;
- приоритет;
- плановое время;
- состав участников, если задача групповая;
- статус: новая, принята, в работе, пауза, завершена, отменена.

Основные действия:

- принять задачу;
- начать работу;
- поставить на паузу с причиной;
- продолжить работу;
- завершить работу;
- указать результат;
- указать причину невыполнения;
- добавить комментарий;
- добавить фото/файл, если требуется;
- увидеть историю своих действий по задаче.

Оффлайн:

- полученные задачи ЭМУ доступны без сети;
- начало, пауза, продолжение и завершение пишутся в локальную очередь;
- если сервер отклоняет действие из-за конфликта, пользователь получает понятное сообщение;
- нельзя потерять локально внесенный результат работы.

### 4.6. Синхронизация без лишних экранов

Обычному сотруднику не нужен отдельный технический экран синхронизации. Приложение должно работать максимально просто: сотрудник выполняет обход, а синхронизация идет в фоне.

В пользовательском интерфейсе показывать только короткие статусы там, где они нужны:

- на карточке задания: "сохранено на телефоне", "отправляется", "принято сервером", "нужно исправить";
- на фото: "ожидает загрузки", "загружено", "ошибка загрузки";
- при отсутствии связи: "нет сети, данные сохраняются на телефоне";
- при конфликте: понятное действие, например "оператор изменил задание, откройте обновление".

Технический журнал sync хранить локально, но не показывать обычному пользователю. Доступ к деталям оставить только для скрытого режима поддержки или web-панели администратора.

## 5. Offline-first модель

### 5.1. Локальная база

На устройстве хранить в SQLite:

- профиль пользователя;
- справочник маршрутов и точек;
- назначения на обход;
- результаты обходов и отметки точек;
- задачи ЭМУ;
- справочники ЭМУ;
- локальные уведомления;
- outbox действий;
- metadata файлов и фото;
- sync cursors.

Не хранить в открытом виде:

- пароль;
- refresh token;
- production secrets;
- лишние персональные данные, не нужные сотруднику.

### 5.2. Outbox

Каждое действие пользователя пишется в локальный outbox:

- `clientOperationId`;
- тип действия;
- payload;
- время создания на устройстве;
- связанная сущность;
- статус отправки;
- число попыток;
- последняя ошибка;
- server acknowledgement.

Все команды должны быть идемпотентными. Повторная отправка одного `clientOperationId` не должна создавать дубль.

### 5.3. Конфликты

Backend является источником истины для:

- прав пользователя;
- актуального состава маршрутов;
- назначений;
- статусов задач после принятия сервером;
- справочников.

Телефон является временным источником фактов, пока нет связи:

- отметка точки;
- фото;
- комментарий;
- начало/пауза/продолжение/завершение работы;
- результат выполнения.

Если конфликт нельзя решить автоматически, действие переводится в статус `conflict`, пользователь видит причину, оператор видит конфликт в web-панели.

### 5.4. Обязательное условие оффлайн-прохождения обхода

Сотрудник может пройти обход без интернета только если заявка/назначение уже были получены и сохранены на телефон.

Перед переходом задания в режим оффлайн на устройстве должны быть сохранены:

- assignment id;
- route id и snapshot маршрута;
- список точек маршрута в правильном порядке;
- правила прохождения точек;
- обязательность фото/комментариев;
- доступные причины пропуска или ручной отметки;
- текущий статус задания на момент загрузки;
- server revision/version задания.

Если заявка не была получена до потери связи, приложение не должно создавать обход "из воздуха". Оно может показать только уже сохраненные задания.

### 5.5. Протокол синхронизации

Синхронизация должна работать по принципу "сначала надежная локальная запись, потом отправка".

Порядок действий:

1. Пользователь выполняет действие: старт обхода, отметка точки, фото, комментарий, завершение обхода.
2. Приложение в одной SQLite-транзакции обновляет локальное состояние и пишет команду в outbox.
3. Интерфейс сразу показывает действие как сохраненное локально.
4. Sync worker отправляет команды на сервер в порядке создания и с учетом зависимостей.
5. Сервер принимает команду, проверяет `clientOperationId`, права, статус задания и revision.
6. Сервер возвращает подтверждение, новый server state, server timestamp и новый sync cursor.
7. Приложение помечает команду как `synced` и обновляет локальные данные.

Если сеть пропала во время отправки, команда остается в outbox и повторяется позже. Повторная отправка той же команды не должна создавать дубли на сервере.

При появлении интернета приложение должно быстро подключиться к серверу и начать отправку без действий пользователя.

Требования к быстрому возобновлению sync:

- отслеживать восстановление сети через NetInfo;
- при появлении сети сразу запускать легкий ping/health check API;
- после успешного ping запускать отправку outbox;
- сначала отправлять маленькие текстовые события: старт, отметки точек, комментарии, статусы;
- затем отправлять фото и файлы маленькими пакетами;
- после загрузки файлов отправлять финальное завершение отчета;
- если сеть снова пропала, не сбрасывать прогресс, а продолжить с последнего подтвержденного server acknowledgement;
- при повторном открытии приложения всегда проверять outbox и пытаться отправить незавершенный отчет;
- не ждать фоновой задачи ОС, если приложение открыто: sync должен стартовать сразу.

Приоритет отправки:

1. команды завершенных отчетов по обходам;
2. незагруженные фото к завершенным отчетам;
3. текущие действия по активным обходам;
4. задачи ЭМУ;
5. вторичные справочники и несрочные обновления.

Главный критерий: если отчет уже завершен локально и интернет появился, приложение должно начать отправку в течение нескольких секунд после определения доступности API.

### 5.6. Статусы outbox

Для каждой локальной команды хранить статус:

- `pending` - ожидает отправки;
- `sending` - отправляется сейчас;
- `synced` - подтверждена сервером;
- `retry` - временная ошибка, будет повтор;
- `failed` - постоянная ошибка, нужно действие пользователя или оператора;
- `conflict` - сервер не принял действие из-за изменения задания/маршрута/статуса;
- `cancelled` - действие отменено пользователем до отправки, если это разрешено бизнес-правилами.

Повторы отправки делать с backoff: сначала быстро, потом реже. Нельзя бесконечно нагружать сервер при плохой связи.

### 5.7. Правила отправки отчета по обходу

Завершение обхода оффлайн разрешено, если:

- все обязательные точки имеют факт прохождения, пропуск с причиной или проблему;
- обязательные фото сохранены локально;
- все выбранные неисправности имеют описание, а фото - только если сотрудник добавил его или если это требует маршрут/точка;
- все точки, где требуется подтверждение исправности, имеют фото с камеры телефона;
- комментарии и причины заполнены по правилам маршрута;
- приложение записало финальную команду `completePatrolAssignment` в outbox.

При восстановлении связи приложение должно:

1. подключиться к API сразу после появления сети;
2. отправить события старта и прохождения точек;
3. загрузить фото и файлы;
4. связать загруженные файлы с точками;
5. отправить финальное завершение обхода;
6. получить итоговый статус от сервера;
7. показать пользователю "Отчет отправлен" или конкретную причину ошибки.

Если часть фото не загрузилась, завершение обхода не считается полностью синхронизированным. Пользователь должен видеть, что отчет сохранен локально, но еще не принят сервером.

### 5.8. Корректировка отчета и меток

После прохождения точки сотрудник может исправить локальный отчет до успешной отправки на сервер:

- изменить комментарий;
- добавить или заменить фото;
- исправить причину ручной отметки;
- отметить проблему;
- добавить пояснение к пропущенной точке.

После успешной синхронизации исправление должно идти отдельной командой `amendPatrolPointReport` или `amendPatrolReport`, чтобы в audit log было видно, что именно изменили и когда.

Нельзя бесследно перезаписывать уже принятый сервером факт. Все исправления должны храниться как история изменений.

Корректировка фото неисправности или исправности после серверного принятия также должна идти отдельной записью истории: кто заменил/добавил фото, когда и по какой причине.

### 5.9. Техническая схема надежной синхронизации

Рекомендуемая схема для первого релиза: собственный sync protocol поверх текущего ASP.NET Core API и PostgreSQL.

Причины:

- проще встроить в текущую доменную модель обходов и ЭМУ;
- не нужна отдельная внешняя sync-инфраструктура;
- можно явно контролировать бизнес-правила, audit log и конфликты;
- легче обеспечить совместимость с текущим сайтом.

Альтернативу вроде готового realtime/offline sync движка можно рассмотреть позже, если появится много двусторонних таблиц и сложная совместная работа. Для обходов и учета работ важнее надежная очередь команд и понятные серверные правила.

## 6. Backend API для mobile

Нужно добавить отдельный слой mobile endpoints, не ломая текущий web API.

Минимальный набор:

```text
POST /api/v1/mobile/auth/login
POST /api/v1/mobile/auth/refresh
POST /api/v1/mobile/auth/logout

POST /api/v1/mobile/devices/register
POST /api/v1/mobile/devices/unregister

GET  /api/v1/mobile/bootstrap
GET  /api/v1/mobile/sync?cursor=...
POST /api/v1/mobile/outbox
GET  /api/v1/mobile/outbox/{clientOperationId}

GET  /api/v1/mobile/patrol/assignments
GET  /api/v1/mobile/patrol/assignments/{id}
GET  /api/v1/mobile/patrol/request-board
POST /api/v1/mobile/patrol/request-board/{id}/take
POST /api/v1/mobile/patrol/events
POST /api/v1/mobile/patrol/amendments

GET  /api/v1/mobile/emu/tasks
GET  /api/v1/mobile/emu/tasks/{id}
POST /api/v1/mobile/emu/events

POST /api/v1/mobile/files
GET  /api/v1/mobile/files/{id}/status
GET  /api/v1/mobile/notifications
POST /api/v1/mobile/notifications/{id}/read
```

`bootstrap` должен вернуть минимальный набор данных для первого оффлайн-старта:

- текущий пользователь;
- employee binding;
- permissions;
- активные назначения на обход;
- маршруты и точки, нужные текущему пользователю;
- активные задачи ЭМУ;
- справочники ЭМУ, нужные для выполнения;
- текущий server time;
- sync cursor.

`outbox` должен принимать пакет команд:

- `clientOperationId`;
- `entityType`;
- `entityId`;
- `commandType`;
- `payload`;
- `clientCreatedAt`;
- `deviceId`;
- `baseRevision`;
- список зависимостей от файлов или предыдущих команд.

Ответ сервера по каждой команде:

- `accepted`;
- `duplicate`;
- `rejected`;
- `conflict`;
- `serverOperationId`;
- `serverTimestamp`;
- `newRevision`;
- `message`;
- обновленный фрагмент server state, если он нужен приложению.

Сервер обязан хранить обработанные `clientOperationId`, чтобы повторная отправка после обрыва сети возвращала тот же результат, а не создавала новое событие.

`request-board` должен возвращать заявки, доступные конкретному сотруднику:

- request id;
- название маршрута;
- ФИО закрепленного сотрудника или `null`, если заявка свободная;
- плановое время начала;
- статус заявки;
- краткий комментарий оператора;
- признак, может ли текущий сотрудник взять заявку.

`take` должен быть атомарной серверной операцией. Если заявка уже взята другим сотрудником, сервер возвращает отказ без изменения локального состояния второго устройства.

## 7. Уведомления на backend

Нужно ввести server-side notification pipeline.

События, которые создают уведомления:

- создана заявка на обход с `notifyEmployee = true`;
- создано назначение;
- назначение изменено;
- назначение отменено;
- назначение просрочено;
- создана задача ЭМУ для сотрудника;
- задача ЭМУ изменена;
- задача ЭМУ требует внимания;
- конфликт синхронизации.

Каждое уведомление хранить в БД:

- id;
- userId / employeeId;
- type;
- title;
- body;
- entityType;
- entityId;
- payloadJson;
- status: pending, sent, failed, read;
- createdAt;
- sentAt;
- readAt.

Push доставка не заменяет запись в БД. Если push не пришел, уведомление остается доступным при следующем sync.

## 8. Работа локально в сети предприятия

Приложение должно поддерживать выбор API endpoint:

- production/stable адрес;
- локальный LAN адрес, например `https://192.168.x.x`;
- dev адрес для тестов.

Требования:

- API base URL задается через build profile или настройки администратора;
- для локального HTTPS нужен доверенный сертификат на телефоне;
- если сертификат не доверен, приложение должно показывать понятную ошибку подключения;
- локальный режим не должен отключать авторизацию и аудит.

## 9. Подробные функциональные сценарии

### 9.1. Вход в приложение

Цель: сотрудник должен быстро войти в приложение и получить свои задания.

Предусловия:

- у сотрудника есть учетная запись в системе;
- телефон имеет доступ к серверу хотя бы для первого входа;
- выбран правильный адрес API: production, локальная сеть или тестовый.

Действия пользователя:

1. Открывает приложение.
2. Видит окно входа.
3. Выбирает сервер, если сервер не выбран автоматически.
4. Вводит логин и пароль.
5. Нажимает "Войти".

Что делает приложение:

- проверяет доступность API;
- отправляет логин/пароль на mobile auth endpoint;
- получает access token, refresh token, профиль, роли и привязку к сотруднику;
- сохраняет токены в secure storage;
- регистрирует устройство и push token;
- выполняет `bootstrap`;
- сохраняет в SQLite активные заявки, назначения, маршруты, точки, задачи ЭМУ и справочники.

Что видит пользователь:

- при успехе: главный экран с заданиями;
- при отсутствии сети до первого входа: сообщение "Для первого входа нужен интернет";
- при неверном пароле: сообщение "Неверный логин или пароль";
- при недоступном сервере: сообщение "Сервер недоступен, проверьте сеть или адрес сервера".

### 9.2. Оффлайн-вход после первой авторизации

Цель: сотрудник может открыть приложение без интернета и продолжить уже полученные задания.

Предусловия:

- сотрудник ранее успешно входил онлайн;
- локальная сессия не истекла;
- на телефоне есть сохраненные задания.

Действия пользователя:

1. Открывает приложение без сети.
2. Нажимает "Продолжить оффлайн" или проходит PIN/биометрию, если включено.

Что делает приложение:

- проверяет локальную сессию;
- открывает сохраненные задания;
- не запрашивает сервер до появления сети;
- все новые действия пишет в SQLite и outbox.

Ограничения:

- нельзя получить новые заявки без ранее выполненной синхронизации;
- нельзя впервые войти в систему без сети;
- если сервер ранее заблокировал пользователя, оффлайн-доступ должен быть отключен после ближайшей успешной синхронизации.

### 9.3. Главный экран "Мои задания"

Цель: пользователь сразу видит рабочие действия без технических экранов.

На экране показывать:

- активные заявки на обход;
- заявки, взятые с доски;
- задачи ЭМУ;
- короткие уведомления;
- незавершенные отчеты, если они есть.

Не показывать:

- техническую очередь outbox;
- подробный журнал sync;
- служебные ошибки API;
- лишние графики и панели.

Статусы должны быть простыми:

- "новая";
- "принята";
- "в работе";
- "сохранено на телефоне";
- "отправляется";
- "принято сервером";
- "нужно исправить".

### 9.4. Доска заявок на обход

Цель: сотрудник может взять доступную заявку на обход.

На доске показывать:

- название маршрута;
- ФИО закрепленного сотрудника или "свободная";
- плановое время начала;
- статус заявки;
- краткий комментарий оператора;
- признак срочности или просрочки.

Действия пользователя:

1. Открывает доску заявок.
2. Выбирает заявку.
3. Открывает карточку заявки.
4. Нажимает "Взять заявку".

Что делает приложение:

- отправляет `POST /api/v1/mobile/patrol/request-board/{id}/take`;
- передает `clientOperationId` и `deviceId`;
- после успешного ответа сохраняет назначение, маршрут и точки в SQLite;
- делает заявку доступной для оффлайн-прохождения.

Что делает сервер:

- проверяет права сотрудника;
- проверяет, что заявка не отменена и доступна;
- атомарно закрепляет заявку за сотрудником;
- если заявку уже взял другой сотрудник, возвращает отказ без дублей.

Что видит пользователь:

- при успехе: "Заявка взята";
- если заявка уже занята: "Заявку уже взял другой сотрудник";
- если нет сети: "Нельзя взять новую заявку без связи с сервером".

### 9.5. Открытие карточки обхода

Цель: сотрудник понимает, что нужно пройти и какие условия обязательны.

Карточка обхода показывает:

- номер заявки;
- маршрут;
- плановое время начала;
- статус;
- комментарий оператора;
- список точек;
- обязательные фото;
- обязательные комментарии;
- сколько точек выполнено и сколько осталось;
- есть ли несинхронизированные действия.

Доступные действия:

- начать обход;
- открыть любую точку;
- продолжить ранее начатый обход;
- завершить обход, если выполнены обязательные условия;
- посмотреть простую причину ошибки, если отчет не принят.

### 9.6. Старт обхода

Цель: зафиксировать начало работы.

Действия пользователя:

1. Открывает карточку обхода.
2. Нажимает "Начать обход".

Что делает приложение:

- создает локальное событие `startPatrolAssignment`;
- записывает событие в SQLite и outbox одной транзакцией;
- ставит обход в статус "в работе";
- если есть сеть, сразу пытается отправить событие на сервер.

Оффлайн:

- старт разрешен, если назначение уже сохранено на телефоне;
- пользователь видит "сохранено на телефоне";
- сервер получит старт после восстановления связи.

### 9.7. Выбор точки маршрута

Цель: сотрудник может начать с любой точки, потому что фактическая работа не всегда идет по порядку.

Правила:

- точки можно проходить в любом порядке;
- приложение показывает рекомендуемый порядок, но не блокирует выбор другой точки;
- строгий порядок возможен только как отдельная настройка конкретного маршрута;
- завершение обхода проверяет полноту, а не последовательность.

Действия пользователя:

1. Открывает маршрут.
2. Видит список точек и их статусы.
3. Выбирает любую доступную точку.

Что делает приложение:

- открывает карточку выбранной точки;
- показывает требования точки: QR/NFC, фото, комментарий, причины пропуска;
- не заставляет возвращаться к предыдущей точке, если маршрут не строгий.

### 9.8. Фиксация исправности точки

Цель: подтвердить, что точка/оборудование в нормальном состоянии.

Действия пользователя:

1. Открывает точку.
2. Выбирает статус "исправно".
3. Сканирует QR/NFC, если требуется.
4. Делает фото с камеры телефона, если фото обязательно.
5. Добавляет комментарий, если это требуется правилом точки.
6. Нажимает "Сохранить".

Что делает приложение:

- проверяет обязательные поля;
- сохраняет фото локально;
- создает preview фото для слабых устройств;
- записывает событие `markPatrolPointOk` в outbox;
- показывает статус "сохранено на телефоне" или "принято сервером".

Сервер после sync:

- проверяет принадлежность точки назначению;
- проверяет QR/NFC код, если он есть;
- принимает фото и metadata;
- обновляет статус точки в отчете.

### 9.9. Фиксация неисправности

Цель: зафиксировать проблему с доказательной базой.

Действия пользователя:

1. Открывает точку.
2. Выбирает статус "неисправность" или "проблема".
3. Делает фото с камеры телефона, если нужно или если это требует маршрут/точка.
4. Выбирает тип неисправности.
5. Указывает критичность, если это включено.
6. Пишет комментарий.
7. Нажимает "Сохранить".

Обязательные данные:

- тип неисправности;
- комментарий;
- точка маршрута;
- время фиксации;
- сотрудник;
- устройство.

Что делает приложение:

- не разрешает сохранить неисправность без комментария и типа неисправности;
- если фото обязательно по правилам маршрута/точки, не разрешает сохранить без фото;
- сохраняет фото локально, если сотрудник его сделал;
- пишет `markPatrolPointIssue` в outbox;
- если нет сети, показывает "сохранено на телефоне";
- не удаляет фото до подтверждения сервером.

Сервер после sync:

- принимает событие и фото;
- создает запись неисправности в отчете;
- показывает проблему оператору в web-панели;
- сохраняет audit log.

### 9.10. Пропуск точки

Цель: разрешить рабочий обход, если точку невозможно пройти, но сохранить причину.

Действия пользователя:

1. Открывает точку.
2. Выбирает "пропустить".
3. Выбирает причину.
4. Добавляет комментарий.
5. При необходимости делает фото.
6. Нажимает "Сохранить".

Правила:

- пропуск без причины запрещен;
- если точка обязательна и пропуск запрещен настройками маршрута, завершить обход нельзя;
- если пропуск разрешен, отчет должен явно показывать пропущенную точку и причину.

### 9.11. Добавление фото

Цель: надежно сохранить фото даже без интернета.

Действия пользователя:

1. Нажимает кнопку камеры.
2. Делает фото.
3. Подтверждает или переснимает.

Что делает приложение:

- сохраняет оригинал во временное локальное хранилище;
- создает легкую preview-копию;
- связывает фото с точкой и локальным событием;
- пишет metadata: localFileId, pointId, assignmentId, deviceId, capturedAt;
- ставит фото в очередь upload.

Ограничения:

- для обязательной фиксации использовать камеру, а не старое фото из галереи;
- не загружать все фото одновременно;
- не удалять локальный файл до подтверждения сервера.

### 9.12. Завершение обхода

Цель: сотрудник закрывает обход после выполнения всех обязательных условий.

Действия пользователя:

1. Нажимает "Завершить обход".
2. Видит проверку обязательных условий.
3. Исправляет незаполненные точки, если они есть.
4. Подтверждает завершение.

Что проверяет приложение:

- все обязательные точки выполнены, пропущены с разрешенной причиной или отмечены как проблема;
- обязательные фото сделаны;
- неисправности имеют тип, критичность и комментарий;
- локальные файлы доступны на устройстве;
- финальная команда может быть записана в outbox.

Что делает приложение:

- записывает `completePatrolAssignment` в outbox;
- переводит отчет в статус "сохранено на телефоне";
- при наличии сети сразу запускает отправку.

### 9.13. Быстрая отправка отчета после появления интернета

Цель: завершенный отчет должен уйти на сервер как можно быстрее.

Триггеры sync:

- приложение открылось;
- пользователь завершил обход;
- сеть восстановилась;
- пользователь потянул список вниз для обновления;
- пришел silent push;
- сработала фоновая задача ОС.

Порядок отправки:

1. Проверить API легким ping.
2. Отправить события обхода.
3. Отправить события точек.
4. Загрузить фото маленькими пакетами.
5. Привязать файлы к точкам.
6. Отправить финальное завершение.
7. Получить итоговый статус.

Что видит пользователь:

- без ошибки: только статус "отправляется" и затем "принято сервером";
- при временной ошибке: "отчет сохранен, отправим при появлении сети";
- при конфликте: понятное сообщение, что нужно исправить или дождаться оператора.

### 9.14. Исправление отчета до отправки

Цель: сотрудник может исправить локально сохраненные данные, пока сервер их не принял.

Разрешено:

- изменить комментарий;
- заменить фото;
- добавить фото;
- исправить тип неисправности;
- изменить критичность;
- исправить причину пропуска.

Что делает приложение:

- обновляет локальный отчет;
- обновляет или добавляет команду в outbox;
- сохраняет историю локальных изменений.

### 9.15. Исправление отчета после принятия сервером

Цель: после server acknowledgement исправления должны быть прозрачными и аудируемыми.

Правила:

- нельзя бесследно перезаписывать принятый факт;
- любое исправление идет отдельной командой amendment;
- сервер хранит историю: кто изменил, что изменил, когда и почему;
- оператор видит изменения в web-панели.

### 9.16. Получение уведомления

Цель: сотрудник узнает о новой заявке или задаче, но push не является единственным источником данных.

Действия:

1. Push приходит на телефон.
2. Пользователь нажимает уведомление.
3. Приложение открывает нужную заявку, назначение или задачу.

Что делает приложение:

- открывает экран по `entityType` и `entityId`;
- если сущности еще нет локально, запускает sync;
- если push не пришел, заявка все равно появится после обычной синхронизации.

### 9.17. Выполнение задачи ЭМУ

Цель: сотрудник может выполнять задачи из "Учет работ" на телефоне.

Действия пользователя:

1. Открывает задачу ЭМУ.
2. Нажимает "Принять", если задача новая.
3. Нажимает "Начать".
4. При необходимости ставит на паузу с причиной.
5. Продолжает работу.
6. Завершает задачу.
7. Добавляет результат, комментарий и фото, если требуется.

Что делает приложение:

- каждое действие пишет в SQLite и outbox;
- позволяет работать оффлайн, если задача уже была получена;
- после sync обновляет статус задачи в web-панели.

### 9.18. Обработка конфликта

Цель: данные не теряются, даже если сервер не может принять действие автоматически.

Примеры конфликтов:

- заявка отменена оператором;
- маршрут изменен после загрузки на телефон;
- точка удалена или заменена;
- задача уже закрыта другим ответственным;
- права сотрудника изменились.

Что делает приложение:

- не удаляет локальные данные;
- переводит действие в статус "нужно исправить";
- показывает короткое понятное сообщение;
- отправляет детали конфликта в web-панель для оператора.

### 9.19. Выход из приложения

Цель: безопасно завершить сессию.

Действия пользователя:

1. Открывает профиль.
2. Нажимает "Выйти".
3. Подтверждает действие.

Что делает приложение:

- при наличии сети вызывает logout endpoint;
- деактивирует push token;
- очищает токены из secure storage;
- очищает пользовательские данные или блокирует доступ к ним;
- не удаляет незавершенный отчет без явного решения администратора.

### 9.20. Сквозная логика работы приложения

Цель: заранее определить, как приложение должно работать внутри, чтобы при разработке не смешивать экранную логику, SQLite, API и синхронизацию.

Основной принцип:

- экран вызывает сценарий feature-модуля;
- сценарий проверяет бизнес-правила;
- изменение сначала записывается локально в SQLite;
- команда добавляется в outbox;
- sync engine отправляет команды на сервер;
- ответ сервера обновляет локальную базу;
- UI только читает актуальное локальное состояние.

Запрещено:

- отправлять важное действие только напрямую в API без локальной записи;
- хранить состояние обхода только в React state;
- удалять фото до server acknowledgement;
- смешивать HTTP-запросы с JSX-компонентами;
- показывать пользователю технические статусы outbox.

### 9.21. Логика запуска приложения

Порядок запуска:

1. Инициализировать конфигурацию приложения и выбранный server profile.
2. Открыть SQLite.
3. Выполнить локальные миграции.
4. Проверить secure storage и локальную сессию.
5. Проверить состояние сети.
6. Если есть активная сессия, открыть `Мои задания`.
7. Если есть сеть, запустить bootstrap/sync.
8. Если есть незавершенный outbox, запустить отправку.
9. Если сессии нет, открыть экран авторизации.

Правило: приложение должно быть способно открыть сохраненные задания даже если сервер недоступен.

### 9.22. Логика записи действия пользователя

Любое рабочее действие сохраняется по одному шаблону:

1. Пользователь нажимает кнопку.
2. UI вызывает command handler.
3. Command handler валидирует действие.
4. SQLite-транзакция обновляет локальную сущность.
5. В той же транзакции создается outbox command.
6. UI сразу показывает новый локальный статус.
7. Sync engine получает сигнал и пытается отправить команду, если есть сеть.

Примеры действий:

- взять заявку;
- начать обход;
- отметить точку исправной;
- зафиксировать неисправность;
- пропустить точку;
- завершить обход;
- принять задачу ЭМУ;
- поставить задачу ЭМУ на паузу;
- завершить задачу ЭМУ.

### 9.23. Логика быстрой отправки отчета

Завершенный отчет имеет высший приоритет синхронизации.

Триггеры:

- отчет завершен локально;
- приложение открыто;
- интернет появился;
- пользователь вернулся в приложение;
- пришел silent push;
- пользователь нажал `Повторить отправку`.

Порядок:

1. Проверить доступность API.
2. Найти завершенные локальные отчеты.
3. Отправить команды обхода и точек.
4. Загрузить фото по очереди или маленькими пакетами.
5. Привязать фото к точкам.
6. Отправить `completePatrolAssignment`.
7. Получить итоговое подтверждение.
8. Показать `Принято сервером`.

Если сеть пропала:

- остановить текущую отправку без потери прогресса;
- оставить команды в outbox;
- продолжить с последнего подтвержденного шага после восстановления связи.

### 9.24. Логика работы с сетью

Приложение должно различать:

- нет сети;
- есть сеть, но сервер недоступен;
- сервер доступен, но авторизация истекла;
- сервер доступен, но вернул ошибку;
- локальная сеть предприятия доступна;
- production сервер доступен.

Поведение:

- при отсутствии сети не блокировать уже полученные задания;
- при появлении сети запускать ping/health check;
- после успешного ping запускать sync;
- если токен истек, выполнить refresh;
- если refresh невозможен, показать необходимость повторного входа;
- если сервер недоступен, сохранить действия локально.

### 9.25. Логика outbox

Outbox является обязательным для всех рабочих команд.

Команда outbox содержит:

- `clientOperationId`;
- `commandType`;
- `entityType`;
- `entityId`;
- `payloadJson`;
- `baseRevision`;
- `deviceId`;
- `createdAtLocal`;
- `status`;
- `attemptCount`;
- `lastError`;
- `serverOperationId`;
- `serverAcknowledgedAt`;

Правила:

- команды отправляются в порядке зависимостей;
- текстовые команды идут раньше фото, если это не ломает бизнес-правило;
- финальное завершение отчета идет после загрузки обязательных фото;
- повторная отправка одного `clientOperationId` не создает дубль;
- после постоянной ошибки команда переводится в `failed` или `conflict`.

### 9.26. Логика файлов и фото

Фото хранится отдельно от событий.

Порядок:

1. Камера сохраняет локальный файл.
2. Приложение создает preview.
3. В SQLite создается запись файла.
4. Файл связывается с точкой и локальным событием.
5. Upload queue отправляет файл.
6. Сервер возвращает `fileId`.
7. Приложение связывает `fileId` с точкой.
8. После принятия отчета и истечения срока хранения локальный файл можно удалить.

Правила:

- если фото обязательно, финальное завершение отчета не отправлять до успешной загрузки фото;
- если фото необязательное, отчет может быть принят по правилам backend;
- если места на телефоне мало, предупредить до съемки, а не после потери фото.

### 9.27. Логика конфликтов

Конфликт не должен приводить к потере данных.

Порядок:

1. Сервер возвращает conflict response.
2. Sync engine сохраняет conflict record.
3. Локальная команда получает статус `conflict`.
4. UI показывает `Нужно исправить`.
5. Пользователь видит короткое объяснение.
6. Оператор видит детали в web-панели.

Пользовательские сообщения должны быть рабочими:

- `Заявка изменена оператором`;
- `Маршрут обновлен`;
- `Точка больше недоступна`;
- `Заявка отменена`;
- `Нужно повторно открыть задание`.

### 9.28. Логика обновления данных с сервера

Sync с сервера не должен ломать локально начатую работу.

Правила:

- server state обновляет справочники, заявки и маршруты;
- локальные несинхронизированные действия имеют приоритет в UI до ответа сервера;
- если сервер прислал новую revision задания, а локально есть outbox, нужно проверить conflict rules;
- отмененные задания не удалять мгновенно, если по ним есть локальный незагруженный отчет;
- после успешного sync показывать пользователю только актуальный рабочий статус.

### 9.29. Логика прав доступа

Права проверяются на телефоне для удобства, но окончательное решение принимает сервер.

На телефоне:

- скрывать недоступные действия;
- блокировать взятие заявки без прав;
- блокировать редактирование принятого отчета без amendment;
- блокировать первый вход без сети.

На сервере:

- повторно проверять все права;
- проверять привязку сотрудника;
- проверять статус заявки;
- проверять доступ к маршруту и задаче ЭМУ;
- писать audit log.

### 9.30. Логика слабого устройства

Если устройство слабое или мало памяти:

- включить экономный режим;
- уменьшить количество элементов в списках;
- не открывать тяжелые preview;
- сжимать фото агрессивнее;
- отправлять файлы по одному;
- отключить декоративные анимации;
- очищать старые принятые фото по политике хранения;
- не выполнять тяжелый sync во время съемки фото.

### 9.31. Логика NFC-сканирования

Цель: подтвердить прохождение точки через NFC-метку, если точка или маршрут требуют NFC.

Предусловия:

- устройство поддерживает NFC;
- NFC включен в настройках телефона;
- приложение собрано как development/production build с native NFC-модулем;
- точка маршрута имеет зарегистрированный `nfcTagId` или защищенный код метки;
- назначение и точка уже сохранены на телефоне.

Порядок действия пользователя:

1. Открывает точку маршрута.
2. Нажимает `Приложить NFC`.
3. Приложение открывает окно чтения NFC.
4. Пользователь прикладывает телефон к метке.
5. Приложение считывает идентификатор/данные метки.
6. Приложение сверяет метку с ожидаемой точкой.
7. Если метка совпала, точка получает подтверждение NFC.
8. Пользователь выбирает результат точки: `Исправно`, `Неисправность` или `Пропустить`, если это разрешено.
9. Приложение сохраняет событие локально и добавляет команду в outbox.

Что сохраняется локально:

- assignmentId;
- pointId;
- scannedNfcTagId или hash/код метки;
- expectedNfcTagId/hash;
- результат проверки: matched/mismatched;
- время чтения на устройстве;
- deviceId;
- employeeId;
- признак ручной корректировки, если NFC не считан и выбран ручной режим;
- комментарий/причина, если NFC не совпал или не считался.

Правила:

- если NFC обязателен и метка не считана, точку нельзя отметить как обычную NFC-проверку;
- если разрешена ручная отметка вместо NFC, сотрудник обязан указать причину;
- если считана чужая метка, приложение должно показать `Метка не соответствует этой точке`;
- чужая метка не должна сохранять точку как успешно пройденную;
- факт попытки чтения чужой метки можно сохранить в audit, если это требуется контролем;
- сервер повторно проверяет NFC-код при sync;
- NFC-событие должно работать оффлайн, если справочник меток для маршрута уже загружен.

Ошибки NFC:

- устройство не поддерживает NFC;
- NFC выключен;
- метка не читается;
- метка не относится к текущей точке;
- метка относится к другому маршруту;
- чтение отменено пользователем;
- NFC-модуль недоступен в текущей сборке приложения.

Для MVP NFC входит в первый релиз через feature flag `mobile.nfc.enabled`. QR остается резервным способом подтверждения точки и используется, если NFC выключен для маршрута, временно недоступен на устройстве или не прошел приемочные тесты на реальных метках.

### 9.32. Рабочий поток обхода территории по NFC-меткам

Цель: сделать основной сценарий максимально быстрым для сотрудника: выбрать заявку, сканировать метки, отметить состояние, заполнить комментарий/фото сразу или отложить, затем отправить отчет.

Основной порядок:

1. Пользователь открывает главный экран.
2. Нажимает основную кнопку `Выбрать заявку на обход`.
3. Открывается окно выбора заявок.
4. Пользователь выбирает заявку.
5. Приложение открывает экран активного обхода.
6. Пользователь нажимает `Сканировать NFC`.
7. После успешного сканирования открывается экран выбора статуса метки.
8. Пользователь выбирает `Исправно` или `Неисправно`.
9. После выбора статуса открывается форма заполнения метки.
10. Пользователь добавляет комментарий.
11. Пользователь добавляет фото, если нужно или если фото обязательно.
12. Пользователь нажимает `Сохранить` или `Отложить на потом`.
13. Приложение обновляет прогресс прохождения меток.
14. Пользователь повторяет сканирование остальных меток.
15. Пользователь открывает вкладку `Все метки`, чтобы проверить полный список.
16. Пользователь нажимает `Отправить отчет`.
17. Приложение проверяет заполненность меток.
18. Если есть отложенные метки, приложение блокирует отправку и показывает список отложенных меток.
19. Если все обязательные данные заполнены, пользователь подтверждает отправку.
20. Отчет сохраняется локально и отправляется сразу или после появления интернета.

Правила:

- после сканирования NFC пользователь должен сразу попасть на выбор статуса;
- статус `Исправно` и `Неисправно` выбирается до заполнения комментария и фото;
- фото может быть обязательным или выборочным по настройкам маршрута/точки;
- `Отложить на потом` сохраняет факт сканирования метки, но помечает метку как незаполненную;
- отложенная метка не считается полностью готовой к отправке;
- отчет нельзя отправить, если есть действующие отложенные обязательные метки;
- вкладка `Все метки` показывает все метки маршрута, их статусы и что осталось заполнить;
- прогресс-бар считается по полностью заполненным меткам, а не только по отсканированным.

Статусы метки в этом сценарии:

- `Ожидает сканирования`;
- `Отсканирована`;
- `Отложена`;
- `Исправно`;
- `Неисправно`;
- `Нужно заполнить`;
- `Готова к отчету`;
- `Сохранено на телефоне`;
- `Принято сервером`.

Логика отложенной метки:

- сохраняется pointId;
- сохраняется NFC-факт;
- сохраняется время сканирования;
- сохраняется сотрудник и устройство;
- статус метки становится `Отложена`;
- пользователь может позже открыть метку из вкладки `Все метки`;
- после заполнения комментария/фото метка переходит в `Готова к отчету`;
- если метка отложена и обязательна, отправка отчета блокируется.

## 10. Экраны, модалки, статусы, кнопки и порядок работы

### 10.1. Общие принципы интерфейса

Интерфейс должен быть простым рабочим инструментом для сотрудника.

Правила:

- не добавлять технические экраны для обычного пользователя;
- основные действия должны быть доступны в 1-2 нажатия;
- кнопки должны быть крупными и понятными для работы в цеху/на объекте;
- экран не должен быть перегружен графиками, журналами и служебными деталями;
- все технические детали sync скрывать, кроме простых рабочих статусов;
- на слабых устройствах интерфейс должен оставаться быстрым.

Основная нижняя навигация:

- `Мои задания`;
- `Доска заявок`;
- `Учет работ`;
- `Уведомления`;
- `Профиль`;
- `Настройки`.

Если нужно упростить первый MVP, `Уведомления` и `Настройки` можно открывать из `Профиль`, а не держать отдельными пунктами нижней навигации.

### 10.2. Экран авторизации

Назначение: вход сотрудника в мобильное приложение.

Поля:

- логин;
- пароль;
- выбор сервера;
- версия приложения;
- индикатор доступности сервера.

Кнопки:

- `Войти`;
- `Продолжить оффлайн`, если локальная сессия уже есть;
- `Настройки сервера`;
- `Показать/скрыть пароль`.

Состояния:

- сервер доступен;
- сервер недоступен;
- нет интернета;
- неверный логин или пароль;
- нужна новая версия приложения;
- локальная сессия истекла.

Порядок работы:

1. Пользователь вводит логин и пароль.
2. Приложение проверяет сервер.
3. При успешном входе получает токены.
4. Выполняет первичную синхронизацию.
5. Открывает экран `Мои задания`.

### 10.3. Модалка выбора сервера

Назначение: выбрать, к какому API подключаться.

Поля:

- основной сервер;
- локальный сервер предприятия;
- тестовый сервер;
- ручной адрес API, если разрешено администратором.

Кнопки:

- `Проверить подключение`;
- `Сохранить`;
- `Отмена`.

Правила:

- обычному пользователю лучше показывать готовые варианты, а не заставлять вводить URL;
- ручной ввод адреса можно скрыть в режиме администратора;
- при ошибке сертификата показывать понятное сообщение.

### 10.4. Главный экран "Мои задания"

Назначение: быстро начать обход территории и показать сотруднику, что нужно сделать сейчас.

Блоки:

- основная кнопка `Выбрать заявку на обход`;
- активный обход;
- новые назначенные обходы;
- незавершенные отчеты;
- задачи ЭМУ;
- срочные уведомления.

Главная кнопка:

- `Выбрать заявку на обход` открывает окно выбора заявок;
- если уже есть активный обход, дополнительно показывать кнопку `Продолжить обход`;
- если есть незавершенный локальный отчет, показывать карточку `Отчет ожидает отправки`;
- если есть отложенные метки, показывать карточку `Есть отложенные метки`.

Карточка обхода показывает:

- название маршрута;
- плановое время начала;
- статус;
- количество точек: выполнено/всего;
- количество отложенных меток;
- есть ли неисправности;
- простой sync-статус.

Кнопки на карточке:

- `Открыть`;
- `Продолжить`;
- `Начать`;
- `Сканировать NFC`;
- `Все метки`;
- `Отправить`, если отчет завершен локально и нужна повторная отправка;
- `Исправить`, если есть ошибка или конфликт.

Статусы на карточке:

- `Новая`;
- `Принята`;
- `В работе`;
- `Есть отложенные метки`;
- `Сохранено на телефоне`;
- `Отправляется`;
- `Принято сервером`;
- `Нужно исправить`;
- `Просрочено`;
- `Отменено`.

Порядок работы:

1. Пользователь открывает приложение.
2. Нажимает `Выбрать заявку на обход`.
3. Выбирает заявку в открывшемся окне.
4. Переходит в активный обход.
5. Сканирует NFC-метки и заполняет статусы.
6. Возвращается на главный экран и видит прогресс/итоговый статус.

### 10.5. Экран "Доска заявок"

Назначение: сотрудник берет доступную заявку на обход.

В мобильном сценарии выбора обхода этот экран может открываться как отдельный экран или как окно поверх главного экрана по кнопке `Выбрать заявку на обход`.

Фильтры:

- все доступные;
- свободные;
- закрепленные за мной;
- срочные;
- просроченные.

Карточка заявки показывает:

- название маршрута;
- ФИО закрепленного сотрудника или `Свободная`;
- плановое время начала;
- статус;
- краткий комментарий оператора;
- метку срочности.

Кнопки:

- `Открыть`;
- `Взять заявку`;
- `Выбрать`;
- `Обновить`.

Статусы:

- `Свободная`;
- `Закреплена`;
- `Принята`;
- `В работе`;
- `Завершена`;
- `Просрочена`;
- `Отменена`.

Порядок работы:

1. Пользователь открывает `Доска заявок`.
2. Выбирает заявку.
3. Проверяет маршрут и время.
4. Нажимает `Взять заявку`.
5. Сервер закрепляет заявку.
6. Приложение загружает маршрут и точки.
7. Заявка появляется в `Мои задания`.

### 10.6. Модалка взятия заявки

Назначение: подтвердить взятие заявки перед закреплением.

Содержит:

- название маршрута;
- плановое время начала;
- ФИО текущего пользователя;
- предупреждение, что заявка будет закреплена за сотрудником.

Кнопки:

- `Взять`;
- `Отмена`.

Ошибки:

- `Заявку уже взял другой сотрудник`;
- `Заявка отменена`;
- `Нет связи с сервером`;
- `Недостаточно прав`.

### 10.7. Окно выбора заявок на обход

Назначение: быстро выбрать заявку с главного экрана.

Открывается:

- по кнопке `Выбрать заявку на обход` на главном экране;
- после push-уведомления о новой заявке, если пользователь подтверждает открытие.

Показывает:

- список доступных заявок;
- ФИО закрепленного сотрудника или `Свободная`;
- название маршрута;
- плановое время начала;
- статус;
- срочность;
- краткий комментарий.

Кнопки:

- `Выбрать`;
- `Взять заявку`;
- `Обновить`;
- `Закрыть`.

Правила:

- если заявка свободная, `Взять заявку` закрепляет ее за сотрудником;
- если заявка уже закреплена за текущим сотрудником, `Выбрать` открывает активный обход;
- если нет сети, можно открыть только уже сохраненные на телефоне заявки;
- после выбора приложение загружает snapshot маршрута и меток.

### 10.8. Экран карточки обхода

Назначение: показать маршрут, точки и прогресс обхода.

Содержит:

- название маршрута;
- номер заявки;
- плановое время начала;
- статус обхода;
- комментарий оператора;
- прогресс точек;
- список точек;
- прогресс-бар прохождения меток;
- количество отложенных меток;
- список обнаруженных неисправностей;
- простой статус отправки отчета.

Кнопки:

- `Начать обход`;
- `Продолжить обход`;
- `Сканировать NFC`;
- `Все метки`;
- `Завершить обход`;
- `Открыть точку`;
- `Обновить`, если есть сеть;
- `Отправить отчет`, если отчет завершен локально.

Правила:

- если обход не начат, кнопка `Завершить обход` скрыта;
- если есть незаполненные обязательные точки, завершение блокируется;
- точки можно открывать в любом порядке;
- строгий порядок применяется только если это задано в настройке маршрута.

### 10.9. Экран маршрута и списка точек

Назначение: выбрать любую точку для прохождения и контролировать, все ли метки заполнены.

Показывать:

- прогресс-бар прохождения меток;
- счетчик: выполнено/всего;
- счетчик отложенных меток;
- список точек;
- рекомендуемый номер точки;
- название точки;
- тип подтверждения: QR, NFC, вручную;
- статус точки;
- признак обязательного фото;
- признак неисправности;
- признак несинхронизированных данных.
- признак отложенной метки.

Кнопки:

- `Открыть точку`;
- `Сканировать NFC`;
- `Сканировать QR`;
- `Фильтр`;
- `Все метки`;
- `Показать невыполненные`;
- `Показать неисправности`;
- `Показать отложенные`.

Статусы точки:

- `Ожидает`;
- `Отсканирована`;
- `Отложена`;
- `Исправно`;
- `Неисправность`;
- `Нужно заполнить`;
- `Готова к отчету`;
- `Пропущена`;
- `Сохранено на телефоне`;
- `Отправляется`;
- `Принято сервером`;
- `Нужно исправить`.

Прогресс-бар:

- считает только полностью заполненные метки;
- отсканированная, но отложенная метка не считается выполненной;
- цвет/статус прогресса должен показывать, есть ли неисправности или отложенные метки;
- нажатие на счетчик отложенных меток открывает фильтр `Показать отложенные`.

### 10.10. Вкладка "Все метки"

Назначение: дать сотруднику полный список меток маршрута и возможность спокойно заполнить отложенные метки.

Показывать:

- все метки маршрута;
- название метки;
- рекомендуемый номер;
- статус NFC/QR;
- выбранный статус: исправно/не исправно/пропущено;
- наличие комментария;
- наличие фото;
- признак отложенной метки;
- признак обязательной метки;
- признак готовности к отчету.

Фильтры:

- `Все`;
- `Ожидают сканирования`;
- `Отложенные`;
- `Не заполнены`;
- `Неисправности`;
- `Готовые`.

Кнопки:

- `Открыть метку`;
- `Заполнить`;
- `Сканировать NFC`;
- `Отправить отчет`;

Правила:

- вкладка доступна из активного обхода;
- отложенные метки должны быть визуально заметны;
- нажатие на отложенную метку открывает форму заполнения комментария/фото;
- если все обязательные метки заполнены, кнопка `Отправить отчет` доступна;
- если есть отложенные обязательные метки, кнопка `Отправить отчет` открывает предупреждение.

### 10.11. Экран точки маршрута

Назначение: зафиксировать состояние конкретной точки.

Содержит:

- название точки;
- описание;
- требования: QR/NFC/ручная отметка;
- требования по фото;
- статус точки;
- статус NFC-сканирования;
- поле комментария;
- блок фото;
- блок выбора результата.

Основные кнопки:

- `Исправно`;
- `Неисправность`;
- `Пропустить`;
- `Сканировать QR`;
- `Приложить NFC`;
- `Сделать фото`;
- `Сохранить`;
- `Отложить на потом`;
- `Назад к маршруту`.

Правила:

- если выбрано `Неисправность`, комментарий обязателен, фото по умолчанию необязательно;
- если выбрано `Исправно`, фото по умолчанию необязательно;
- для `Исправно` и `Неисправность` обязательность фото включается только настройкой точки/маршрута;
- если выбрано `Пропустить`, причина обязательна;
- старое фото из галереи не использовать для обязательной фиксации, если правило требует камеру;
- после сохранения точка сразу получает локальный статус.
- `Отложить на потом` разрешено после сканирования метки, если сотрудник не может спокойно заполнить комментарий/фото на месте;
- отложенная метка должна быть видна во вкладке `Все метки`;
- отложенная обязательная метка блокирует отправку отчета.

### 10.12. Модалка выбора результата точки

Назначение: быстро выбрать состояние точки после сканирования NFC/QR или ручного открытия точки.

Варианты:

- `Исправно`;
- `Не исправно`;
- `Пропустить`.

После выбора:

- `Исправно` открывает блок фото/комментария по правилам точки;
- `Не исправно` открывает форму неисправности;
- `Пропустить` открывает форму причины пропуска.

Правила:

- после успешного NFC-сканирования это окно открывается автоматически;
- выбор статуса идет до комментария и фото;
- пользователь может закрыть окно и оставить метку как `Отсканирована`, но для отчета она будет считаться незаполненной;
- после выбора статуса пользователь переходит к заполнению комментария и фото.

### 10.13. Модалка/экран фиксации неисправности

Назначение: подробно описать проблему.

Поля:

- тип неисправности;
- критичность;
- комментарий;
- фото с камеры;
- дополнительное фото, если нужно;
- признак `требует срочного внимания`, если разрешено бизнес-правилами.

Кнопки:

- `Сделать фото`;
- `Переснять`;
- `Сохранить неисправность`;
- `Отложить на потом`;
- `Отмена`.

Валидация:

- без фото сохранить нельзя;
- без комментария сохранить нельзя;
- без типа неисправности сохранить нельзя.
- если сотрудник нажал `Отложить на потом`, NFC-факт остается сохраненным, но метка получает статус `Отложена`.

### 10.14. Камера

Назначение: фотофиксация исправности и неисправности.

Кнопки:

- `Снять`;
- `Переснять`;
- `Использовать фото`;
- `Назад`.

После снимка:

- приложение сохраняет оригинал локально;
- создает preview;
- привязывает фото к точке;
- ставит фото в очередь upload.

Ограничения:

- камера должна открываться быстро;
- фото не должно зависать интерфейс;
- при нехватке памяти показывать понятное сообщение;
- на слабых устройствах не открывать несколько фото в полном размере одновременно.

### 10.15. Модалка пропуска точки

Назначение: сохранить причину, почему точка не пройдена.

Поля:

- причина пропуска;
- комментарий;
- фото, если требуется;

Кнопки:

- `Сохранить пропуск`;
- `Отмена`.

Правила:

- причина обязательна;
- если пропуск запрещен настройкой маршрута, кнопка не показывается;
- отчет должен явно показывать пропущенную точку.

### 10.16. Модалка завершения обхода

Назначение: проверить отчет перед закрытием.

Показывает:

- выполнено точек;
- осталось обязательных точек;
- количество отложенных меток;
- количество неисправностей;
- количество фото;
- есть ли незаполненные комментарии;
- будет ли отчет отправлен сразу или сохранен на телефоне.

Кнопки:

- `Завершить обход`;
- `Вернуться к точкам`;
- `Отмена`.

Если есть ошибки:

- показывать список, что нужно исправить;
- кнопку `Завершить обход` блокировать;
- дать быстрый переход к проблемной точке.

Если есть отложенные метки:

- показывать текст `У вас есть отложенные метки. Заполните их перед отправкой отчета`;
- показывать список отложенных меток;
- давать кнопку `Открыть отложенные`;
- кнопку `Завершить обход` или `Отправить отчет` блокировать, если отложенные метки обязательны.

### 10.17. Экран/карточка отправки отчета

Правила:

- если есть отложенные метки, отправку блокировать и показывать список;

### 10.24. Настройки

Назначение: персонализация, внешний вид, аккаунт, подключение и служебные действия.

Раздел `Уведомления`:

Кнопки:

Правила:

- настройки темы и размера текста применяются сразу;
- технические параметры не показывать обычному пользователю без необходимости;
- смена сервера требует подтверждения, если есть незавершенные отчеты;
- выход или смена аккаунта не должны удалять незагруженные отчеты без явного решения администратора;
- при смене аккаунта локальные данные предыдущего пользователя должны быть недоступны новому пользователю.

### 10.25. Модалка смены аккаунта

Назначение: безопасно перейти на другого пользователя.

Показывает:

- текущий пользователь;
- есть ли незагруженные отчеты;
- есть ли незагруженные фото;
- что произойдет после смены аккаунта.

Кнопки:

- `Сменить аккаунт`;
- `Сначала отправить данные`;
- `Отмена`.

Правила:

- если есть незагруженный отчет, по умолчанию предлагать сначала отправить данные;
- если сети нет, смену аккаунта лучше блокировать или требовать права администратора;
- данные одного сотрудника не должны показываться другому;
- после смены аккаунта приложение возвращается на экран авторизации.

### 10.26. Модалка выхода

Назначение: подтвердить выход и не потерять локальные данные.

Показывает:

- текущий пользователь;
- статус незавершенных отчетов;
- предупреждение, если есть данные, не принятые сервером.

Кнопки:

- `Выйти`;
- `Сначала отправить данные`;
- `Отмена`.

Правила:

- если все данные синхронизированы, выход очищает токены и локальную сессию;
- если есть незагруженные отчеты, приложение не должно молча удалять данные;
- при выходе push token деактивируется на сервере, если есть сеть;
- если сети нет, деактивация push token ставится в очередь или выполняется при следующем входе.

### 10.27. Модалка выбора темы

### 10.30. Основной порядок работы сотрудника

Типовой порядок работы:

1. Сотрудник входит в приложение.
2. На главном экране нажимает `Выбрать заявку на обход`.
3. В окне выбора заявок берет или открывает заявку.
4. Нажимает `Начать обход`.
5. Нажимает `Сканировать NFC`.
6. После сканирования выбирает `Исправно` или `Не исправно`.
7. Заполняет комментарий.
8. Добавляет фото по необходимости; обязательность фото действует только по правилу маршрута/точки.
9. Нажимает `Сохранить` или `Отложить на потом`.
10. Повторяет действия по остальным меткам.
11. Открывает `Все метки` и проверяет, что все обязательные метки заполнены.
12. Нажимает `Отправить отчет`.
13. Приложение показывает подтверждение и проверяет отложенные метки.
14. Если есть отложенные обязательные метки, приложение просит заполнить их.
15. Если все заполнено, отчет сохраняется локально и отправляется.
16. Если сеть есть, отчет отправляется сразу.
17. Если сети нет, отчет сохраняется на телефоне.
18. При появлении интернета приложение отправляет отчет автоматически.
19. Пользователь видит `Принято сервером` или понятное действие для исправления.

### 10.31. Окно запроса разрешений

Назначение: запросить доступы только тогда, когда они нужны.

Разрешения:

- камера;
- уведомления;
- NFC, если включено;
- геолокация, если включена настройками предприятия.

Правила:

- не запрашивать все разрешения при первом запуске;
- камеру запрашивать при первом открытии съемки;
- уведомления запрашивать после успешного входа;
- геолокацию запрашивать только если маршрут требует GPS;
- если разрешение отклонено, показать понятное объяснение и кнопку перехода в настройки.

Кнопки:

- `Разрешить`;
- `Позже`;
- `Открыть настройки`.

### 10.32. Экран/модалка QR-сканера

Назначение: подтвердить точку через QR.

Кнопки:

- `Сканировать`;
- `Ввести вручную`, если разрешено;
- `Назад`;
- `Включить фонарик`.

Состояния:

- QR найден;
- QR не относится к этой точке;
- QR поврежден или не читается;
- камера недоступна;
- ручной ввод требует причины.

Правила:

- неправильный QR не должен сохранять точку как выполненную;
- при ручном обходе QR-проблемы нужна причина;
- результат сканирования пишется в событие точки.

### 10.33. Окно NFC

Назначение: подтвердить точку через NFC, если это включено.

Содержит:

- название ожидаемой точки;
- инструкцию приложить телефон к метке;
- статус чтения.
- тип проверки: обязательная NFC или дополнительная NFC;
- результат сверки метки.

Кнопки:

- `Повторить`;
- `Ручная отметка`;
- `Отмена`.
- `Открыть настройки NFC`, если NFC выключен.

Состояния:

- NFC прочитан;
- метка не совпадает с точкой;
- NFC отключен;
- устройство не поддерживает NFC;
- ошибка чтения.
- чтение отменено;
- NFC недоступен в этой сборке приложения;
- ручная отметка ожидает причины.

Правила:

- NFC можно вынести во второй релиз;
- если NFC обязателен, ручная отметка требует причины;
- факт ручной отметки должен быть виден оператору.
- если метка совпала, приложение возвращает пользователя на экран точки и показывает `NFC подтвержден`;
- если метка совпала, приложение автоматически открывает окно выбора статуса `Исправно` / `Не исправно`;
- если метка не совпала, точку нельзя сохранить как подтвержденную NFC;
- если NFC не поддерживается устройством, приложение должно предложить ручную отметку только если это разрешено правилами маршрута;
- NFC-событие пишется в outbox как часть отчета точки;
- сервер должен видеть способ подтверждения: NFC, QR или ручная отметка.

Пользовательские сообщения:

- `Поднесите телефон к NFC-метке`;
- `NFC подтвержден`;
- `Метка не соответствует этой точке`;
- `NFC выключен на телефоне`;
- `Телефон не поддерживает NFC`;
- `Укажите причину ручной отметки`.

### 10.34. Окно preview фото

Назначение: сотрудник проверяет снимок перед сохранением.

Показывает:

- фото;
- точку маршрута;
- тип фиксации: исправность или неисправность;
- предупреждение, если фото размыто, если такая проверка будет добавлена позже.

Кнопки:

- `Использовать фото`;
- `Переснять`;
- `Удалить`;

Правила:

- после `Использовать фото` файл сохраняется локально;
- после `Переснять` старый неподтвержденный временный файл можно удалить;
- удаление обязательного фото блокирует сохранение точки, пока не сделано новое фото.

### 10.35. Модалка ошибки сети

Назначение: объяснить, что работа продолжится локально.

Показывать только когда это влияет на действие пользователя.

Тексты:

- `Нет интернета. Данные сохраняются на телефоне`;
- `Сервер недоступен. Отчет отправится позже`;
- `Подключение восстановлено. Отправляем отчет`;
- `Отчет принят сервером`.

Кнопки:

- `Понятно`;
- `Повторить`, если действие требует немедленного сервера;
- `Настройки сервера`, если проблема в endpoint.

Правила:

- не показывать модалку при каждом мелком обрыве сети;
- для обычной оффлайн-работы достаточно компактного статуса на карточке;
- модалка нужна только при входе, взятии новой заявки, конфликте или постоянной ошибке.

### 10.36. Модалка конфликта

Назначение: объяснить, что сервер не принял действие автоматически.

Содержит:

- короткую причину;
- что сохранено на телефоне;
- что нужно сделать дальше;
- кнопку перехода к проблемной сущности.

Кнопки:

- `Открыть`;
- `Обновить задание`;
- `Понятно`.

Типовые тексты:

- `Заявка была изменена оператором`;
- `Маршрут обновлен`;
- `Заявка отменена`;
- `Эта точка больше не доступна`;
- `Отчет сохранен, требуется проверка оператора`.

### 10.37. Модалка нехватки памяти

Назначение: не дать потерять фото или отчет на слабом устройстве.

Показывать:

- сколько примерно места нужно;
- что можно сделать;
- что уже сохранено.

Кнопки:

- `Повторить`;
- `Открыть настройки телефона`;
- `Продолжить без фото`, только если фото необязательное;
- `Отмена`.

Правила:

- если фото обязательное, нельзя завершить точку без фото;
- приложение должно предупредить до съемки, если места явно недостаточно;
- синхронизированные старые фото можно очистить автоматически по политике хранения.

### 10.38. Модалка обязательного обновления

Назначение: не дать старой версии приложения работать с несовместимым backend protocol.

Показывать:

- текущую версию приложения;
- минимальную требуемую версию;
- короткую причину.

Кнопки:

- `Обновить`;
- `Выйти`.

Правила:

- если версия несовместима, sync не запускать;
- если есть локальный незагруженный отчет, показать, что данные сохранены на телефоне, но для отправки нужно обновление;
- backend должен возвращать признак `minimumSupportedMobileVersion`.

### 10.39. Модалка истекшей сессии

Назначение: безопасно восстановить авторизацию.

Показывать:

- `Сессия истекла`;
- `Для отправки данных нужен повторный вход`;
- статус локально сохраненных отчетов.

Кнопки:

- `Войти снова`;
- `Продолжить оффлайн`, если это разрешено;

Правила:

- локальные отчеты не удалять;
- отправку на сервер не выполнять без действительного токена;
- после повторного входа sync должен продолжиться.

### 10.40. Пустые состояния экранов

Для каждого пустого списка нужен понятный текст.

Примеры:

- `Сегодня нет назначенных обходов`;
- `Нет доступных заявок`;
- `Нет задач учета работ`;
- `Нет уведомлений`;
- `Все точки выполнены`;
- `Нет неисправностей`.

Кнопки:

- `Обновить`;
- `Открыть доску заявок`, если нет своих обходов;
- `Проверить подключение`, если данные не загружены.

### 10.41. Состояния загрузки

Загрузки должны быть легкими и не мешать работе.

Использовать:

- компактный индикатор в карточке;
- skeleton только для первого открытия списка;
- текст `Обновляем данные`;
- текст `Отправляем отчет`;
- текст `Загружаем фото`.

Не использовать:

- полноэкранные блокировки для обычного sync;
- тяжелые анимации;
- бесконечные индикаторы без текста.

### 10.42. Соответствие технических и пользовательских статусов

```text
outbox.pending       -> Сохранено на телефоне
outbox.sending       -> Отправляется
outbox.retry         -> Отправим позже
outbox.synced        -> Принято сервером
outbox.failed        -> Ошибка отправки
outbox.conflict      -> Нужно исправить
file.localOnly       -> Фото сохранено
file.queued          -> Фото ожидает отправки
file.uploading       -> Фото загружается
file.uploaded        -> Фото загружено
file.failed          -> Ошибка загрузки фото
point.scanned        -> Отсканирована
point.deferred       -> Отложена
point.needsInput     -> Нужно заполнить
point.ready          -> Готова к отчету
nfc.waiting          -> Поднесите телефон к NFC-метке
nfc.matched          -> NFC подтвержден
nfc.mismatched       -> Метка не соответствует этой точке
nfc.disabled         -> NFC выключен на телефоне
nfc.unsupported      -> Телефон не поддерживает NFC
network.offline      -> Нет интернета
network.serverDown   -> Сервер недоступен
auth.expired         -> Нужно войти снова
version.unsupported  -> Нужно обновить приложение
```

В UI использовать только правую колонку. Левая колонка остается для кода, логов и backend.

### 10.43. Релизный интерфейс MVP

В первый релиз входит только интерфейс, который нужен сотруднику для выполнения обхода территории. Лишние технические экраны, расширенная аналитика и администрирование не включаются.

Основной пользовательский поток первого релиза:

```text
Вход -> Главный экран -> Выбрать заявку -> Активный обход -> Сканировать NFC -> Выбрать статус метки -> Заполнить комментарий/фото -> Сохранить или Отложить -> Все метки -> Отправить отчет
```

Нижняя навигация первого релиза:

- `Обход`;
- `Все метки`;
- `Учет работ`;
- `Профиль`.

`Настройки` открываются из `Профиль`, а не отдельной вкладкой в нижней навигации.

`Уведомления` в первом релизе показываются внутри `Обход` и `Профиль`, отдельную вкладку можно добавить позже.

### 10.44. Экран релиза: "Обход"

Это первый экран после входа.

На экране:

- большая основная кнопка `Выбрать заявку на обход`;
- карточка активного обхода, если обход уже выбран;
- кнопка `Сканировать NFC`, если обход активен;
- прогресс-бар меток;
- счетчики: `заполнено`, `отложено`, `неисправности`;
- карточка `Отчет ожидает отправки`, если отчет завершен локально;
- короткий статус сети: `онлайн`, `нет сети`, `сервер недоступен`.

Основные кнопки:

- `Выбрать заявку на обход`;
- `Продолжить обход`;
- `Сканировать NFC`;
- `Все метки`;
- `Отправить отчет`.

Правила:

- если обход не выбран, главная кнопка только `Выбрать заявку на обход`;
- если обход выбран, кнопка `Сканировать NFC` становится основной;
- если есть отложенные метки, показывать заметную строку `Есть отложенные метки`;
- если отчет завершен локально, но не отправлен, показывать `Отчет сохранен на телефоне`.

### 10.45. Окно релиза: выбор заявки

Открывается по кнопке `Выбрать заявку на обход`.

Показывает:

- название маршрута;
- ФИО сотрудника, если заявка закреплена;
- плановое время начала;
- статус;
- комментарий оператора.

Кнопки:

- `Выбрать`;
- `Взять`;
- `Обновить`;
- `Закрыть`.

Правила:

- если заявка свободная, сотрудник нажимает `Взять`;
- если заявка уже закреплена за ним, нажимает `Выбрать`;
- если нет сети, можно открыть только уже сохраненную заявку;
- после выбора сразу открывается активный обход.

### 10.46. Экран релиза: активный обход

Экран активного обхода должен быть максимально простым.

Показывает:

- название маршрута;
- плановое время начала;
- прогресс-бар меток;
- количество заполненных меток;
- количество отложенных меток;
- количество неисправностей;
- последнюю отсканированную метку;
- простую подсказку следующего действия.

Кнопки:

- `Сканировать NFC`;
- `Все метки`;
- `Отправить отчет`;
- `Назад`.

Правила:

- `Сканировать NFC` основная кнопка;
- `Отправить отчет` доступна только после проверки обязательных условий;
- если есть отложенные обязательные метки, `Отправить отчет` открывает предупреждение;
- прогресс считается по полностью заполненным меткам.

### 10.47. Экран релиза: сканирование NFC

Показывает:

- текст `Поднесите телефон к NFC-метке`;
- название текущего обхода;
- индикатор ожидания;
- результат чтения.

Состояния:

- ожидание метки;
- метка считана;
- метка не относится к маршруту;
- NFC выключен;
- устройство не поддерживает NFC;
- ошибка чтения.

Кнопки:

- `Отмена`;
- `Повторить`;
- `Ручная отметка`, если разрешено правилами маршрута;
- `Открыть настройки NFC`, если NFC выключен.

После успешного сканирования:

- сохранить факт сканирования локально;
- открыть экран выбора статуса метки;
- не требовать интернет для продолжения.

### 10.48. Экран релиза: выбор статуса метки

Открывается автоматически после успешного NFC-сканирования.

Показывает:

- название метки;
- подтверждение `NFC подтвержден`;
- две основные кнопки выбора статуса.

Кнопки:

- `Исправно`;
- `Не исправно`;
- `Отложить на потом`;
- `Назад`.

Правила:

- выбор статуса обязателен для готовности метки к отчету;
- `Отложить на потом` сохраняет NFC-факт, но метка остается незаполненной;
- отложенная обязательная метка блокирует отправку отчета.

### 10.49. Экран релиза: заполнение метки

Открывается после выбора `Исправно` или `Не исправно`.

Для `Исправно`:

- комментарий, если требуется правилами;
- фото, если требуется правилами;
- кнопки `Сохранить`, `Отложить на потом`, `Назад`.

Для `Не исправно`:

- тип неисправности;
- критичность;
- обязательный комментарий;
- фото с камеры, если сотрудник хочет добавить доказательство или если это требует маршрут/точка;
- кнопки `Сохранить`, `Отложить на потом`, `Назад`.

Правила:

- если обязательные поля не заполнены, `Сохранить` неактивна;
- `Отложить на потом` сохраняет текущий черновик;
- сохраненная метка получает статус `Готова к отчету`;
- черновик/отложенная метка видна во вкладке `Все метки`.

### 10.50. Экран релиза: "Все метки"

Назначение: проверить полный список меток и заполнить отложенные.

Показывает:

- список всех меток;
- статус каждой метки;
- признак отложенной метки;
- признак неисправности;
- наличие комментария;
- наличие фото;
- признак готовности к отчету.

Фильтры:

- `Все`;
- `Отложенные`;
- `Не заполнены`;
- `Неисправности`;
- `Готовые`.

Кнопки:

- `Открыть`;
- `Заполнить`;
- `Сканировать NFC`;
- `Отправить отчет`.

Правила:

- отложенные метки должны быть первыми в фильтре `Отложенные`;
- нажатие на отложенную метку открывает заполнение метки;
- если все обязательные метки готовы, `Отправить отчет` доступна;
- если есть отложенные обязательные метки, показывать предупреждение.

### 10.51. Окно релиза: подтверждение отправки отчета

Открывается после нажатия `Отправить отчет`.

Показывает:

- количество заполненных меток;
- количество отложенных меток;
- количество неисправностей;
- количество фото;
- статус сети;
- что будет сделано после подтверждения.

Если все заполнено:

- кнопка `Подтвердить и отправить`;
- кнопка `Отмена`.

Если есть отложенные обязательные метки:

- текст `У вас есть отложенные метки. Заполните их перед отправкой отчета`;
- список отложенных меток;
- кнопка `Открыть отложенные`;
- кнопка `Отмена`;
- кнопка отправки заблокирована.

После подтверждения:

- отчет записывается локально;
- создается `completePatrolAssignment`;
- если сеть есть, sync стартует сразу;
- если сети нет, показывается `Отчет сохранен на телефоне`.

### 10.52. Экран релиза: профиль и настройки

Профиль показывает:

- ФИО;
- роль;
- текущий сервер;
- версию приложения;
- последнее подключение;
- кнопку `Настройки`;
- кнопку `Сменить аккаунт`;
- кнопку `Выйти`.

Настройки первого релиза:

- тема: `Как в системе`, `Светлая`, `Темная`;
- размер текста;
- экономный режим;
- проверить подключение;
- очистить синхронизированные фото;
- сменить PIN, если PIN включен;
- отправить отладочный отчет, только по просьбе поддержки.

Смена аккаунта:

- если есть незагруженный отчет, показывать предупреждение;
- данные одного пользователя не показывать другому;
- по умолчанию предложить сначала отправить данные.

### 10.53. Что не входит в интерфейс первого релиза

В первый релиз не включать:

- отдельный технический экран outbox;
- подробный журнал sync для сотрудника;
- аналитику;
- карту;
- администрирование маршрутов на телефоне;
- создание заявок на телефоне;
- управление пользователями;
- сложные графики;
- постоянный GPS-трекинг;
- редактирование справочников.

Технические детали остаются в скрытом отладочном отчете и web-панели администратора.

## 11. Релиз 1: состав MVP мобильного приложения

В MVP входят:

1. Login/logout.
2. Регистрация устройства для push.
3. Получение уведомлений.
4. Список уведомлений в приложении.
5. Доска заявок на обход.
6. Окно выбора заявок с главного экрана.
7. Взятие свободной заявки с доски.
8. Список назначений на обход.
9. Детальная карточка обхода.
10. Маршрут и точки.
11. Старт обхода с любой точки маршрута.
12. Отметка точки вручную и через QR.
13. NFC-сканирование для точек, где NFC включен настройкой маршрута и устройство поддерживает NFC.
14. Выбор статуса метки после сканирования: исправно / не исправно.
15. Отложить метку на потом.
16. Вкладка `Все метки`.
17. Прогресс-бар прохождения меток.
18. Фотофиксация исправности точки с камеры телефона.
19. Фотофиксация неисправности с камеры телефона.
20. Комментарий, тип неисправности и критичность к точке.
21. Завершение обхода.
22. Проверка отложенных меток перед отправкой отчета.
23. Быстрая отправка завершенного отчета после появления интернета.
24. Список задач ЭМУ.
25. Детальная карточка задачи ЭМУ.
26. Начать, пауза, продолжить, завершить задачу ЭМУ.
27. Локальная SQLite база.
28. Outbox и повторная отправка.
29. Минимальные статусы в заданиях: сохранено локально, отправляется, принято сервером, ошибка.

Если первый MVP нужно стабилизировать быстрее, NFC можно включить feature flag и тестировать только на маршрутах с подготовленными метками. QR оставить обязательно, потому что он проще для тестирования и внедрения.

## 12. Что не входит в первый MVP

- Полный функционал web-панели на телефоне.
- Администрирование пользователей.
- Управление маршрутами с телефона.
- Создание справочников ЭМУ с телефона.
- Массовая аналитика.
- Полноценная бухгалтерия и СИЗ на телефоне.
- Сложная геолокационная аналитика.
- Payroll/зарплата.

## 13. Требования к надежности

- Стабильная отправка завершенного отчета и синхронизация с сервером являются главным приоритетом мобильного приложения.
- Приложение не должно терять локально созданные действия после перезапуска.
- Повторная отправка outbox не должна создавать дубли.
- При плохой сети пользователь должен видеть, что действие сохранено локально.
- При восстановлении связи sync должен продолжиться без ручного вмешательства.
- При появлении интернета приложение должно быстро подключиться к серверу и начать отправку завершенных отчетов.
- Фото не удаляются локально до подтверждения загрузки сервером.
- Серверные ошибки должны показываться пользователю только в понятной рабочей форме, без технических деталей, и записываться в скрытый локальный журнал sync.
- Приложение должно корректно работать при разнице времени телефона и сервера; источник истины при сохранении - серверное время.
- Завершенный оффлайн-обход должен сохраняться на устройстве до подтверждения сервером.
- Если приложение закрыли во время обхода, после открытия оно должно восстановить последний локальный статус.
- Если приложение закрыли во время отправки отчета, sync должен продолжиться после запуска приложения или при разрешенной фоновой синхронизации.
- Если пользователь нажал "Отправить" несколько раз, сервер должен принять отчет один раз.
- Если сеть появилась только частично и фото не загружаются, текстовые события не должны теряться, а отчет должен оставаться в статусе "ожидает полной отправки".
- Push-уведомление не считается единственным каналом доставки задания: все новые задания обязаны приходить через обычный sync.
- Фоновая синхронизация является ускорением, но не гарантией. Основная синхронизация должна запускаться при открытии приложения, при входе, при pull-to-refresh и при восстановлении сети.

## 14. Требования к безопасности

- Все endpoints только по HTTPS, кроме явно разрешенного dev-режима.
- Токены хранить только в защищенном хранилище.
- При logout очищать токены и локальные пользовательские данные.
- При смене пользователя локальная база предыдущего пользователя не должна быть доступна.
- Push token привязывается к пользователю и устройству, при logout деактивируется.
- Все мобильные команды пишут actor, deviceId и clientOperationId в audit.
- Пароль никогда не хранить в SQLite, AsyncStorage или логах.
- Refresh token хранить только в secure storage.
- Для оффлайн-входа использовать локальный PIN/биометрию только после успешной серверной авторизации.
- При блокировке пользователя на сервере следующая успешная синхронизация должна отключить локальный оффлайн-доступ.
- Локальная база должна быть привязана к пользователю и устройству; данные одного сотрудника не должны показываться другому.

## 15. Acceptance criteria первого релиза

Первый релиз мобильного MVP считается готовым, если:

- сотрудник может войти в приложение;
- после входа открывается экран `Обход`;
- на экране `Обход` есть основная кнопка `Выбрать заявку на обход`;
- окно выбора заявок показывает маршрут, ФИО, плановое время начала и статус;
- сотрудник видит доску заявок с ФИО, названием маршрута и плановым временем начала;
- сотрудник может взять доступную заявку с доски, и сервер не допускает двойного взятия одной заявки;
- сотрудник может открыть назначение, увидеть маршрут и точки;
- сотрудник может начать обход с любой точки маршрута;
- сотрудник может проходить точки не по порядку, если маршрут не настроен как строго последовательный;
- сотрудник может подтвердить точку через NFC, если маршрут требует NFC и устройство поддерживает NFC;
- при неправильной NFC-метке точка не сохраняется как подтвержденная;
- после сканирования NFC сотрудник выбирает статус метки `Исправно` или `Не исправно`;
- сотрудник может отложить заполнение метки на потом;
- вкладка `Все метки` показывает полный список меток, отложенные метки и прогресс прохождения;
- отчет нельзя отправить, если есть обязательная отложенная метка;
- прогресс-бар считается по полностью заполненным меткам;
- сотрудник может пройти обход без сети и отправить результат после восстановления связи;
- сотрудник может завершить обход без сети, закрыть приложение, открыть его снова и увидеть сохраненный незагруженный отчет;
- сотрудник может с камеры телефона зафиксировать исправность точки;
- сотрудник может с камеры телефона зафиксировать неисправность, добавить тип, критичность и комментарий;
- фото исправности/неисправности не теряются без сети и загружаются после восстановления связи, если сотрудник их сделал;
- после восстановления связи отчет отправляется на сервер без дублей;
- после появления интернета приложение само подключается к серверу и начинает отправку завершенного отчета без действий пользователя;
- если фото не загрузилось, приложение показывает незавершенную синхронизацию и повторяет отправку;
- оператор в web-панели видит результат обхода после sync;
- сотрудник получает задачу ЭМУ на телефон;
- сотрудник может начать, поставить на паузу, продолжить и завершить задачу ЭМУ;
- web-панель ЭМУ видит актуальный статус после sync;
- повторная отправка одного действия не создает дубль;
- конфликт синхронизации не теряет данные и отображается пользователю;
- локальная сеть предприятия работает через заданный API URL;
- профиль и настройки доступны из приложения;
- тема `Светлая` / `Темная` переключается без перезапуска;
- базовый сценарий покрыт automated tests и ручным чеклистом.

Блокеры релиза:

- отчет можно отправить с обязательной отложенной меткой;
- фото теряется после закрытия приложения;
- повторная отправка создает дубль на сервере;
- приложение не открывает сохраненный обход без интернета;
- неправильная NFC-метка засчитывает точку как пройденную;
- после появления интернета завершенный отчет не начинает отправку автоматически;
- смена аккаунта показывает данные предыдущего пользователя;
- слабое устройство не может пройти базовый сценарий обхода.

### 15.1. Релизный чеклист интерфейса

Перед выпуском проверить:

- после входа открывается экран `Обход`, а не технический экран;
- главная кнопка `Выбрать заявку на обход` видна сразу;
- окно выбора заявки открывается без перехода в лишнюю навигацию;
- активный обход показывает прогресс-бар меток;
- кнопка `Сканировать NFC` является основной в активном обходе;
- после NFC сразу открывается выбор `Исправно` / `Не исправно`;
- после выбора статуса открывается заполнение комментария/фото;
- `Отложить на потом` видна и работает;
- вкладка `Все метки` показывает отложенные метки;
- кнопка `Отправить отчет` блокируется при обязательных отложенных метках;
- профиль и настройки не перегружают основной сценарий обхода;
- темная и светлая тема не ломают читаемость статусов.

### 15.2. Релизный чеклист оффлайн-режима

Перед выпуском проверить:

- заявка, маршрут и метки доступны после отключения интернета;
- NFC-факт сохраняется без интернета;
- статус метки сохраняется без интернета;
- комментарий сохраняется без интернета;
- фото сохраняется без интернета;
- отложенная метка сохраняется после закрытия приложения;
- завершенный отчет сохраняется локально;
- после появления интернета отчет отправляется автоматически;
- повторная отправка не создает дубли;
- пользователь видит только простые статусы, без технического outbox.

### 15.3. Релизный чеклист NFC

Перед выпуском проверить на реальном Android-устройстве:

- NFC включен и приложение читает метку;
- корректная метка подтверждает нужную точку;
- неправильная метка не засчитывает точку;
- повторное сканирование той же метки не создает дубль;
- NFC работает без интернета, если маршрут уже загружен;
- при выключенном NFC показывается понятное сообщение;
- при устройстве без NFC показывается понятное сообщение;
- ручная отметка доступна только если разрешена правилами маршрута.

### 15.4. Релизный чеклист фото

Перед выпуском проверить:

- камера открывается из формы метки;
- фото сохраняется локально;
- preview отображается без зависания;
- фото для `Исправно` и `Не исправно` по умолчанию необязательно;
- обязательное фото проверяется только там, где это включено правилом маршрута/точки;
- фото не теряется после закрытия приложения;
- фото загружается после восстановления интернета;
- локальный файл не удаляется до подтверждения сервера;
- при нехватке памяти приложение предупреждает пользователя.

### 15.5. Релизный чеклист backend

Перед выпуском проверить:

- login/refresh/logout работают;
- device registration работает;
- bootstrap возвращает только нужные данные сотрудника;
- доска заявок фильтруется по правам;
- взятие заявки атомарно;
- outbox idempotency работает по `clientOperationId`;
- upload фото idempotent по `clientFileId`;
- `completePatrolAssignment` отклоняет отчет с обязательными отложенными метками;
- backend возвращает понятные business errors;
- web-панель видит принятый отчет и фото.

### 15.6. Definition of Done первого релиза

Релиз можно считать готовым, когда:

- основной сценарий обхода проходит на реальном Android-устройстве;
- сценарий проходит без интернета после загрузки заявки;
- завершенный отчет отправляется после восстановления интернета;
- NFC проверен на реальных метках;
- камера и фото проверены на слабом устройстве;
- нет блокеров релиза;
- backend не создает дубли при повторной отправке;
- web-панель видит результат отчета;
- сборка APK/AAB установлена и протестирована вне dev-сервера;
- есть короткая инструкция для сотрудников.

## 18. Предложения по улучшению

### 18.1. Надежность без лишних экранов

- Не добавлять отдельный экран "Состояние синхронизации" для обычных сотрудников.
- Показывать sync-статусы только внутри рабочих карточек: задание, точка, фото, отчет.
- Оставить локальный журнал событий приложения скрытым: вход, sync, ошибки сети, ошибки сервера, загрузка фото, конфликты.
- Добавить backend-панель мониторинга мобильной синхронизации для администратора: активные устройства, неотправленные отчеты, последние ошибки, зависшие outbox-команды.
- Добавить server health check в фоне, без отдельного пользовательского экрана.
- Добавить скрытый режим "отладочный отчет" для поддержки: пользователь отправляет технический пакет без паролей и токенов только по просьбе оператора.

### 18.2. Удобство прохождения обхода

- Сделать главный экран "Мои задания сегодня": обходы, задачи ЭМУ, срочные уведомления, незавершенная синхронизация.
- Добавить пошаговый режим прохождения маршрута: следующая точка, статус, камера, QR/NFC, комментарий, дальше.
- Добавить цветовые статусы точек: ожидает, исправно, неисправность, пропущено, ожидает отправки, принято сервером.
- Добавить быстрые шаблоны комментариев для типовых неисправностей.
- Добавить автосохранение формы точки при каждом изменении.
- Добавить запрет случайного завершения обхода: перед завершением показывать список незаполненных обязательных полей.

### 18.3. Фото, видео и доказательная база

- Сжимать фото перед загрузкой, но хранить качество, достаточное для проверки неисправности.
- Хранить оригинал локально до подтверждения сервером, затем очищать по политике хранения.
- Добавлять к фото metadata: точка, заявка, сотрудник, устройство, локальное время, server time после sync, координаты при включенном GPS.
- Добавить водяной знак на фото в отчете: дата, время, точка, сотрудник, номер заявки.
- Добавить серию фото для одной неисправности: общий план, крупный план, итог после исправления.
- Для второго этапа рассмотреть короткое видео по критичным неисправностям, если фото недостаточно.

### 18.4. Геолокация и контроль точки

- Добавить GPS как дополнительное подтверждение прохождения точки, если это разрешено правилами предприятия.
- Хранить координаты только для рабочих событий, а не вести постоянное слежение.
- Добавить допустимый радиус точки: если сотрудник далеко, требовать комментарий или подтверждение оператора.
- Отображать предупреждение, если QR/NFC точки не совпадает с ожидаемой точкой маршрута.

### 18.5. Уведомления и приоритеты

- Разделить уведомления по типам: новая заявка, изменение маршрута, просрочка, конфликт, новая задача ЭМУ.
- Добавить локальные напоминания, если обход начат, но долго не завершен.
- Добавить повторное уведомление по срочным заданиям, если сотрудник не открыл заявку.
- Добавить silent push только как ускоритель sync, но не делать его единственным способом доставки.

### 18.6. Безопасность и доступ

- Добавить PIN/биометрию для быстрого входа после первой серверной авторизации.
- Добавить срок действия оффлайн-сессии, например 24-72 часа по настройке предприятия.
- Добавить remote logout: сервер может принудительно сбросить сессию устройства при увольнении, потере телефона или смене прав.
- Добавить список доверенных устройств в web-панели администратора.
- Добавить шифрование чувствительных локальных данных, если на устройстве будут персональные данные или закрытые документы.

### 18.7. Эксплуатация и обновления

- Добавить обязательную минимальную версию приложения: если backend protocol изменился, старое приложение должно просить обновление.
- Добавить feature flags для поэтапного включения QR, NFC, GPS, обязательных фото и ЭМУ.
- Добавить telemetry без персональных данных: время sync, частота ошибок, средний размер очереди, скорость загрузки фото.
- Добавить отдельные build profile: dev, test, local enterprise, production.
- Добавить ручной offline-checklist для приемки каждого релиза на реальном Android-устройстве.

### 18.8. Интеграция с сайтом

- В web-панели добавить статус мобильного отчета: локально завершен, частично отправлен, ожидает фото, принят сервером, конфликт.
- В карточке обхода на сайте показывать историю мобильных действий по времени.
- В карточке точки показывать фото исправности/неисправности, комментарий, источник фиксации: QR, NFC, вручную.
- В web-панели дать оператору инструмент разбора конфликтов: принять мобильный отчет, отклонить, запросить исправление.
- Добавить фильтр "есть неисправности" и "есть незагруженные фото" для контроля проблемных обходов.

### 18.9. Работа на слабых устройствах

Приложение должно нормально работать на бюджетных Android-устройствах с ограниченной памятью и слабым процессором.

Требования:

- минимизировать количество экранов и переходов;
- не использовать тяжелые анимации, сложные графики, карты и большие списки без необходимости;
- все списки делать виртуализированными и порционными;
- не держать в памяти большие фото, открывать и обрабатывать их по одному;
- сжимать фото перед отправкой и хранить отдельную легкую preview-копию для интерфейса;
- ограничить размер локальной базы и автоматически очищать старые синхронизированные фото по политике хранения;
- загружать с сервера только задания пользователя и нужные справочники, а не весь объем данных;
- bootstrap должен быть небольшим: активные обходы, активные задачи ЭМУ, маршруты для этих заданий, минимальные справочники;
- sync выполнять маленькими пакетами, чтобы приложение не зависало на слабой сети и слабом телефоне;
- тяжелые операции: сжатие фото, загрузка файлов, повторная отправка - выполнять в фоне и не блокировать прохождение обхода;
- не использовать постоянный GPS-трекинг; координаты брать только в момент фиксации точки, если это включено;
- не хранить длинную историю на телефоне, если она уже принята сервером;
- сделать простой интерфейс с крупными кнопками, без лишних панелей и технических экранов.

Целевой минимум для первого релиза:

- Android 9+;
- 3 ГБ RAM как комфортный минимум;
- 2 ГБ RAM как допустимый минимум при урезанных анимациях и ограниченном кеше;
- свободное место на устройстве не менее 500 МБ для фото, SQLite и временных файлов.

Если устройство слабое или мало памяти, приложение должно автоматически включать экономный режим:

- меньше фото в локальном кеше;
- отключение декоративных анимаций;
- более агрессивное сжатие preview;
- отправка файлов меньшими пакетами;
- предупреждение пользователю только если памяти уже недостаточно для сохранения отчета.

## 19. Административные настройки и правила маршрутов

### 19.1. Назначение

Административные настройки маршрута нужны, чтобы сайт, мобильное приложение и backend одинаково понимали правила прохождения обхода.

Главный принцип: мобильное приложение может показывать и предварительно проверять правила, но окончательная валидация всегда выполняется на сервере.

Правила маршрута должны попадать в `bootstrap` и `sync`, чтобы сотрудник мог пройти уже полученный обход без интернета.

### 19.2. Настройки маршрута

В web-панели для каждого маршрута нужны настройки:

- название маршрута;
- активен/неактивен;
- участок/подразделение;
- плановая длительность;
- разрешен свободный порядок точек;
- строгий порядок точек;
- разрешен старт с любой точки;
- разрешен пропуск точки;
- требуется комментарий при пропуске;
- требуется фото при пропуске;
- разрешена ручная отметка;
- требуется причина ручной отметки;
- требовать QR;
- требовать NFC;
- разрешить QR или NFC на выбор;
- разрешить ручную отметку при недоступном QR/NFC;
- требовать фото исправности;
- требовать фото неисправности;
- требовать комментарий при исправности;
- требовать комментарий при неисправности;
- требовать тип неисправности;
- требовать критичность неисправности;
- включить GPS при фиксации точки;
- допустимый радиус GPS, если используется;
- срок хранения фото на телефоне после успешной синхронизации;
- правила просрочки обхода;
- кому доступен маршрут;
- кто может брать заявки по этому маршруту.

### 19.3. Настройки точки маршрута

Для каждой точки маршрута нужны настройки:

- название точки;
- описание;
- порядковый номер;
- активна/неактивна;
- обязательная/необязательная;
- тип подтверждения: `none`, `manual`, `qr`, `nfc`, `qrOrNfc`;
- QR-код точки;
- NFC-код/метка точки;
- разрешена ручная отметка;
- требуется причина ручной отметки;
- требуется фото исправности;
- требуется фото неисправности;
- требуется комментарий при исправности;
- требуется комментарий при неисправности;
- разрешен пропуск;
- список разрешенных причин пропуска;
- требуется фото при пропуске;
- подсказка для сотрудника;
- признаки опасной/критичной точки;
- настройка GPS для этой точки, если отличается от маршрута.

### 19.4. Настройки доски заявок

В web-панели нужно настраивать, какие заявки видит сотрудник.

Параметры:

- видит только свои заявки;
- видит свободные заявки своего участка;
- видит свободные заявки всех доступных участков;
- может брать свободную заявку;
- может отказаться от взятой заявки;
- требуется подтверждение оператора после взятия;
- максимальное количество активных заявок на сотрудника;
- можно ли брать просроченную заявку;
- можно ли брать заявку до планового времени;
- за сколько минут до начала показывать заявку;
- какие роли видят доску заявок.

### 19.5. Настройки фотофиксации

Нужно централизованно задавать правила фото:

- фото исправности обязательно/необязательно;
- фото неисправности обязательно/необязательно;
- фото при пропуске обязательно/необязательно;
- разрешено ли выбирать фото из галереи;
- обязательна ли съемка только с камеры;
- максимальное количество фото на точку;
- минимальное количество фото при неисправности;
- разрешено ли дополнительное фото;
- качество сжатия;
- максимальный размер файла;
- добавлять ли водяной знак;
- какие metadata сохранять;
- срок хранения фото на телефоне после подтверждения сервером.

Для обязательной доказательной фиксации по умолчанию использовать камеру телефона, а не галерею.

### 19.6. Настройки неисправностей

В web-панели нужны справочники:

- типы неисправностей;
- критичность;
- причины пропуска;
- причины ручной отметки;
- типовые комментарии;
- признаки срочного внимания.

Для каждого типа неисправности можно настроить:

- требуется фото;
- требуется комментарий;
- требуется критичность;
- можно ли завершить обход с такой неисправностью;
- нужно ли уведомлять оператора;
- нужно ли создавать задачу ЭМУ автоматически.

### 19.7. Настройки QR/NFC

Для QR:

- хранить код точки;
- показывать код в web-панели;
- печать QR-меток для точек;
- замена QR-кода с историей;
- признак устаревшего QR.

Для NFC:

- хранить идентификатор/код метки;
- привязать NFC-метку к точке;
- заменить NFC-метку с историей;
- проверить NFC-метку администратором;
- признак обязательности NFC;
- разрешить ручной обход NFC с причиной;
- список устройств, поддерживающих NFC.

Backend должен хранить историю смены QR/NFC, чтобы старые отчеты оставались проверяемыми.

### 19.8. Настройки ролей и прав

Нужно описать права:

- кто создает маршруты;
- кто редактирует маршруты;
- кто активирует/деактивирует маршрут;
- кто создает точки;
- кто меняет QR/NFC;
- кто создает заявку на обход;
- кто назначает сотрудника;
- кто может брать свободные заявки;
- кто может делать ручную отметку;
- кто может пропускать обязательную точку;
- кто может исправлять отправленный отчет;
- кто разбирает конфликты;
- кто видит фото и неисправности;
- кто управляет мобильными устройствами.

Права должны проверяться на backend при каждом mobile command.

### 19.9. Передача правил на телефон

`bootstrap` и `sync` должны передавать на телефон:

- настройки маршрута;
- настройки точек;
- справочник причин пропуска;
- справочник причин ручной отметки;
- справочник типов неисправностей;
- правила фото;
- правила QR/NFC;
- правила свободного/строгого порядка;
- доступные действия текущего сотрудника.

Телефон должен сохранять эти правила в SQLite вместе с assignment snapshot.

Если правила маршрута изменились после загрузки на телефон:

- сервер возвращает новую revision;
- sync проверяет, есть ли локальные несинхронизированные действия;
- если изменения конфликтуют с локальным отчетом, создается conflict;
- если не конфликтуют, правила обновляются локально.

### 19.10. Серверная валидация отчета

При приеме отчета backend обязан проверить:

- сотрудник имеет право на заявку;
- заявка не отменена;
- маршрут и точки соответствуют assignment snapshot/revision;
- все обязательные точки закрыты;
- пропуски имеют разрешенные причины;
- обязательные фото загружены;
- неисправности имеют тип, комментарий и критичность, если требуется;
- фото неисправности проверяется только там, где это включено правилом маршрута/точки;
- QR/NFC совпадают с настройками точки;
- ручная отметка разрешена и имеет причину;
- порядок точек соблюден, если маршрут строгий;
- GPS-условия соблюдены, если включены;
- `clientOperationId` не создает дубль.

Если проверка не пройдена:

- временная ошибка переводит команду в retry;
- бизнес-ошибка возвращает понятное сообщение;
- спорное состояние создает conflict;
- локальные данные на телефоне не удаляются.

### 19.11. Web-интерфейс настройки маршрута

В web-панели нужен экран маршрута с вкладками:

- `Основное`;
- `Точки`;
- `Правила прохождения`;
- `Фото`;
- `QR/NFC`;
- `Доступ`;
- `История изменений`.

На вкладке `Основное`:

- название;
- участок;
- активность;
- плановая длительность;
- комментарий.

На вкладке `Точки`:

- список точек;
- порядок;
- обязательность;
- тип подтверждения;
- правила фото;
- QR/NFC;
- подсказка.

На вкладке `Правила прохождения`:

- свободный/строгий порядок;
- старт с любой точки;
- пропуски;
- ручные отметки;
- просрочки.

На вкладке `Фото`:

- обязательность фото;
- качество;
- лимиты;
- водяной знак;
- хранение.

На вкладке `QR/NFC`:

- генерация QR;
- печать QR;
- привязка NFC;
- замена меток;
- проверка меток.

На вкладке `Доступ`:

- роли;
- участки;
- сотрудники;
- кто может брать заявки.

На вкладке `История изменений`:

- кто изменил правило;
- что изменилось;
- когда;
- старая и новая версия.

### 19.12. Версионирование правил

Каждый маршрут и каждая точка должны иметь revision/version.

Правила:

- изменение правил создает новую revision;
- назначение на обход хранит snapshot правил на момент выдачи;
- мобильный отчет отправляет `baseRevision`;
- сервер понимает, по каким правилам сотрудник проходил обход;
- старые отчеты не должны ломаться после изменения маршрута.

Это особенно важно для offline-режима: сотрудник мог пройти обход по правилам, которые были актуальны на момент загрузки задания.

## 20. Оптимизации для эффективной работы

### 20.1. Цель оптимизаций

Приложение должно стабильно работать на слабых Android-устройствах, быстро открывать задания, не терять данные без сети и не мешать сотруднику выполнять обход.

Оптимизации являются обязательной частью архитектуры, а не этапом "когда-нибудь потом".

Главные цели:

- быстрый запуск приложения;
- быстрый переход к активному заданию;
- минимальная нагрузка на память;
- экономное хранение фото;
- быстрая отправка отчета после появления интернета;
- отсутствие лишних технических экранов;
- предсказуемая работа на плохой сети;
- низкое потребление батареи.

### 20.2. Оптимизация запуска приложения

Требования:

- сначала открыть локальную SQLite-базу и показать сохраненные задания;
- не ждать полного sync перед показом главного экрана;
- bootstrap с сервера запускать в фоне, если локальные данные уже есть;
- тяжелые операции не выполнять на старте;
- проверку обновлений и health check делать быстро и с timeout;
- при плохой сети не держать пользователя на загрузочном экране.

Целевое поведение:

- если локальная сессия есть, пользователь должен увидеть `Мои задания` максимально быстро;
- если сервер недоступен, приложение открывает сохраненные задания;
- если есть незавершенный отчет, он должен быть виден сразу.

### 20.3. Оптимизация навигации и экранов

Требования:

- не перегружать нижнюю навигацию лишними пунктами;
- `Уведомления` и `Настройки` можно открыть из `Профиль`, если нижняя навигация перегружена;
- списки делать виртуализированными;
- карточки делать компактными;
- не использовать тяжелые карты, графики и анимации в MVP;
- не перерисовывать весь экран при изменении одного статуса;
- формы сохранять локально при изменениях, чтобы не терять ввод.

UI должен быть оптимизирован под рабочий сценарий:

1. открыть задание;
2. открыть точку;
3. отметить состояние;
4. сделать фото;
5. сохранить;
6. завершить обход.

### 20.4. Оптимизация SQLite

Требования:

- использовать bounded queries, не читать всю базу на экран;
- добавить индексы для `assignmentId`, `routeId`, `pointId`, `status`, `createdAt`, `syncStatus`;
- outbox читать пакетами;
- фото хранить как файлы, а не как blob в SQLite;
- в SQLite хранить только metadata фото;
- все критичные изменения выполнять в транзакциях;
- миграции делать версионированными;
- старые синхронизированные данные очищать по политике хранения.

Локальная база должна быть источником UI-состояния. Экран читает локальные данные, а не ждет сетевой ответ.

### 20.5. Оптимизация outbox и sync

Требования:

- завершенные отчеты имеют высший приоритет отправки;
- outbox отправляется маленькими пакетами;
- при восстановлении сети sync стартует сразу;
- использовать backoff при ошибках сети;
- не делать бесконечные частые повторы при недоступном сервере;
- разделить отправку текстовых событий и фото;
- финальное завершение отчета отправлять после обязательных фото;
- сохранять server acknowledgement после каждого принятого шага;
- продолжать отправку с последнего подтвержденного шага после обрыва.

Приоритет sync:

1. завершенные отчеты обходов;
2. обязательные фото завершенных отчетов;
3. активные действия текущего обхода;
4. задачи ЭМУ;
5. уведомления;
6. справочники и вторичные обновления.

### 20.6. Оптимизация фото

Требования:

- не хранить фото в SQLite;
- создавать preview для интерфейса;
- оригинал хранить локально до подтверждения сервером;
- сжимать фото перед отправкой;
- отправлять фото по одному или маленькими пакетами;
- не открывать несколько больших фото одновременно;
- удалять синхронизированные фото по политике хранения;
- показывать предупреждение при нехватке места;
- не блокировать UI во время сжатия и upload.

Рекомендуемое поведение:

- для интерфейса использовать preview;
- для отправки использовать сжатый файл достаточного качества;
- если фото обязательно и не загружено, отчет остается в статусе ожидания полной отправки.

### 20.7. Оптимизация сети

Требования:

- все сетевые запросы должны иметь timeout;
- повторять только временные ошибки;
- не повторять постоянные бизнес-ошибки;
- использовать компактные DTO;
- не отдавать на телефон лишние маршруты и справочники;
- bootstrap должен возвращать только данные текущего пользователя;
- sync должен поддерживать cursor/revision;
- при слабой сети сначала отправлять маленькие JSON-команды, потом файлы;
- не держать долгие соединения без необходимости.

Плохая сеть не должна ломать рабочий сценарий. Максимум, что видит пользователь: `Данные сохранены на телефоне`.

### 20.8. Оптимизация push-уведомлений

Требования:

- push использовать как ускоритель, а не как единственный канал доставки;
- не отправлять дубли уведомлений на одно устройство;
- silent push может запускать sync, но обычный sync все равно обязателен;
- уведомление должно открывать конкретную сущность;
- если сущность не загружена, приложение сначала запускает sync;
- прочитанные уведомления не должны бесконечно храниться на телефоне.

### 20.9. Оптимизация батареи

Требования:

- не использовать постоянный GPS-трекинг;
- GPS брать только в момент фиксации точки, если включено;
- NFC/QR активировать только на экране точки;
- камеру держать открытой только во время съемки;
- фоновые задачи использовать умеренно;
- sync не должен бесконечно работать при недоступном сервере;
- при низком заряде можно уменьшить частоту фоновых попыток, но не блокировать ручную отправку.

### 20.10. Оптимизация для слабых устройств

Экономный режим включается автоматически или вручную в настройках.

Экономный режим:

- отключает декоративные анимации;
- уменьшает количество одновременно отображаемых элементов;
- использует preview вместо больших фото;
- отправляет фото по одному;
- агрессивнее очищает старые синхронизированные файлы;
- снижает частоту фоновых sync-попыток;
- не запускает тяжелую обработку во время съемки.

Целевые устройства:

- Android 9+;
- комфортно: 3 ГБ RAM;
- допустимо: 2 ГБ RAM;
- свободное место: не менее 500 МБ.

### 20.11. Оптимизация backend API

Требования к backend:

- mobile endpoints должны быть отдельным слоем, не ломая web API;
- запросы должны отдавать только нужные поля;
- поддерживать pagination/cursor;
- поддерживать idempotency по `clientOperationId`;
- поддерживать file upload acknowledgement;
- возвращать понятные business errors;
- логировать mobile audit;
- не отдавать на телефон данные других сотрудников без прав;
- health check должен быть легким.

Для доски заявок:

- отдавать только доступные заявки;
- фильтровать по правам на backend;
- взятие заявки делать атомарно;
- исключить двойное закрепление заявки.

### 20.12. Оптимизация web-панели

Web-панель должна помогать оператору, а не перекладывать проблемы на сотрудника.

Требования:

- показывать конфликты синхронизации;
- показывать отчеты, ожидающие фото;
- показывать устройства с давней синхронизацией;
- показывать заявки, завершенные локально, но еще не принятые сервером;
- быстро фильтровать неисправности;
- открывать фото без загрузки тяжелых оригиналов, используя preview;
- давать оператору понятные действия: принять, отклонить, запросить исправление.

### 20.13. Оптимизация тестирования

Нужно тестировать не только happy path.

Обязательные проверки:

- слабый Android 2 ГБ RAM;
- плохая сеть;
- пропадание сети во время upload фото;
- повторная отправка отчета;
- большой маршрут с большим количеством точек;
- много фото в отчете;
- низкое свободное место;
- неправильная NFC-метка;
- неправильный QR;
- смена аккаунта с незагруженным отчетом;
- обязательное обновление приложения.

### 20.14. Метрики эффективности

Нужно собирать технические метрики без персональных данных:

- время запуска приложения;
- время открытия главного экрана;
- время открытия карточки точки;
- количество команд в outbox;
- время отправки завершенного отчета после появления сети;
- количество retry;
- количество conflict;
- средний размер фото;
- время загрузки фото;
- ошибки по типам;
- версия приложения;
- модель устройства и версия Android.

Метрики нужны для поддержки и оптимизации, но не должны превращаться в пользовательский экран.

```json
  "pointId": "uuid",
  "scannedNfcTagHash": "hash-value",
  "expectedNfcTagHash": "hash-value",
  "matched": true,
  "scannedAt": "2026-05-25T12:10:00Z",
  "deviceId": "uuid",
  "baseRevision": 12
}
```

Правила:

- если метка совпала, точка получает локальный статус `scanned`;
- после сканирования UI открывает выбор `Исправно` / `Не исправно`;
- если метка не совпала, точка не считается подтвержденной;
- сервер повторно проверяет соответствие NFC.

## 22. Финальные релизные решения перед стартом разработки

Этот раздел закрывает технические пробелы предыдущих редакций и фиксирует решения для первого Android-релиза. Новые продуктовые функции не добавляются: телефон берет доступные заявки, проходит обход, фиксирует метки, фото и отчет, а затем незаметно синхронизирует результат с сервером.

### 22.1. Mobile DTO и SQLite-таблицы

Все локальные данные должны храниться с привязкой к `ownerUserId` и `deviceId`. Смена аккаунта не удаляет незагруженные данные другого пользователя, но полностью скрывает их из UI нового пользователя.

| Сущность | Server DTO | SQLite-таблица | Ключевые поля | Кто меняет |
|---|---|---|---|---|
| Пользователь | `MobileUserDto` | `users` | `serverUserId`, `fullName`, `roles`, `permissions`, `updatedAtServer` | сервер |
| Устройство | `MobileDeviceDto` | `devices` | `deviceId`, `ownerUserId`, `pushToken`, `trusted`, `blockedAt` | сервер + mobile registration |
| Доска заявок | `PatrolRequestBoardItemDto` | `patrol_request_board` | `requestId`, `routeId`, `routeName`, `plannedStartAt`, `assignedFullName`, `status`, `revision` | сервер |
| Заявка на обход | `PatrolRequestDto` | `patrol_requests` | `requestId`, `routeId`, `status`, `revision`, `takenByUserId`, `takenAt` | сервер, mobile только через команды |
| Назначение обхода | `PatrolAssignmentDto` | `patrol_assignments` | `assignmentId`, `requestId`, `routeId`, `status`, `startedAtLocal`, `completedAtLocal`, `revision` | сервер + mobile commands |
| Маршрут | `PatrolRouteDto` | `routes` | `routeId`, `name`, `version`, `allowFreeOrder`, `nfcEnabled`, `qrFallbackEnabled` | сервер |
| Точка маршрута | `PatrolPointDto` | `route_points` | `pointId`, `routeId`, `name`, `orderIndex`, `nfcUidHash`, `qrCodeHash`, `required`, `revision` | сервер |
| Результат точки | `PatrolPointResultDto` | `point_results` | `localResultId`, `assignmentId`, `pointId`, `status`, `comment`, `deferredReason`, `completedAtLocal` | mobile, сервер подтверждает |

Поля маршрутов, точек, прав, правил и назначений считаются серверными; телефон меняет их только через outbox-команды, а не прямым редактированием таблиц.

### 22.2. Машины состояний

Заявка на обход:

```text
available -> taking -> accepted -> inProgress -> completedLocal -> syncing -> completedServer
available -> assigned -> accepted
available|assigned|accepted|inProgress -> cancelled
accepted|inProgress|completedLocal|syncing -> conflict
available|assigned|accepted|inProgress -> overdue
```

Правила:

- `taking` локальный краткий статус на время атомарного взятия заявки;
- `completedLocal` означает, что сотрудник завершил обход без гарантии приема сервером;
- `completedServer` ставится только после подтверждения `completePatrolAssignment`;
- из `cancelled`, `completedServer` и `conflict` назад в работу переводить нельзя без отдельного решения оператора.

Точка маршрута:

```text
pending -> scanned -> ok -> queued -> synced
pending -> scanned -> issue -> queued -> synced
pending -> scanned -> deferred -> scanned
pending -> skipped -> queued -> synced
queued -> conflict
```

Правила:

- `deferred` разрешен только после подтверждения NFC/QR/ручной отметки по правилам маршрута;
- обязательная точка в `deferred` блокирует отправку отчета;
- `ok`, `issue` и `skipped` можно исправить до отправки финального отчета;
- после `synced` исправление идет отдельной командой `amendPatrolPointReport`, если это разрешено backend.

Фото:

```text
localOnly -> queued -> uploading -> uploaded -> linked -> retained
uploading -> failed -> queued
retained -> deletedAfterRetention
```

Правила:

- фото нельзя удалить локально, пока нет `uploaded` или `linked`;
- `failed` не является ошибкой пользователя, upload повторяется автоматически;
- отчет может ждать фото, если правила маршрута требуют обязательное фото.

Outbox-команда:

```text
pending -> sending -> accepted
pending -> sending -> duplicate
pending -> sending -> retryLater -> pending
pending -> sending -> rejected
pending -> sending -> conflict
```

Правила:

- `duplicate` считается успешным результатом и должен вернуть тот же итог, что первая обработка;
- `retryLater` используется только для временных ошибок сети, 5xx, timeout и занятости сервера;
- `rejected` используется для постоянной ошибки, которую пользователь или оператор должен исправить;
- `conflict` сохраняет локальные данные и показывает спорный отчет оператору в web-панели.

ЭМУ-задача в первом релизе:

```text
new -> accepted -> inProgress -> paused -> inProgress -> completedLocal -> completedServer
accepted|inProgress|paused|completedLocal -> conflict
new|accepted|inProgress|paused -> cancelled
```

### 22.3. Единый формат outbox-команды

Каждое действие телефона, которое меняет серверные данные, записывается в `outbox_commands` до попытки отправки.

```json
{
  "clientOperationId": "uuid",
  "commandType": "markPatrolPointIssue",
  "entityType": "patrolPoint",
  "entityLocalId": "uuid",
  "entityServerId": "uuid-or-null",
  "payload": {},
  "createdAtLocal": "2026-05-25T10:30:00+05:00",
  "attemptCount": 0,
  "status": "pending"
}
```

Обязательные правила:

- `clientOperationId` генерируется один раз и не меняется при повторной отправке;
- `payload` хранит снимок данных на момент действия пользователя;
- одна команда не должна менять несколько независимых сущностей, кроме явно описанных batch-команд;
- backend хранит `MobileCommandInbox` и возвращает тот же результат при повторе `clientOperationId`;
- телефон отправляет команды в порядке создания, но upload фото может идти отдельной очередью до финального завершения отчета.

Ответ backend на команду:

```json
{
  "clientOperationId": "uuid",
  "status": "accepted",
  "serverEntityId": "uuid",
  "serverRevision": 12,
  "message": "Принято",
  "conflictId": null,
  "retryAfterSeconds": null
}
```

Допустимые `status`: `accepted`, `duplicate`, `retryLater`, `rejected`, `conflict`.

### 22.4. Sync-контракты первого релиза

| Команда | Endpoint | Payload | Успех | Reject | Conflict |
|---|---|---|---|---|---|
| `takePatrolRequest` | `POST /api/v1/mobile/outbox` | `requestId`, `routeId`, `requestRevision`, `takenAtLocal` | заявка закреплена за пользователем | нет прав, заявка недоступна | заявку уже взял другой сотрудник |
| `scanPatrolPointNfc` | `POST /api/v1/mobile/outbox` | `assignmentId`, `pointId`, `nfcUidHash`, `scannedAtLocal` | метка подтверждена | NFC не относится к точке | маршрут/точка изменены |
| `markPatrolPointOk` | `POST /api/v1/mobile/outbox` | `assignmentId`, `pointId`, `comment`, `photoClientFileIds`, `completedAtLocal` | результат точки принят | не хватает обязательного фото/комментария | точка изменилась после bootstrap |
| `markPatrolPointIssue` | `POST /api/v1/mobile/outbox` | `assignmentId`, `pointId`, `issueTypeId`, `comment`, `photoClientFileIds`, `completedAtLocal` | неисправность принята | нет комментария или типа неисправности | маршрут/назначение изменены |
| `uploadPatrolPhoto` | `POST /api/v1/mobile/files` | `clientFileId`, `sha256`, `mimeType`, `sizeBytes`, `assignmentId`, `pointId` | файл сохранен и получил `serverFileId` | файл поврежден или превышает лимит | файл относится к спорному отчету |
| `completePatrolAssignment` | `POST /api/v1/mobile/outbox` | `assignmentId`, `requestId`, `summary`, `completedAtLocal`, `pointResultIds` | отчет завершен на сервере | есть обязательные незаполненные/отложенные метки | назначение или маршрут изменились |

Для первого релиза NFC остается P0 через feature flag `mobile.nfc.enabled`. QR остается P1 как резервный способ подтверждения точки, включаемый правилом маршрута. Если NFC выключен для маршрута, приложение не показывает пользователю техническую ошибку, а использует доступный разрешенный способ подтверждения.

### 22.5. Правила конфликтов и retry без дублей

Общее правило: локальные данные и фото не удаляются, пока сервер явно не принял отчет или оператор не принял решение по конфликту.

| Ситуация | Поведение mobile | Поведение backend/web |
|---|---|---|
| Заявку взял другой сотрудник | команда получает `conflict`, заявка закрывается для выполнения | оператор видит обе попытки и фактического владельца |
| Заявка отменена после offline-bootstrap | отчет не удаляется, отправка переводится в `conflict` | оператор видит локально выполненный обход по отмененной заявке |
| Маршрут изменен во время offline-обхода | телефон отправляет отчет по сохраненной revision | backend принимает или создает `conflict` с причиной изменения |
| Неверная NFC-метка | точка не засчитывается, пользователь остается на сканировании | backend возвращает `rejected` без создания результата точки |
| Фото загружено, финальная команда не принята | фото остается связанным с черновиком отчета | web показывает отчет как `ожидает решения` |
| Пользователь потерял права | новые действия блокируются после sync | уже созданные локальные действия уходят в `conflict`, не удаляются |

Retry:

- сеть пропала, timeout, HTTP 408/429/5xx: `retryLater`;
- HTTP 400 по бизнес-валидации: `rejected`;
- HTTP 401/403: обновить токен, затем повторить; если право отозвано, создать `conflict`;
- повторная отправка одного `clientOperationId`: `duplicate` с исходным результатом;
- повторная загрузка одного `clientFileId`: вернуть существующий `serverFileId`.

### 22.6. Pipeline фото

Фото делается камерой телефона и сохраняется локально до подтверждения сервером.

Релизные лимиты:

- формат: JPEG;
- максимальный размер после сжатия: 1.5 МБ;
- целевая ширина: до 1600 px по длинной стороне;
- качество сжатия: 70-80%;
- preview: до 320 px по длинной стороне;
- параллельная загрузка: не больше 2 фото;
- максимальное количество фото на одну точку: 3;
- максимальное количество фото в одном отчете: 150.

Порядок:

1. Камера создает оригинал во временной папке приложения.
2. Приложение создает сжатую версию и preview.
3. В SQLite создается запись `files` со статусом `localOnly`.
4. При сохранении точки фото получает связь с `point_results`.
5. Upload queue переводит файл в `queued` и отправляет `clientFileId`, `sha256`, metadata.
6. Сервер возвращает `serverFileId`.
7. После связи с точкой статус становится `linked`.
8. Локальный оригинал можно удалить только после `linked` и истечения retention.

Если сотрудник закрыл приложение во время съемки или отправки, при следующем запуске приложение должно восстановить файл по записи в SQLite или показать понятную ошибку восстановления конкретной точки.

### 22.7. Авторизация, устройства и смена аккаунта

Релизное решение:

- первый вход только онлайн;
- повторный вход без сети разрешен только для последнего успешно вошедшего пользователя на устройстве;
- access token короткий, refresh token хранится в Secure Storage;
- offline-сессия ограничена политикой backend и должна проверяться при следующем sync;
- устройство регистрируется в `MobileDevices` и получает `deviceId`;
- администратор может заблокировать потерянное устройство в web-панели;
- один телефон может использоваться несколькими сотрудниками только через явную смену аккаунта;
- данные каждого пользователя хранятся с `ownerUserId` и не показываются другому пользователю.

Смена аккаунта:

1. Приложение проверяет незагруженные данные текущего пользователя.
2. Если есть незавершенный отчет, показывается предупреждение и предлагается отправить позже или выйти без удаления.
3. Токены текущего пользователя удаляются из Secure Storage.
4. UI очищает память состояния и открывает экран входа.
5. После входа нового пользователя bootstrap загружает только его данные.
6. Локальные данные прошлого пользователя остаются недоступными до входа под тем же пользователем или административного решения.

### 22.8. Web-панель оператора и администратора

Для первого релиза web-панель должна поддерживать мобильный контур без отдельного пользовательского экрана синхронизации на телефоне.

Обязательные экраны web:

- доска заявок: создать/назначить/отменить заявку, видеть ФИО, маршрут, время начала и статус;
- маршрут и правила: свободный порядок точек, NFC feature flag, QR fallback, обязательность фото/комментария;
- отчет обхода: список точек, статусы, комментарии, фото, время локальное и серверное;
- неисправности: фильтр по маршруту, дате, точке, типу неисправности;
- конфликты sync: причина, payload, фото, действие оператора `принять`, `отклонить`, `запросить повтор`;
- устройства: пользователь, модель, версия приложения, последняя синхронизация, push token, статус доверия, блокировка;
- аудит: действия mobile с `clientOperationId`, `deviceId`, пользователем и временем.

Обязательные статусы mobile-отчета в web: `в работе`, `завершен на телефоне`, `ожидает фото`, `отправляется`, `принят сервером`, `конфликт`, `отклонен`.

### 22.9. Производительность и слабые Android-устройства

Первый релиз должен работать на слабом Android-устройстве с 2-3 ГБ RAM без тяжелых карт, графиков и лишних анимаций.

Целевое устройство для пилота: `Kenshi Armor C1s`. Приложение должно быть проверено на всех Android-версиях и прошивках `Kenshi Armor C1s`, которые фактически используются или будут закуплены предприятием. Если в парке окажутся разные версии Android на одной модели телефона, каждая версия считается отдельной тестовой конфигурацией.

Измеримые лимиты:

- активных заявок на телефоне: до 10;
- точек в одном маршруте: до 300;
- фото в одном отчете: до 150;
- размер локальной SQLite-базы без фото: до 100 МБ;
- размер локального файлового хранилища приложения: предупреждение после 1 ГБ;
- открытие экрана `Обход`: до 2 секунд после холодного запуска на слабом устройстве;
- открытие карточки точки: до 1 секунды;
- первичный bootstrap обычного маршрута до 300 точек: до 30 секунд при стабильной сети;
- начало отправки завершенного отчета после появления интернета: до 10 секунд;
- повтор sync при временной ошибке: 15 секунд, затем 30 секунд, затем 60 секунд, далее не чаще 5 минут.

Оптимизации:

- списки точек через виртуализированный список;
- фото показывать preview, оригинал не загружать в память без необходимости;
- фоновые sync-задачи не запускать параллельно в нескольких экземплярах;
- большие JSON payload разбивать по назначению, а не хранить один огромный объект состояния;
- не держать камеру активной после ухода с экрана.

### 22.10. Выпуск, окружения и версии протокола

Сборки:

- `dev` - для разработки и локального API;
- `test` - для тестового сервера предприятия;
- `local-enterprise` - для локальной сети предприятия;
- `production` - рабочая сборка.

Android-релиз выпускается как APK для внутренней установки и AAB, если выбран Google Play Managed/закрытый канал. Для Expo использовать EAS Build или локальную Android-сборку. Expo Go не подходит для NFC, камеры, Secure Storage, push и production-проверок.

Каждый запрос mobile передает:

- `appVersion`;
- `syncProtocolVersion`;
- `deviceId`;
- `platform`;
- `buildChannel`.

Если backend не поддерживает `syncProtocolVersion`, приложение блокирует sync и показывает рабочее сообщение: `Нужно обновить приложение. Обратитесь к ответственному.` Технические детали остаются в логах и web-панели администратора.

Эти параметры не блокируют старт разработки первого релиза. Для MVP зафиксированы рабочие значения по умолчанию, а пересмотр допускается только после пилота или при изменении парка устройств/маршрутов.

| Вопрос | Решение для первого релиза | Когда пересматривать |
|---|---|---|
| Какие Android-устройства утвердить для NFC-тестов | основное устройство пилота - Kenshi Armor C1s; проверить все Android-версии/прошивки этой модели, которые есть в парке предприятия | перед пилотной установкой APK |
| Требовать ли GPS при прохождении точки | GPS не обязателен, остается P2-улучшением | после пилота, если нужны геодоказательства |
| Может ли сотрудник сам брать новые ЭМУ-задачи | в MVP сотрудник выполняет назначенные задачи, самоприем ЭМУ не включается | после стабилизации обходов |
| Какие фото обязательны | по умолчанию фото необязательно и для `Исправно`, и для `Не исправно`; обязательность включается только настройкой маршрута/точки | при настройке маршрутов в web |
| Как долго хранить локальную историю | metadata принятых отчетов хранить до 30 дней, локальные фото после `linked` очищать по retention не раньше 7 дней | после оценки объема данных на телефонах |
| Нужен ли режим нескольких сотрудников на одном устройстве | разрешена только явная смена аккаунта; одновременная работа нескольких пользователей в одной сессии не поддерживается | если предприятие утвердит общий парк телефонов |

Финальные решения первого релиза:

- NFC включаем через feature flag `mobile.nfc.enabled` для подготовленных маршрутов и реальных NFC-меток.
- QR оставляем как P1-резервный способ подтверждения точки.
- Телефон не создает новые заявки на обход, а берет доступные заявки и выполняет назначенные.
- Отдельный технический экран синхронизации для сотрудника не делаем.
- Основной критерий готовности: отчет, созданный без интернета, не теряет фото и отправляется без дублей после восстановления связи.

## 24. Готовность к реализации MVP

ТЗ готово к началу разработки первого релиза. Доработки после этого раздела не должны менять базовый сценарий MVP без отдельного решения: заявка берется с доски, обход проходится по NFC/QR-меткам в свободном порядке, результат точки сохраняется локально, отчет отправляется после появления сети без дублей.

### 24.1. Что считаем зафиксированным

Зафиксированные решения:

- платформа разработки: React Native + Expo + TypeScript, Android first;
- целевое устройство пилота: `Kenshi Armor C1s`;
- проверка совместимости: все Android-версии/прошивки `Kenshi Armor C1s`, которые есть в парке предприятия;
- первый импорт маршрутов из `atom_obhod`: `Обход печей`, затем `Помол`;
- NFC входит в MVP через feature flag `mobile.nfc.enabled`;
- QR остается резервным способом подтверждения точки;
- порядок прохождения точек свободный, если маршрут не включил строгий порядок;
- фото для `Исправно` и `Не исправно` по умолчанию необязательно;
- обязательность фото включается только настройкой маршрута/точки;
- отдельного пользовательского экрана синхронизации нет;
- телефон не создает заявки, а берет доступные и выполняет назначенные;
- стабильность sync, сохранность фото и отсутствие дублей важнее дополнительных экранов.

### 24.2. Минимальный порядок старта разработки

1. Backend: добавить mobile auth, регистрацию устройств, bootstrap, доску заявок и `MobileCommandInbox`.
2. Mobile: создать проект в папке `Мобильное приложение`, настроить Expo dev build, SQLite, Secure Storage, окружения и базовую авторизацию.
3. Backend + mobile: реализовать outbox-контракт с `clientOperationId` и статусами `accepted`, `duplicate`, `retryLater`, `rejected`, `conflict`.
4. Mobile: реализовать экран `Обход`, выбор заявки, активный обход, `Все метки`, NFC/QR-подтверждение и заполнение точки.
5. Mobile + backend: реализовать фото pipeline с `clientFileId`, локальным хранением, upload queue и подтверждением сервера.
6. Backend + web: реализовать прием отчета, web-просмотр результата, фото, конфликтов и устройств.
7. QA: пройти P0-сценарии из раздела 22.11 на `Kenshi Armor C1s`.
8. Release: собрать APK/AAB, проверить окружения и зафиксировать known limitations пилота.

### 24.3. Что нельзя менять без пересогласования ТЗ

- заменять offline-first на online-only;
- отправлять отчет без сохраненного локального outbox;
- удалять локальные фото до подтверждения сервера;
- показывать новому пользователю данные прошлого аккаунта;
- делать фото обязательным для всех `Не исправно` без правила маршрута/точки;
- требовать строгий порядок точек для всех маршрутов;
- добавлять отдельный технический экран синхронизации для сотрудника;
- разрешать телефону создавать новые заявки на обход;
- начинать пилот без проверки `Kenshi Armor C1s` и маршрутов `Обход печей` / `Помол`.

### 24.4. Definition of Ready для задач разработки

Перед взятием задачи в разработку у нее должны быть:

- ссылка на раздел ТЗ или backlog ID из раздела 17;

### 24.5. Критерии готовности MVP к пилоту

MVP можно отдавать на пилот, если выполнены условия:

- сотрудник входит в приложение и получает данные после bootstrap;
- сотрудник берет заявку `Обход печей` или `Помол` с доски;
- активный обход открывается без сети после первой синхронизации;
- NFC или QR подтверждает правильную точку и не засчитывает чужую;
- точку можно отметить `Исправно` или `Не исправно` без обязательного фото по умолчанию;
- отложенная обязательная метка блокирует отправку отчета;
- сделанные фото не теряются после закрытия приложения;
- завершенный оффлайн-отчет отправляется после появления интернета;
- повторная отправка не создает дубли отчета, точек и фото;
- смена аккаунта не показывает данные прошлого пользователя;
- оператор видит отчет, точки, комментарии, фото и конфликты в web-панели;
- приложение проходит P0 smoke на всех утвержденных Android-версиях `Kenshi Armor C1s`.

### 25.2. Release gates

Gate 1: технический фундамент.

- backend auth, devices, bootstrap и command inbox работают;
- mobile dev build запускается на Android и имеет SQLite/Secure Storage;
- есть test/local/prod профили API;
- приложение не использует Expo Go для NFC/camera проверок.

Gate 2: рабочий обход без фото.

- сотрудник входит, берет заявку и видит активный обход;
- точки можно проходить в свободном порядке;
- NFC или QR подтверждает точку;
- `Исправно` и `Не исправно` сохраняются локально;
- отложенная обязательная метка блокирует отчет.

Gate 3: фото и offline-sync.

- фото сохраняется после перезапуска приложения;
- upload не создает дубли по `clientFileId`;
- отчет завершенный без сети уходит после появления интернета;
- повторная отправка не создает дубли по `clientOperationId`.

Gate 4: web-панель и конфликты.

- оператор видит заявку, отчет, точки, комментарии и фото;
- конфликт маршрута или заявки виден в web-панели;
- устройство можно заблокировать;
- аудит показывает пользователя, устройство и `clientOperationId`.

Gate 5: пилот.

- APK установлен на `Kenshi Armor C1s`;
- проверены все Android-версии/прошивки из парка предприятия;
- маршруты `Обход печей` и `Помол` импортированы и сверены;
- P0 QA из раздела 22.11 пройден без критичных дефектов;
- known limitations записаны перед выдачей сотрудникам.

### 25.3. Матрица критичных отказов

| Отказ | Что должно произойти | Где проверяется |
|---|---|---|
| Нет интернета во время обхода | все действия сохраняются локально, пользователь продолжает обход | mobile SQLite + outbox |
| Нет интернета во время отправки | команда остается в очереди и повторяется после сети | outbox processor |
| Приложение закрыли во время фото | фото или черновик точки восстанавливается после запуска | file metadata + SQLite |
| Повторили отправку отчета | backend возвращает `duplicate`, дубль не создается | `MobileCommandInbox` |
| Приложили чужую NFC-метку | точка не засчитывается | mobile validation + backend validation |
| Сменили аккаунт | новый пользователь не видит локальные данные прошлого | `ownerUserId` isolation |
| Маршрут изменился во время offline-обхода | отчет не удаляется, создается conflict | `MobileSyncConflicts` |
| Не хватает места на телефоне | приложение предупреждает и не удаляет уже сделанные фото | storage guard |

### 25.4. Definition of Done для MVP

MVP считается выполненным, когда:

- все P0 задачи из backlog раздела 17 закрыты;
- все release gates из раздела 25.2 пройдены;
- P0 QA из раздела 22.11 пройден на `Kenshi Armor C1s`;
- backend не создает дубли отчетов, точек и фото при повторной отправке;
- web-панель показывает отчеты и конфликты без ручного обращения к базе;
- импорт `Обход печей` и `Помол` подтвержден сверкой;
- для сотрудников нет отдельного технического экрана синхронизации;
- ТЗ, known limitations и инструкция установки APK лежат рядом с релизом.

## 26. Финальная передача ТЗ в реализацию

Документ v1.0 является базой для разработки MVP. Реализация должна идти по backlog раздела 17, техническим контрактам раздела 21, финальным решениям раздела 22 и release gates раздела 25.

### 26.1. Состав финального MVP

В первый релиз входят:

- Android-приложение на React Native + Expo + TypeScript;
- авторизация, регистрация устройства и безопасная смена аккаунта;
- offline-first локальная база SQLite;
- bootstrap маршрутов, заявок, правил и назначений;
- доска заявок на обход;
- активный обход с прохождением точек в свободном порядке;
- NFC через feature flag и QR как резерв;
- статусы точки `Исправно` и `Не исправно`;
- комментарий, тип неисправности и критичность;
- необязательное фото по умолчанию для обоих статусов;
- обязательное фото только по правилу маршрута/точки;
- отложенные метки;
- отправка отчета через outbox без дублей;
- загрузка фото через `clientFileId` без дублей;
- web-просмотр отчетов, фото, устройств и конфликтов;
- пилот на `Kenshi Armor C1s`;
- первичный импорт маршрутов `Обход печей` и `Помол`.

### 26.2. Не входит в первый релиз

В первый релиз не входят:

- iOS;
- карта маршрута;
- GPS как обязательное подтверждение точки;
- видеофиксация;
- создание новых заявок с телефона;
- редактирование маршрутов с телефона;
- отдельный пользовательский экран технической синхронизации;
- аналитика и расширенные отчеты;
- самоприем новых ЭМУ-задач сотрудником;
- обязательное фото для всех неисправностей без настройки маршрута/точки.

### 26.3. Правила изменения ТЗ после v1.0

После начала реализации изменения делятся на три типа:

- `P0 change` - влияет на безопасность, потерю данных, sync, отчет, NFC/QR, смену аккаунта или приемку MVP; требует пересмотра backlog и QA.
- `P1 change` - уточняет UI, текст, web-фильтр, инструкцию или настройку без изменения контрактов; можно добавлять в текущий релиз при наличии времени.
- `P2 change` - новая функция или улучшение после пилота; переносится за пределы MVP.

Нельзя вносить без пересогласования:

- изменение offline-first подхода;
- изменение outbox/idempotency правил;
- обязательное фото для всех неисправностей;
- отказ от `Kenshi Armor C1s` как пилотного устройства;
- замену маршрутов пилота `Обход печей` и `Помол`;
- добавление новых пользовательских экранов, не описанных в разделе 10.

### 26.4. Комплект артефактов для старта работ

Перед стартом разработки должны быть подготовлены:

- задача на создание проекта в папке `Мобильное приложение`;
- список Android-версий/прошивок `Kenshi Armor C1s` в парке предприятия;
- доступ к старому проекту `C:\Users\AI_server\Desktop\atom_obhod` для импорта маршрутов;

- тестовый backend environment;
- тестовые учетные записи сотрудника, оператора и администратора;
- тестовая заявка на обход;
- тестовая задача ЭМУ;
- место хранения APK и инструкций установки;
- ответственный за приемку P0 QA.

### 26.5. Финальный критерий старта разработки

Разработку можно начинать, если команда согласна со следующими условиями:

- MVP реализуется строго по P0/P1 backlog из раздела 17;
- любые P2-идеи не блокируют первый релиз;
- mobile не начинает работу без SQLite/outbox;
- backend не принимает мобильные команды без `clientOperationId`;
- file upload не принимает фото без `clientFileId`;
- QA проверяет реальные устройства, а не только эмулятор;
- пилот считается успешным только после прохождения маршрутов `Обход печей` и `Помол` на `Kenshi Armor C1s`.
