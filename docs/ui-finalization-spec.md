# ТЗ финальной доработки UI интерфейса

> **Статус документа: требования к UI-проходу, а не текущий implementation status.**
> Упоминания локальных сценариев без backend относятся к этапу проектирования.
> Текущая frontend-архитектура описана в [frontend-architecture.md](./frontend-architecture.md), готовность модулей — в [modules.md](./modules.md).

## 1. Назначение документа

Документ фиксирует финальные требования к frontend-части веб-панели обходов территории до перехода к полноценной backend-интеграции.

Цель этапа: довести текущий React/Vite UI-прототип до устойчивого состояния, где основные пользовательские сценарии работают локально, экраны разложены по понятным frontend-boundary, кнопки выполняют ожидаемые действия, а интерфейс готов к подключению ASP.NET Core API без переписывания экранов.

## 2. Текущее состояние

Фронтенд уже содержит:

- общий shell приложения: sidebar, topbar, workspace header, screen router;
- вкладки: дашборд, результаты обходов, назначения, сотрудники, плановый обход, мобильные аккаунты, маршруты и точки, пользователи сайта;
- локальный CRUD маршрутов и точек;
- локальное создание заявки на обход;
- создание активного обхода из заявки;
- локальное хранение временных данных через `localStorage`;
- typed mock/API data source layer в `apps/web/src/api`;
- доменные frontend helpers в `apps/web/src/domain`;
- архитектурную документацию в `docs/frontend-architecture.md`.

Оценка готовности на момент фиксации:

- UI-прототип для демонстрации: около 80%;
- frontend для будущего backend-подключения: около 60-65%;
- общий frontend-прогресс: около 70%.

## 3. Финальная цель UI-этапа

После завершения этого этапа интерфейс должен:

- открываться без ошибок на всех основных вкладках;
- иметь единый стиль модалок, боковых панелей, таблиц, фильтров, кнопок и empty-state блоков;
- поддерживать рабочие локальные сценарии без backend;
- иметь минимальное количество кнопок, которые только показывают toast;
- иметь разложенную компонентную архитектуру для крупных экранов;
- сохранять временные данные в `localStorage` через typed/domain слой, а не напрямую в экранах;
- быть готовым к замене local/mock источников на API-клиенты;
- иметь базовый набор тестов для доменной логики и ключевых UI-сценариев.

## 4. Основные ограничения

- Backend пока не подключается.
- Вся логика работает локально через mock/localStorage.
- Нельзя ломать будущий API-переход: экраны не должны напрямую зависеть от статических массивов данных.
- Worklog остается вне текущего UI-этапа, если отдельно не будет принято решение вернуть его в scope.
- Интерфейс остается русскоязычным.
- Дизайн остается dense admin/operations style, без маркетинговых hero-блоков.

## 5. Зоны, которые снижают текущий процент готовности

### 5.1. Крупные экраны еще не все разложены на компоненты

Проблема:

- часть экранов содержит слишком много JSX, локального состояния, таблиц, drawer-разметки и обработчиков в одном файле;
- сложнее подключать API, тестировать и переиспользовать элементы;
- риск регрессий выше при каждом изменении UI.

Требование:

- каждый крупный screen должен стать композиционным слоем;
- таблицы, панели, drawer, формы и toolbar должны быть вынесены в `components/<feature>`;
- бизнес-операции должны жить в `domain/*` или hooks, а не внутри JSX.

Кандидаты на декомпозицию:

| Экран | Текущий риск | Требуемые компоненты |
|---|---|---|
| `DashboardScreen` | много панелей и зависимостей | `DashboardMetrics`, `DashboardReadinessPanel`, `DashboardRequestsPanel`, `DashboardIncidentsPanel`, `DashboardRouteSummaryPanel` |
| `RequestModals` | большая модалка создания/просмотра заявки | `CreatePatrolRequestModal`, `ViewPatrolRequestModal`, `RequestFormFields`, `RequestTimeline` |
| `AssignmentScreen` | выбор сотрудника, маршрута, активные назначения в одном файле | `EmployeePickerPanel`, `RoutePickerPanel`, `ActiveAssignmentsPanel`, `AssignmentDrawer` |
| `ResultsScreen` | таблица, фильтры, drawer и аналитика вместе | `ResultsFilterBar`, `ResultsTable`, `ResultDetailDrawer`, `ResultsAnalyticsPanel` |
| `ScheduleScreen` | визуальная сетка и карточки планирования вместе | `ScheduleToolbar`, `ScheduleGrid`, `ScheduleSidePanel`, `ScheduleEditModal` |
| `EmployeesScreen` | реестр, профиль и mobile access в одном потоке | `EmployeeDirectoryTable`, `EmployeeProfilePanel`, `EmployeeFormDrawer` |
| `SiteUsersScreen` | форма, таблица, профиль пользователя и RBAC карточки вместе | `SiteUserForm`, `SiteUsersTable`, `SiteUserProfile`, `RolesOverview` |

Критерии приемки:

- ни один screen-файл не должен быть главным местом хранения сложной таблицы и формы одновременно;
- screen-файл должен читать как композиция блоков;
- новые компоненты должны иметь явные typed props;
- общая логика должна быть вынесена из JSX в helper/hook.

### 5.2. Часть кнопок пока показывает toast вместо действия

Проблема:

- часть кнопок имитирует действие через `onNotify`;
- пользователь не получает реального состояния: открытую форму, выбранную запись, созданную заявку, изменение статуса или drawer;
- при демонстрации интерфейс воспринимается как частично статичный.

Требование:

Каждая кнопка должна попасть в одну из категорий:

1. Рабочая UI-кнопка: меняет локальное состояние, открывает drawer/modal, создает/редактирует/удаляет локальную запись.
2. Навигационная кнопка: переводит на нужную вкладку или выбирает связанную запись.
3. Disabled/locked кнопка: явно отключена и имеет понятное объяснение, почему действие недоступно.
4. Temporary API placeholder: допускается только если действие невозможно без backend, но кнопка должна быть помечена как временная в коде и документации.

Приоритетные кнопки для замены toast на действие:

| Экран | Кнопка | Ожидаемое действие |
|---|---|---|
| Дашборд | `Создать заявку` | открыть модалку создания заявки |
| Дашборд | строка заявки | открыть просмотр заявки |
| Дашборд | строка активного обхода | открыть detail drawer активного обхода |
| Результаты | `Создать заявку` | создать заявку из выбранного результата |
| Результаты | `Открыть заявку` | открыть существующую заявку или форму создания |
| Назначения | `Назначить сейчас` | создать заявку/активный обход локально |
| Сотрудники | `Создать сотрудника` | открыть форму/drawer создания сотрудника |
| Сотрудники | `Редактировать` | открыть форму редактирования выбранного сотрудника |
| Мобильные аккаунты | `Создать аккаунт` | создать локальный аккаунт |
| Мобильные аккаунты | `Привязать` | привязать сотрудника к выбранному аккаунту |
| Пользователи сайта | `Сохранить` | создать/обновить локального пользователя |
| Плановый обход | `Сохранить план` | сохранить локальный draft расписания |

Критерии приемки:

- toast используется только как подтверждение успешного действия, а не как замена действия;
- основные CRUD-кнопки меняют состояние интерфейса;
- если кнопка не реализована, она disabled или помечена как будущая API-функция.

### 5.3. Результаты обходов не полностью связаны с заявками и активными обходами

Проблема:

- экран результатов визуально есть, но связь с активными обходами и заявками не завершена;
- результат должен быть не отдельной статической записью, а фактом прохождения точки в рамках обхода;
- замечание по результату должно уметь превращаться в заявку.

Требование:

Модель результата должна быть связана с:

- заявкой на обход;
- активным обходом;
- маршрутом;
- точкой маршрута;
- сотрудником;
- фактическим временем прохождения;
- статусом точки;
- комментариями;
- фото/видео;
- созданными заявками по замечанию.

Минимальная локальная логика:

- при создании активного обхода создается структура точек;
- при имитации прохождения точки появляется результат;
- результат может иметь статус: `Норма`, `С замечанием`, `Просрочено`, `Не подтверждено`;
- из результата можно создать заявку;
- если заявка уже создана, кнопка `Открыть заявку` открывает ее, а не создает дубль.

Критерии приемки:

- `ResultsScreen` показывает результаты, привязанные к текущим локальным обходам;
- detail drawer результата показывает источник: заявка, обход, маршрут, точка;
- создание заявки из результата работает без дублей;
- дашборд использует эти результаты для блоков последних результатов и проблемных точек.

### 5.4. Плановый обход пока больше визуальный

Проблема:

- экран планирования выглядит как рабочая сетка, но логика правил и сохранения ограничена;
- карточки надо визуально стабилизировать;
- не хватает локального состояния правил, исключений и конфликтов.

Требование:

Плановый обход должен поддерживать минимум:

- weekly/monthly режимы;
- создание планового обхода на дату/дни недели;
- выбор маршрута;
- выбор сотрудника;
- дневная/ночная смена;
- исключения: отпуск, больничный, перенос, замена;
- статус ячейки: запланировано, не запланировано, конфликт, перенос, исключение;
- локальное сохранение плана в `localStorage`.

Визуальные требования к карточкам:

- единая высота карточек;
- не должно быть прыжков сетки при разном тексте;
- цвет статуса должен быть понятен: план, перенос, конфликт, исключение;
- дневная и ночная смена должны отличаться визуально;
- длинные названия маршрутов должны переноситься без выхода за контейнер.

Критерии приемки:

- можно создать локальную плановую запись;
- можно отредактировать запись;
- можно добавить исключение;
- конфликт показывается отдельным статусом;
- карточки выглядят стабильно на desktop viewport 1440px и 1680px.

### 5.5. Нет автотестов компонентов и сценариев

Проблема:

- сейчас качество проверяется вручную через typecheck/build/browser smoke;
- при росте UI легко сломать сценарий создания заявки или CRUD маршрутов;
- нет регрессии по localStorage и доменным helper-ам.

Требование:

Добавить тестовый контур frontend:

- Vitest для domain/helpers;
- React Testing Library для ключевых UI-компонентов;
- Playwright smoke/e2e для основных пользовательских сценариев.

Минимальный набор unit tests:

| Модуль | Что проверить |
|---|---|
| `domain/routes.ts` | создание маршрута, создание точки, порядок точек, duplicate NFC allowed |
| `domain/serviceRequests.ts` | дата по умолчанию, уведомление, создание draft заявки |
| `domain/activeAssignments.ts` | создание активного обхода из заявки и маршрута |
| `domain/activePatrolDetails.ts` | сортировка точек по фактическому времени прохождения |
| `domain/mobileAccounts.ts` | создание аккаунта, привязка сотрудника, reset password |

Минимальный набор e2e:

1. Создать маршрут.
2. Добавить точку.
3. Создать заявку на обход.
4. Проверить появление активного обхода на дашборде.
5. Открыть detail drawer активного обхода.
6. Создать заявку из результата.
7. Создать мобильный аккаунт и привязать сотрудника.

Критерии приемки:

- `npm run test` проходит;
- `npm run typecheck` проходит;
- `npm run verify` проходит;
- e2e smoke не требует backend.

### 5.6. `styles.css` слишком большой

Проблема:

- один CSS-файл содержит всю стилизацию;
- сложно искать стили конкретного экрана;
- растет риск случайных конфликтов;
- тяжело привести UI к единой палитре и scale.

Требование:

Разделить стили на feature-файлы без изменения визуального результата:

```text
apps/web/src/styles/
  base.css
  tokens.css
  layout.css
  components.css
  dashboard.css
  routes.css
  requests.css
  results.css
  assignments.css
  employees.css
  accounts.css
  schedule.css
  users.css
```

Рекомендуемый порядок:

1. Вынести design tokens: цвета, spacing, radius, shadows, font scale.
2. Вынести shell/layout: app-shell, sidebar, topbar, workspace.
3. Вынести reusable components: button, panel, chip, table, modal, drawer, empty-state.
4. Вынести feature-specific стили.
5. Оставить `styles.css` как entrypoint с `@import`.

Критерии приемки:

- визуальный результат не меняется;
- нет дублирующих CSS-селекторов с разным поведением;
- все цвета берутся из tokens;
- новые компоненты не добавляют стили в общий хаотичный файл.

### 5.7. Нет реального backend API, часть логики временная

Проблема:

- данные создаются локально;
- API слой есть, но backend еще не реализован;
- некоторые DTO и статусы временные;
- невозможно проверить реальные ошибки сети, права доступа, пагинацию, concurrency и серверную валидацию.

Требование frontend-этапа:

- не подключать backend в рамках UI-финализации;
- но привести все данные к contract-ready виду;
- подготовить client/repository слой для будущих endpoint-ов;
- разделить `mock`, `localStorage`, `api` источники данных.

Целевой frontend data boundary:

```text
api/
  contracts.ts
  patrolData.ts
  routesClient.ts
  requestsClient.ts
  assignmentsClient.ts
  resultsClient.ts
  employeesClient.ts
  mobileAccountsClient.ts

repositories/
  routesRepository.ts
  requestsRepository.ts
  assignmentsRepository.ts
  resultsRepository.ts
  mobileAccountsRepository.ts
```

Критерии приемки:

- UI экраны не знают, откуда пришли данные;
- mock/local/API переключаются через один boundary;
- localStorage используется только внутри hooks/repositories;
- после появления ASP.NET Core API меняется client/repository слой, а не JSX экранов.

## 6. Детальное ТЗ по экранам

### 6.1. Дашборд

Цель:

Оперативный экран текущей смены. Должен показывать реальное состояние маршрутов, заявок, активных обходов и результатов.

Требования:

- метрики строятся из локального состояния или API snapshot;
- блок активных обходов показывает только: сотрудник, маршрут, прогресс;
- клик по строке открывает detail drawer;
- detail drawer показывает точки в фактическом порядке прохождения;
- блок заявок показывает заявки, отправленные сотрудникам;
- блок проблемных точек связан с результатами;
- блок последних результатов связан с результатами активных обходов;
- empty-state должен быть понятным, без фальшивых данных.

Критерии приемки:

- после создания заявки активный обход появляется на дашборде;
- после выбора обхода drawer показывает маршрутные точки;
- кнопка `Создать заявку` открывает модалку;
- клик по заявке открывает просмотр заявки.

### 6.2. Маршруты и точки

Цель:

Справочник маршрутов и контрольных точек.

Требования:

- создать маршрут;
- редактировать маршрут;
- удалить маршрут;
- создать точку;
- редактировать точку;
- удалить точку;
- менять порядок точек;
- сохранять в `localStorage`;
- разрешить одинаковую NFC-метку в разных маршрутах;
- показывать предупреждение, что повтор метки допустим.

Критерии приемки:

- все CRUD-действия работают локально;
- после перезагрузки страницы данные сохраняются;
- одинаковый NFC/tag не блокируется.

### 6.3. Заявки и назначения обхода

Цель:

Оператор создает заявку на проведение обхода.

Требования:

- дата по умолчанию сегодня;
- сотрудник обязателен;
- маршрут обязателен;
- время прохождения необязательно;
- уведомление сотруднику включается checkbox/toggle;
- текст уведомления можно редактировать;
- после создания заявка получает статус;
- из заявки создается активный обход;
- созданная заявка отображается на дашборде.

Критерии приемки:

- нельзя создать заявку без сотрудника и маршрута;
- можно создать без времени;
- активный обход появляется после создания;
- уведомление отображается как часть заявки.

### 6.4. Активный обход

Цель:

Показать процесс прохождения маршрута сотрудником.

Требования:

- таблица активных обходов: сотрудник, маршрут, progress bar;
- detail drawer:
  - сотрудник;
  - маршрут;
  - время старта;
  - общее время;
  - список точек;
  - фактическое время активации метки;
  - статус точки;
  - комментарий;
  - фото/видео placeholder;
  - кнопка создания заявки;
- сортировка точек по фактическому времени прохождения, а не по порядку маршрута.

Критерии приемки:

- клик по обходу меняет drawer;
- точки отображаются в корректном порядке;
- отсутствие медиа показывается empty-state, а не пустым блоком.

### 6.5. Результаты обходов

Цель:

Журнал фактического прохождения точек.

Требования:

- фильтры: период, маршрут, сотрудник, статус, тип замечания;
- таблица результатов;
- detail drawer;
- создание заявки из результата;
- открытие связанной заявки;
- связь с активным обходом;
- связь с маршрутом и точкой;
- аналитические блоки снизу либо реальные, либо empty-state.

Критерии приемки:

- выбранный результат отображается в drawer;
- кнопка `Создать заявку` создает связанную заявку;
- повторное открытие не создает дубль.

### 6.6. Сотрудники

Цель:

Реестр сотрудников, отдельно от мобильных аккаунтов.

Требования:

- список сотрудников;
- профиль сотрудника;
- создать сотрудника;
- редактировать сотрудника;
- показать статус мобильного доступа;
- переход к созданию/привязке мобильного аккаунта;
- не смешивать сотрудника и аккаунт телефона как одну сущность.

Критерии приемки:

- сотрудник может существовать без мобильного аккаунта;
- мобильный аккаунт может быть привязан к одному/нескольким/всем сотрудникам;
- UI явно показывает разницу между сотрудником и mobile login.

Критерии приемки:

- `selected/all` scope отображается в таблице;
- вкладки `Аккаунты`, `Сессии`, `Привязки` работают.

### 6.8. Плановый обход

Цель:

Управление пользователями веб-панели.

## 9. Эксплуатационное качество UI

Этот раздел фиксирует требования, которые не являются отдельными бизнес-сценариями, но напрямую влияют на качество работы операторов, сопровождаемость фронтенда и готовность к production-подключению backend.

### 9.1. Accessibility и клавиатурная доступность

Цель:

Интерфейс должен быть управляемым не только мышью, но и клавиатурой. Это важно для операторской панели, где часть действий выполняется быстро и повторяется много раз за смену.

Требования:

- все интерактивные элементы должны быть доступны через `Tab`;
- порядок фокуса должен соответствовать визуальному порядку экрана;
- у icon-only кнопок должны быть `aria-label` или понятный `title`;
- кнопки закрытия modal/drawer должны иметь `aria-label="Закрыть"`;
- формы должны иметь связанные label/input;
- ошибки валидации должны быть связаны с полем через `aria-describedby`;
- активная вкладка в `SectionTabs` должна быть понятна визуально и семантически;
- focus state должен быть видимым на светлом фоне;
- disabled элементы должны выглядеть отключенными, но оставаться читаемыми;
- модалки должны закрываться по `Escape`;
- при открытии модалки/дровера фокус должен попадать внутрь;
- после закрытия модалки фокус должен возвращаться на кнопку, которая ее открыла.

Минимальные элементы для проверки:

| UI элемент | Требование |
|---|---|
| Sidebar nav | переход по `Tab`, активация по `Enter` |
| Topbar search | фокус, ввод, submit по `Enter` |
| Таблица маршрутов | выбор строки мышью и клавиатурой |
| Таблица результатов | выбор строки и открытие detail drawer |
| Modal заявки | фокус внутри, `Escape`, submit, inline errors |
| Drawer активного обхода | кнопки доступны с клавиатуры |
| Icon buttons | имеют `aria-label`/`title` |

Критерии приемки:

- на основных вкладках можно пройти интерактивные элементы через `Tab`;
- нет icon-only кнопок без доступного имени;
- контраст текста и controls достаточен на светлом фоне;
- modal/drawer не оставляет фокус в фоне.

### 9.2. Responsive desktop breakpoints

Цель:

Панель проектируется как desktop/admin UI, но должна стабильно работать на разных ширинах рабочих мониторов.

Целевые viewport:

| Viewport | Назначение |
|---|---|
| 1366px | минимальный поддерживаемый desktop |
| 1440px | базовый дизайн-размер |
| 1680px | основной широкий рабочий монитор |
| 1920px | full HD рабочее место |

Требования:

- sidebar не должен перекрывать контент;
- collapsed sidebar должен освобождать место без сдвига критичных controls за экран;
- topbar должен сохранять читаемость даты, смены, территории, поиска и пользователя;
- таблицы должны иметь горизонтальный scroll только внутри `table-wrap`, а не на всей странице;
- drawer должен иметь стабильную ширину и не перекрывать основной action bar;
- модалки должны помещаться по высоте или иметь внутренний scroll;
- карточки планового обхода не должны ломать сетку при длинных названиях маршрутов;
- длинные ФИО, маршруты и названия точек должны переноситься или обрезаться с понятным tooltip/title;
- кнопки в toolbar не должны переноситься хаотично.

Desktop layout rules:

- основной контент не должен быть уже 960px при открытом drawer;
- side drawer: 360-420px для обычных деталей, 440-520px для сложных форм;
- таблицы: sticky header желательно, но не обязательно на UI-этапе;
- max-width для форм в drawer должен предотвращать растяжение input на слишком широких экранах;
- grid gap должен быть единым: 12/16/20px по токенам.

Критерии приемки:

- browser smoke выполняется на 1440px и 1680px;
- нет горизонтального scroll у всего body;
- нет обрезанного русского текста в кнопках первичных действий;
- drawer и таблица не накладываются друг на друга.

### 9.3. Loading, skeleton и pending states

Цель:

Подготовить UI к будущему backend API. Сейчас есть empty-state, но при реальной загрузке нужны отдельные loading-состояния.

Типы состояний:

| Состояние | Когда используется | UI |
|---|---|---|
| `idle` | экран готов, данных нет или не запрашивались | empty-state |
| `loading` | данные загружаются | skeleton/table rows shimmer |
| `refreshing` | данные уже есть, идет фоновое обновление | small spinner/status в toolbar |
| `submitting` | форма отправляется | disabled submit + spinner/label |
| `success` | действие выполнено | toast + обновленное состояние |
| `error` | запрос не выполнен | error banner + retry |

Требования:

- таблицы должны иметь skeleton rows;
- карточки метрик должны иметь skeleton value;
- detail drawer должен уметь показывать loading state;
- submit-кнопки должны блокироваться на время отправки;
- повторный submit должен быть невозможен;
- при ошибке сохранения форма не должна терять введенные данные;
- retry action должен быть доступен для failed loading state;
- empty-state не должен использоваться вместо loading.

Где обязательно добавить loading/pending:

- загрузка дашборд-метрик;
- загрузка активных обходов;
- загрузка маршрутов;
- загрузка результатов;
- создание заявки;
- создание/сохранение маршрута;
- создание/сохранение точки;
- создание mobile account;
- reset password;
- сохранение планового обхода.

Критерии приемки:

- у ключевых форм есть `isSubmitting` состояние;
- у key tables есть loading placeholder;
- API error не приводит к белому экрану;
- пользователь понимает, идет загрузка или данных действительно нет.

### 9.4. Dirty state и несохраненные изменения

Цель:

Не дать оператору случайно потерять изменения при закрытии формы, переключении вкладки или выборе другой записи.

Формы, где dirty-state обязателен:

- создание/редактирование маршрута;
- создание/редактирование точки;
- создание заявки;
- создание/редактирование планового обхода;
- создание/редактирование сотрудника;
- создание mobile account;
- создание/редактирование пользователя сайта.

Требования:

- форма должна знать, отличается ли draft от исходного состояния;
- при закрытии dirty формы должен открываться confirm modal;
- при переключении выбранной строки, если текущая форма dirty, нужно спросить подтверждение;
- `Escape` не должен молча закрывать dirty modal;
- после успешного сохранения dirty-state сбрасывается;
- после `Отмена` с подтверждением draft очищается;
- при API error draft сохраняется.

Текст confirm:

```text
Есть несохраненные изменения
Если закрыть форму сейчас, введенные данные будут потеряны.

[Остаться] [Закрыть без сохранения]
```

Критерии приемки:

- dirty форма не закрывается без подтверждения;
- после сохранения confirm не появляется;
- переключение записи не теряет draft молча.

### 9.5. Search/filter behavior

Цель:

Фильтры должны работать одинаково на всех вкладках и быть готовы к серверной пагинации.

Общие требования:

- текстовый поиск должен иметь debounce 250-400 ms;
- `Enter` должен запускать поиск немедленно;
- кнопка `Сбросить` должна очищать все фильтры экрана;
- активные фильтры должны иметь счетчик;
- выбранные фильтры должны быть видимы пользователю;
- пустой результат фильтра должен показывать empty-state с кнопкой сброса;
- фильтры не должны очищаться при открытии detail drawer;
- при смене вкладки внутри экрана фильтры сохраняются;
- при reload допустимо сохранять фильтры в localStorage или URL query.

Фильтры по экранам:

| Экран | Фильтры |
|---|---|
| Дашборд | территория, смена, поиск по маршруту/сотруднику |
| Результаты | период, маршрут, сотрудник, статус, тип замечания, поиск |
| Назначения | дата, смена, специальность, тип маршрута, только свободные |
| Сотрудники | поиск, участок, должность, статус, мобильный доступ |
| Мобильные аккаунты | поиск, статус аккаунта, статус сессии, роль, scope |
| Маршруты | поиск, территория, статус, архив |
| Плановый обход | период, территория, маршрут, сотрудник, смена |
| Пользователи сайта | поиск, роль, статус |

API-ready поведение:

- filters state должен легко превращаться в query params;
- pagination должна сбрасываться на первую страницу при изменении фильтра;
- сортировка должна быть частью query state;
- frontend не должен смешивать client filter и server filter без явного режима.

Критерии приемки:

- на каждой таблице есть понятный reset filters;
- active filter count корректно считается;
- пустой filtered state отличается от полного отсутствия данных;
- search/filter logic вынесена из JSX при усложнении.

### 9.6. Таблицы, пагинация и плотность

Цель:

Все реестры должны выглядеть и работать одинаково.

Единый стандарт таблиц:

- compact плотность по умолчанию;
- высота строки: 44-52px;
- заголовок таблицы визуально отделен;
- строки кликабельны только если открывают detail;
- selected row имеет единый style;
- hover row не должен менять высоту;
- ячейки с длинным текстом должны переноситься или иметь ellipsis;
- numeric/progress колонки должны иметь фиксированную ширину;
- actions column всегда справа;
- если actions icon-only, нужен `aria-label`.

Пагинация:

- размеры страницы: 10, 20, 50;
- текущая страница видна;
- disabled state для невозможных переходов;
- при пустом списке пагинация скрыта или disabled;
- при смене фильтра страница сбрасывается на 1;
- в будущем должна поддерживать server pagination.

Сортировка:

- сортируемые колонки должны иметь indicator;
- сортировка по умолчанию должна быть понятна;
- для результатов: по времени убыванию;
- для активного обхода detail: точки по фактическому времени прохождения;
- для маршрутов: активные выше архивных, затем по названию;
- для сотрудников: активные/на смене выше offline.

Критерии приемки:

- таблицы выглядят единообразно;
- нет body-level horizontal scroll;
- selected row виден;
- actions не теряются при узком viewport;
- pagination не показывает недоступные страницы как активные.

### 9.7. Frontend error boundary

Цель:

При ошибке React-компонента пользователь не должен видеть пустой экран. Ошибка должна быть локализована на уровне экрана или приложения.

Требования:

- добавить общий `ErrorBoundary`;
- обернуть `ScreenRouter` или каждый screen-level route;
- fallback должен быть на русском языке;
- fallback должен иметь кнопку `Перезагрузить экран`;
- fallback должен иметь кнопку `На дашборд`;
- техническая ошибка может быть свернута в details для dev-режима;
- production fallback не должен показывать stack trace обычному пользователю.

Текст fallback:

```text
Не удалось отобразить экран
Произошла ошибка интерфейса. Данные не были удалены.

[Перезагрузить экран] [На дашборд]
```

Где ловить:

- ошибка рендера экрана;
- ошибка detail drawer;
- ошибка modal;
- ошибка неизвестного статуса/типа данных.

Критерии приемки:

- искусственная ошибка в одном экране не ломает весь app shell;
- пользователь может вернуться на дашборд;
- ошибка логируется в console для dev;
- fallback визуально соответствует общему стилю.

### 9.8. Audit-friendly UI

Цель:

Оператор и руководитель должны понимать, кто и когда создал или изменил важную запись.

Поля аудита в UI:

- создано кем;
- создано когда;
- изменено кем;
- изменено когда;
- источник изменения: web, mobile, system, import;
- история статусов;
- связанная заявка/обход/результат.

Где показывать:

- просмотр заявки;
- detail активного обхода;
- detail результата;
- карточка маршрута;
- карточка точки;
- карточка mobile account;
- пользователь сайта;
- плановая запись.

Критерии приемки:

- audit fields не мешают основному сценарию;
- история статусов отображается компактно;
- если данных аудита нет, показывается `Нет данных аудита`, а не пустота.

### 9.9. Export/print placeholders

Цель:

Заранее зафиксировать места будущей печати и выгрузки, не реализуя backend генерацию документов на UI-этапе.

Кнопки:

- `Экспорт XLSX`;
- `Экспорт CSV`;
- `Печать`;
- `Скачать PDF`;
- `Сформировать отчет`.

Где нужны:

- результаты обходов;
- заявки;
- маршруты и точки;
- плановый обход;
- сотрудники;
- mobile accounts audit/security events;
- пользователи сайта.

UI-правила:

- если backend не подключен, кнопки disabled или API-only placeholder;
- tooltip должен объяснять, что выгрузка появится после report API;
- нельзя показывать успешный toast вместо отсутствующей выгрузки;
- будущий export должен запускаться как job, а не мгновенная синхронная операция.

Критерии приемки:

- пользователь видит, где будет экспорт;
- disabled state не выглядит как ошибка;
- нет имитации скачивания без файла.

### 9.10. Notification center

Цель:

Собрать важные события смены в отдельную панель уведомлений, а не только в toast.

Типы уведомлений:

- новая заявка;
- заявка просрочена;
- активный обход задержан;
- потеря связи с мобильным клиентом;
- результат с замечанием;
- конфликт планирования;
- mobile account заблокирован;
- ошибка API/data source;
- завершение отчета/export job.

Требования:

- topbar notification button открывает panel/dropdown;
- уведомления имеют severity: info, warning, critical, success;
- можно отметить уведомление прочитанным;
- можно перейти к связанной сущности;
- empty-state: `Уведомлений пока нет`;
- toast используется для кратких событий, notification center - для сохраняемых событий.

Критерии приемки:

- кнопка уведомлений не является только toast;
- панель уведомлений открывается;
- уведомление может вести к заявке, обходу или результату;
- прочитанные уведомления визуально отличаются.

## 11. Матрица статусов UI

Цель:

Зафиксировать единые статусы, переходы и визуальные обозначения, чтобы разные вкладки не использовали разные названия и цвета для одной сущности.

### 11.1. Заявка на обход

| Статус | Когда используется | Допустимый следующий статус | UI tone |
|---|---|---|---|
| `Новая` | заявка создана оператором, сотрудник еще не начал обход | `Назначена`, `Отменена` | blue |
| `Назначена` | заявка отправлена сотруднику или подготовлена к старту | `В работе`, `Отменена` | violet |
| `В работе` | сотрудник начал прохождение маршрута | `Закрыта`, `Просрочена` | orange |
| `Просрочена` | срок/плановое время нарушены | `В работе`, `Закрыта` | red |
| `Закрыта` | обход завершен, результат принят | нет | green |
| `Отменена` | оператор отменил заявку | нет | slate |

Правила:

- `Новая` не может сразу стать `Закрыта`;
- `Отменена` и `Закрыта` являются финальными статусами;
- повторное создание заявки из одного результата запрещено, если уже есть связанная заявка.

### 11.2. Активный обход

| Статус | Когда используется | UI tone |
|---|---|---|
| `Ожидает` | заявка создана, обход еще не начат | slate |
| `В пути` | сотрудник начал обход | blue |
| `Задержка` | есть превышение планового времени | red |
| `Нет связи` | мобильный клиент не передает данные | orange |
| `Завершает` | маршрут почти завершен | violet |
| `Завершен` | все обязательные точки пройдены | green |
| `Отменен` | обход отменен оператором | slate |

Правила:

- активный обход создается из заявки;
- progress считается по фактически пройденным обязательным точкам;
- точки в detail drawer сортируются по фактическому времени первого скана.

### 11.3. Точка маршрута

| Статус | Значение | UI tone |
|---|---|---|
| `Активна` | точка используется в маршруте | green |
| `Черновик` | точка еще не готова для обхода | slate |
| `Повтор метки` | NFC/tag повторяется внутри текущего маршрута или предупреждается как повтор | orange |
| `Архив` | точка не используется в новых обходах | slate |

Правила:

- одинаковая NFC-метка разрешена в разных маршрутах;
- одинаковая NFC-метка внутри одного маршрута должна показывать предупреждение, но финальное решение о блокировке остается за backend validation;
- ручной контроль и QR-код имеют такие же права на участие в маршруте, как NFC.

### 11.4. Результат обхода

| Статус | Значение | UI tone |
|---|---|---|
| `Норма` | точка пройдена без замечаний | green |
| `С замечанием` | сотрудник оставил проблему/комментарий | orange |
| `Просрочено` | точка пройдена позже плана или не пройдена вовремя | red |
| `Не подтверждено` | нет достаточного подтверждения: фото, NFC, гео или факт скана | violet |

Правила:

- результат принадлежит активному обходу, маршруту и точке;
- из результата со статусом `С замечанием`, `Просрочено`, `Не подтверждено` можно создать заявку;
- из результата `Норма` создание заявки допустимо только вручную через отдельное действие.

### 11.5. Мобильный аккаунт

| Статус | Значение | UI tone |
|---|---|---|
| `Активен` | вход разрешен | green |
| `Не привязан` | аккаунт создан, но нет сотрудника/scope | orange |
| `Заблокирован` | вход запрещен | red |
| `Архив` | аккаунт отключен и скрыт из основной выдачи | slate |

Сессии:

| Статус | Значение | UI tone |
|---|---|---|
| `Онлайн` | активная мобильная сессия | green |
| `Офлайн` | нет активной сессии | slate |
| `Заблокирована` | сессия закрыта администратором | red |

Правила:

- мобильный аккаунт не равен сотруднику;
- аккаунт может быть привязан к одному, нескольким или всем сотрудникам;
- неактивность мобильного клиента сама не меняет статус сессии; завершение или блокировка должны приходить явным событием backend/mobile контура;
- reset password требует подтверждения.

### 11.6. Плановый обход

| Статус | Значение | UI tone |
|---|---|---|
| `Запланировано` | плановая запись активна | green |
| `Не запланировано` | ячейка пустая | slate |
| `Конфликт` | пересечение смен, сотрудника или маршрута | red |
| `Перенос` | обход перенесен на другую дату/смену | violet |
| `Исключение` | отпуск, больничный, ремонт, закрытая зона | orange |
| `Замена` | назначен другой сотрудник | blue |

Правила:

- конфликт не должен молча перезаписывать план;
- исключения должны быть видны в боковой панели;
- дневная и ночная смена должны иметь разные визуальные маркеры.

## 13. Ошибки, подтверждения и валидация

Цель:

Разделить, где нужен confirm modal, где inline validation, а где достаточно toast.

### 13.1. Confirm modal обязателен

| Действие | Причина |
|---|---|
| Удалить маршрут | может удалить точки и повлиять на будущие обходы |
| Удалить точку | меняет структуру маршрута |
| Удалить мобильный аккаунт | закрывает вход в мобильное приложение |
| Сбросить пароль мобильного аккаунта | меняет учетные данные |
| Заблокировать пользователя сайта | закрывает доступ в веб-панель |
| Отменить заявку на обход | может сорвать текущий обход |
| Очистить плановую сетку | массовое изменение данных |
| Перезаписать план смены | риск потери расписания |

Confirm modal должен показывать:

- заголовок действия;
- краткое описание последствий;
- имя/название сущности;
- primary danger action;
- cancel action;
- при критичном действии: текстовое подтверждение не требуется на UI-этапе, но может быть добавлено позже.

### 13.2. Inline validation обязательна

| Форма | Поля |
|---|---|
| Создание заявки | сотрудник, маршрут, дата |
| Создание маршрута | название, территория |
| Создание точки | название, зона/локация, тип точки |
| Создание mobile account | scope доступа, роль |
| Создание сотрудника | ФИО, табельный номер или уникальный код |
| Создание пользователя сайта | login, ФИО, роль |
| Плановый обход | маршрут, сотрудник, период, смена |

Правила:

- ошибка показывается рядом с полем;
- форма не отправляется при ошибках;
- кнопка submit может оставаться активной, но должна показать ошибки;
- пустое необязательное время заявки допустимо.

### 13.3. Toast разрешен

Toast используется только для:

- подтверждения успешного действия;
- короткого сообщения о временно недоступной API-only функции;
- уведомления о сохранении локального draft;
- сообщения о переключении источника данных mock/API.

Toast запрещен как единственная реакция для:

- создания;
- удаления;
- редактирования;
- открытия просмотра;
- назначения сотрудника;
- перехода к заявке;
- сброса пароля.

### 13.4. Ошибки API на будущем этапе

Frontend должен быть готов показывать:

- `400` как inline/form validation;
- `401` как переход к login/session expired;
- `403` как access denied state;
- `404` как entity not found empty-state;
- `409` как conflict modal или conflict panel;
- `422` как business validation;
- `500` как retry/error banner.

## 15. Frontend data contracts

Цель:

Зафиксировать минимальные DTO, которые нужны фронтенду для перехода от локального прототипа к ASP.NET Core API. Эти контракты не заменяют будущую OpenAPI-спецификацию, но задают форму данных, которую уже ожидает интерфейс.

Общие правила DTO:

- все идентификаторы серверных сущностей: `string` UUID;
- даты и время: ISO 8601 UTC от backend, локальное отображение во frontend;
- enum/status значения должны быть согласованы с матрицей статусов UI;
- nullable поля должны быть явно отмечены;
- вложенные массивы допустимы для детального просмотра, но списки должны возвращать компактные DTO;
- для таблиц нужны `total`, `page`, `pageSize`, `items`;

## 17. Local-to-API migration plan

Цель:

Заменить `localStorage` и mock-данные на API без переписывания экранов.

### 17.1. Текущая модель

Сейчас:

- `usePatrolWorkspaceData` управляет локальными маршрутами, заявками и активными обходами;
- `useStoredState` сохраняет временные данные в `localStorage`;
- `usePatrolDataSource` переключает `mock/api` snapshot;
- часть данных все еще приходит из `data.ts`.

### 17.2. Целевая модель

```text
screen -> feature hook -> repository -> api client -> backend
                         -> local repository -> localStorage
                         -> mock repository -> seed data
```

Правила:

- screen не вызывает `fetch`;
- screen не читает `localStorage`;
- screen не импортирует seed data напрямую, если это не временный fallback;
- feature hook отвечает за orchestration;
- repository отвечает за источник данных;
- api client отвечает за HTTP и DTO;
- domain helper отвечает за чистую бизнес-логику.

### 17.3. Этапы миграции

1. Создать repositories для маршрутов, заявок, активных обходов, результатов, сотрудников, mobile accounts.
2. Перенести localStorage операции из hooks в local repositories.
3. Перенести seed arrays в mock repositories.
4. Добавить API repositories с тем же интерфейсом.
5. Добавить feature hooks: `useRoutes`, `usePatrolRequests`, `useActivePatrols`, `useResults`, `useEmployees`, `useMobileAccounts`.
6. Переключать источник данных через единый `DataSourceMode`.
7. После появления backend сделать API source default.
8. Оставить mock/local source для dev/demo режима.

### 17.4. Optimistic update

Где допустимо:

- создание заявки;
- редактирование маршрута;
- изменение порядка точек;
- привязка сотрудника к mobile account;
- отметка уведомления прочитанным.

Где не допустимо без server response:

- reset password;
- удаление маршрута;
- блокировка пользователя;
- массовое автозаполнение плана;
- экспорт/отчет.

Правила:

- optimistic update должен иметь rollback;
- при `409 conflict` показывать conflict modal;
- при `422 validation` показывать inline ошибки;
- при network error сохранять draft, но не делать вид, что запись сохранена.

## 18. Frontend state management rules

Цель:

Зафиксировать, где живет каждый тип состояния.

| Тип состояния | Где хранить |
|---|---|
| Серверные списки и карточки | repository/query hook |
| Фильтры таблиц | URL query или localStorage по экрану |
| Открытая вкладка внутри экрана | local component state, при необходимости URL |
| Выбранная строка | local state или URL param для shareable views |
| Draft формы | local form state |
| Dirty state | form hook |
| Toast | global app-level toast |
| Notification center | repository/query hook |
| Current user/permissions | auth context |
| Theme/tokens | CSS variables |
| Data source mode | localStorage dev setting |

Правила:

- form draft не должен уходить в global state без причины;
- server state не должен дублироваться в нескольких useState;
- derived values считать через selector/helper, а не хранить отдельно;
- route/page filters должны быть сериализуемыми;
- localStorage key должен иметь version;
- при изменении версии localStorage должна быть migration или fallback.

## 19. API-ready error model для frontend

Backend должен возвращать ошибки в формате `application/problem+json`.

Минимальный формат:

```ts
interface ProblemDetailsDto {
  type?: string;
  title: string;
  status: number;
  detail?: string;
  instance?: string;
  traceId?: string;
  errors?: Record<string, string[]>;
  businessCode?: string;
}
```

Frontend mapping:

| HTTP | UI |
|---|---|
| 400 | form/global validation |
| 401 | session expired |
| 403 | access denied |
| 404 | not found empty-state |
| 409 | conflict modal |
| 422 | inline business validation |
| 429 | retry later banner |
| 500 | error banner + retry |

Требования:

- `traceId` показывать в dev details или support copy;
- `errors` маппить на поля формы;
- `businessCode` использовать для специальных UI-сообщений;
- fallback message должен быть на русском языке.

## 24. Интеграция нового UI-референса

Новый большой `Patrol360FullModule` и приложенные экраны используются как визуальный референс, а не как замена текущего приложения. Причина: в проекте уже есть рабочая архитектура `App -> ScreenRouter -> screens -> feature components -> repositories`, API/mock переключатель, локальные CRUD-сценарии и backend-контракты. Прямой перенос монолитного Tailwind-компонента сломает эту структуру и вернет фальшивые данные.

Что переносим из референса:

- четкие KPI-карточки с иконками и цветовой индикацией;
- плотную таблицу результатов с правой панелью деталей;
- аналитические панели результатов: типы замечаний, разбивка по сменам, последние аномалии;
- карточную структуру назначений: сотрудник, маршрут, параметры отправки, конфликт;
- визуальную модель мобильного аккаунта: аккаунт отдельно, привязанные сотрудники отдельно;
- ясные действия в правых панелях: создать, сохранить, открыть, удалить, привязать;
- компактные статусные chips и единый стиль кнопок.

Что не переносим:

- верхние фильтры даты, смены, территории и mock/backend в topbar, так как они были явно убраны из интерфейса;
- статические фальшивые строки там, где данных нет;
- TailwindCSS как отдельную зависимость;
- `lucide-react` как обязательную зависимость, пока в проекте есть собственная система SVG-иконок;
- монолитный компонент на несколько тысяч строк.

Текущее принятое решение:

- KPI-карточки получают собственные SVG-иконки через общий `StatTile`;
- экран результатов получает аналитические панели, которые строятся только от массива результатов;
- если данных нет, панели показывают честные пустые состояния;
- topbar остается очищенным: поиск, уведомления, пользователь панели.
