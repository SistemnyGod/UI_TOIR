# Модули и функциональные границы

Этот документ фиксирует продуктовые модули системы. Он нужен, чтобы backend, frontend, БД и будущие API-контракты развивались вокруг одних и тех же границ.

## 1. Dashboard / Operations

Назначение:

- общая оперативная сводка;
- активные обходы;
- просроченные обходы;
- выявленные замечания;
- покрытие дневной/ночной смены;
- быстрые переходы к назначению, плану, маршрутам и аккаунтам.

Текущий UI:

- `DashboardScreen`;
- таблица активных обходов;
- правая панель активного обхода;
- блок проблемных точек;
- быстрые действия.

Будущий API:

- `GET /api/v1/dashboards/summary`;
- `GET /api/v1/dashboards/operations`;
- SignalR updates.

## 2. Results / Inspections

Назначение:

- журнал результатов обходов;
- фильтрация по статусам, фото, сменам и маршрутам;
- детализация результата;
- замечания, просрочки, неподтвержденные точки;
- фото и хронология действий.

Текущий UI:

- `ResultsScreen`;
- таблица результатов;
- фильтры;
- detail drawer;
- аналитические нижние панели.

Будущий API:

- `GET /api/v1/results`;
- `GET /api/v1/inspections/{id}`;
- `GET /api/v1/issues`;
- file links для фото.

## 3. Assignments / Current Assignment

Назначение:

- назначение сотрудников на маршрут сейчас;
- проверка доступности сотрудника;
- выбор маршрута;
- предупреждения о конфликтах;
- отправка назначения в мобильный контур.

Текущий UI:

- `AssignmentScreen`;
- список сотрудников;
- список маршрутов;
- активные назначения;
- draft drawer назначения;
- локальный toast при назначении.

Будущий API:

- `POST /api/v1/assignments`;
- `POST /api/v1/assignments/{id}/start`;
- `POST /api/v1/assignments/{id}/cancel`;
- conflict-check endpoint или application policy.

## 4. Scheduling / Planned Patrol

Назначение:

- недельное и месячное планирование;
- дневные и ночные смены;
- исключения;
- корректировки;
- замены;
- конфликт двойного назначения.

Текущий UI:

- `ScheduleScreen`;
- tabs: неделя, месяц, исключения;
- сетка планирования;
- modal-like редактор ячейки;
- боковые блоки исключений и конфликтов.

Будущий API:

- `GET /api/v1/schedules`;
- `POST /api/v1/schedules`;
- `PUT /api/v1/schedules/{id}`;
- `POST /api/v1/schedule-exceptions`;
- recalculation job.

## 5. Mobile Accounts

Назначение:

- создать мобильный login/password;
- сбросить пароль;
- удалить/заблокировать аккаунт;
- привязать или отвязать сотрудника по ФИО;
- видеть статус аккаунта, сессии, устройства и версии приложения.

Текущий UI:

- `MobileAccountsScreen`;
- tabs: аккаунты, сессии, привязки;
- мастер создания;
- локальное действие `Привязать`;
- локальное действие `Сбросить пароль`.

Будущий API:

- `GET /api/v1/mobile-accounts`;
- `POST /api/v1/mobile-accounts`;
- `POST /api/v1/mobile-accounts/{id}/reset-password`;
- `POST /api/v1/mobile-accounts/{id}/attach-employee`;
- `POST /api/v1/mobile-accounts/{id}/detach-employee`;
- `DELETE /api/v1/mobile-accounts/{id}`.

## 6. Routes / Route Points / NFC

Назначение:

- CRUD маршрутов;
- вложенные точки маршрута;
- порядок точек;
- NFC/QR/manual тип точки;
- интервалы и требования фото/описания;
- разрешение повторного использования NFC-метки между разными маршрутами.

Текущий UI:

- `RoutesScreen`;
- список маршрутов;
- редактор маршрута;
- таблица точек;
- схема маршрута;
- drawer редактирования точки;
- явное уведомление, что одна NFC-метка может использоваться в разных маршрутах.

Будущий API:

- `GET /api/v1/routes`;
- `GET /api/v1/routes/{id}`;
- `POST /api/v1/routes`;
- `PUT /api/v1/routes/{id}`;
- `POST /api/v1/routes/{id}/points`;
- `PUT /api/v1/route-points/{id}`;
- `POST /api/v1/routes/{id}/versions`;

Важное правило:

- Повтор NFC между разными маршрутами разрешен.
- Повтор NFC внутри одной версии маршрута пока считается отдельным решением и должен показывать предупреждение.

## 7. Users / Employees / RBAC

Назначение:

- пользователи панели;
- сотрудники;
- роли и политики доступа;
- аудит прав.

Текущий UI:

- экраны сотрудников и пользователей сайта уже есть в `apps/web`;
- вкладки работают без фальшивого наполнения и показывают пустые состояния до подключения API;
- сотрудники будут использоваться в назначениях и мобильных аккаунтах после подключения справочника.

Будущий API:

- `GET /api/v1/users`;
- `GET /api/v1/employees`;
- `GET /api/v1/roles`;
- `GET /api/v1/permissions`;
- `POST /api/v1/user-roles`.

## 8. Files / Attachments

Назначение:

- фото обходов;
- вложения замечаний;
- отчеты;
- версии файлов.

Текущий UI:

- фото показаны как placeholders в Results.

Будущий API:

- `POST /api/v1/files`;
- `GET /api/v1/files/{id}`;
- `GET /api/v1/files/{id}/download`;
- `GET /api/v1/files/{id}/versions`.

## 9. Reports / Exports

Назначение:

- PDF/DOCX/XLSX/CSV;
- фоновые report jobs;
- история сформированных документов.

Текущий UI:

- кнопки `Экспорт` без backend.

Будущий API:

- `POST /api/v1/report-jobs`;
- `GET /api/v1/report-jobs/{id}`;
- `GET /api/v1/exports/{id}`.

## 10. Worklog

Статус:

- намеренно вне первого UI-прохода;
- в навигации текущей панели отсутствует.

Причина:

- первая итерация сфокусирована на обходах территории, маршрутах, назначениях, планировании и мобильных аккаунтах.
