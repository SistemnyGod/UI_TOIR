# Patrol360 Inventory -> СИЗ: жесткий технический аудит

Дата: 15.07.2026  
Область: активный web-сценарий СИЗ, API-контракт, EF persistence, DOCX/preview, mock-слой, тесты и CSS.  
Ограничения аудита: API не запускался, миграции не применялись, рабочая БД не использовалась.

## 1. Итоговый вердикт

Текущий модуль СИЗ компилируется и имеет работающий employee-centered экран, но его модель данных пока не соответствует утвержденному процессу учета СИЗ. В текущем виде модуль нельзя считать готовым к производственному ведению норм и печати юридически значимых карточек.

Главная причина: нормативная строка, номенклатура и факт выдачи связаны текстовыми эвристиками и обязательным `ItemId`, а нормативный порядок, группы, версии норм и реквизиты фактической выдачи не представлены устойчивыми сущностями.

Критичный вывод: зеленые typecheck/tests/build подтверждают техническую собираемость, но не бизнес-корректность. Часть тестов закрепляет взаимоисключающие правила browser preview и server DOCX.

## 2. Зафиксированное правило двух документов

### Личная карточка сотрудника

- содержит все применимые нормативные строки, включая невыданные;
- сохраняет исходный нормативный порядок;
- поддерживает корневые строки без искусственной категории;
- поддерживает условные группы и их дочерние строки;
- заголовок группы должен оставаться непосредственно перед своими строками;
- не сортируется по дате выдачи и не сортируется по алфавиту.

Пример структуры:

1. Корневые позиции.
2. `На наружных работах, зимой дополнительно:`.
3. Дочерние зимние позиции.
4. `При выполнении наружных работ, для защиты от атмосферных осадков дополнительно:`.
5. Дочерние позиции защиты от осадков.

### Лист подписи / карточка фактической выдачи

- содержит только выбранные и фактически выданные позиции;
- не содержит заголовки групп и невыданные нормы;
- сортируется по фактической дате выдачи;
- при одинаковой дате использует стабильный вторичный порядок: `IssueSequence`, затем идентификатор факта;
- нормативное наименование остается в первой колонке, а модель/марка/артикул относятся к фактически выбранной номенклатуре.

Текущая реализация смешивает эти два правила: общий `sortPpePrintLines` применяется и к личной карточке, и к листу подписи.

## 3. Критичные дефекты P0

### P0.1. API отбрасывает нормы, не попавшие в первые 50 позиций

Файл: `apps/api/Controllers/InventoryController.cs`, метод `PpeOptions`.

Алгоритм:

1. Загружается вся СИЗ-номенклатура.
2. В options передаются только первые 50 позиций.
3. `PositionNorms` фильтруются по `itemIds` этих 50 позиций.

Следствие: карточка сотрудника получает неполный набор норм в зависимости от сортировки номенклатуры. Это прямой дефект данных активного экрана.

### P0.2. Возможны несколько активных карточек одного сотрудника

Файлы:

- `apps/web/src/features/inventory/InventoryScreen.tsx`;
- `apps/web/src/features/inventory/InventoryPpeScreen.tsx`;
- `libs/infrastructure/Persistence/Inventory/EfInventoryWorkflowService.Issue.cs`;
- `libs/infrastructure/Persistence/Patrol360DbContext.cs`.

Web загружает только первую страницу из 25 карточек и считает отсутствие сотрудника на этой странице отсутствием карточки. `ensureCard` после этого создает новую карточку. Backend не проверяет существующую активную карточку, а индекс `(employee_id, archived_at)` не уникальный.

Результат: сотруднику можно создать несколько активных карточек, а факты и печать окажутся разделены между ними.

### P0.3. Гонка при быстром переключении сотрудников

Файл: `apps/web/src/features/inventory/InventoryPpeScreen.tsx`, `loadEmployeeCard`.

Асинхронная загрузка не имеет AbortController или request token. Поздний ответ предыдущего сотрудника может записать `detail` уже после выбора следующего. Затем `ensureCard` способен вызвать `updatePpeCard` с новым `employeeId` и перепривязать чужую карточку.

Backend это допускает: `UpdatePpeCard` изменяет `EmployeeId` существующей карточки.

Риск: повреждение данных, а не только неправильное отображение.

### P0.4. Норма и факт связываются изменяемым текстовым ключом

Файлы:

- `apps/web/src/features/inventory/InventoryPpeScreen.tsx`, `buildEmployeePpeRows`;
- `apps/web/src/features/inventory/ppe/ppeNormMapping.ts`.

Ключ строится из должности, нормативного наименования и пункта нормы. Исправление опечатки, изменение формулировки или наличие одинаковых строк может:

- потерять связь с фактом выдачи;
- создать orphan row;
- связать факт с другой нормой.

В строке выдачи отсутствует стабильный `NormId`/`NormRevisionRowId`.

### P0.5. Нормативный порядок не хранится и уничтожается сортировкой

Файлы:

- `apps/web/src/features/inventory/ppe/ppePrintMapping.ts`;
- `libs/infrastructure/Persistence/Inventory/EfInventoryWorkflowService.Issue.cs`;
- `libs/infrastructure/Persistence/Inventory/EfInventoryExportService.Print.cs`;
- `libs/infrastructure/Persistence/EfInventoryCatalogQuery.cs`.

Подтверждено:

- web preview сортирует все строки по `IssuedAt`, затем по названию;
- backend detail сортирует строки по `Item.Name`;
- backend DOCX личной карточки сортирует по `Item.Name`;
- настройки норм сортируются по должности и имени товара;
- в таблицах нет `SortOrder` и `ParentGroupId`.

Группу и ее дочерние строки невозможно надежно удержать вместе.

### P0.6. Статусы browser preview и server DOCX противоречат друг другу

Файлы:

- `apps/web/src/features/inventory/ppe/ppeStatusCatalog.ts`;
- `apps/web/src/features/inventory/ppe/ppePrintMapping.ts`;
- `libs/infrastructure/Persistence/Inventory/PpeIssueStatusCatalog.cs`;
- `apps/web/src/__tests__/ppePrintMapping.test.ts`;
- `tests/Patrol360.Infrastructure.Tests/InventoryPpePrintDbIntegrationTests.cs`.

Наборы строк листа подписи различаются:

- один web predicate включает `partial`;
- другой web predicate дополнительно включает `returned` и `written_off`;
- backend включает только `issued`, `replacement`, `reissued`.

Web unit tests требуют печатать returned/written-off, а DB integration tests требуют их исключать. Оба набора тестов могут быть зелеными, потому что проверяют разные реализации.

Дополнительный риск: web preview подставляет текущую дату, если статус считается выданным, а `IssuedAt` пустой. Документ может показать дату, которой пользователь не вводил.

### P0.7. Способ выдачи и размер вводятся, но не сохраняются

Файл: `apps/web/src/features/inventory/InventoryPpeScreen.tsx`, `buildLinePayload`.

Форма содержит `issueMethod` и `sizeText`, однако payload их не отправляет. В DTO и EF entity таких полей нет. Печать пытается вывести `Лично/Дозатор` эвристикой по `isConsumable`.

Следствие: выбор пользователя теряется после сохранения, а печатный документ может содержать неверный способ выдачи.

### P0.8. Модель нормы требует ровно одну номенклатурную позицию

Файлы:

- `libs/contracts/InventoryContracts.cs`;
- `libs/infrastructure/Persistence/Entities/InventoryWorkflowEntities.cs`;
- `libs/infrastructure/Persistence/Patrol360DbContext.cs`;
- `libs/infrastructure/Persistence/Inventory/EfInventoryCatalogCommandService.cs`.

`InventoryPositionNorm` имеет обязательный `ItemId` и unique key `(PositionName, ItemId)`. Это противоречит утвержденной модели:

- норма существует независимо от каталога;
- одной норме соответствует несколько допустимых товаров;
- конкретный товар выбирается при выдаче;
- заголовок группы не является товаром.

## 4. Высокие дефекты P1

### P1.1. Mapping хранится только в localStorage

Связь нормы с номенклатурой:

- видна только в конкретном браузере;
- не переносится на другое рабочее место;
- не видна серверной DOCX-печати;
- не имеет автора, даты, версии и аудита;
- поддерживает только один товар на норму.

### P1.2. Структура групп представлена одним boolean

`IsSectionTitle` не различает:

- обычную корневую позицию;
- условную группу;
- дочернюю позицию группы;
- информационное примечание.

Эвристика `название заканчивается двоеточием` ненадежна. Нужны `RowType`, nullable `ParentGroupId`, `SortOrder` и стабильный row id.

### P1.3. Редактор норм не способен собрать утвержденный документ

`InventorySettingsScreen.tsx` позволяет изменить только:

- должность;
- одну позицию номенклатуры;
- числовое количество;
- срок в месяцах.

Не редактируются нормативное наименование, пункт, исходная периодичность, количество текстом, тип строки, группа, порядок, область действия, версия, автор и дата подтверждения.

### P1.4. Нет versioned norm set и публикации

Отсутствуют:

- источник и версия набора норм;
- effective date;
- draft/published status;
- автор и дата подтверждения;
- snapshot нормы в карточке на историческую дату;
- employee override.

Изменение общей нормы нельзя безопасно отделить от старых карточек и выдач.

### P1.5. Должность связывается точным текстом

Нормы выбираются сравнением нормализованных строк `positionName`. Нет стабильного position id, alias table и интерфейса подтверждения несовпадений. Опечатка или вариант названия дает пустую карточку без явной ошибки.

### P1.6. Возврат и списание не имеют необходимых реквизитов

В строке есть только `IssuedAt` и `DueAt`. Отсутствуют фактическая дата и количество возврата, акт списания, номер акта и подпись. Web использует `DueAt` как дату возврата и полное количество выдачи как возвращенное количество, хотя `DueAt` одновременно является сроком эксплуатации.

### P1.7. В проекте два параллельных PPE-сценария

Активен большой `InventoryPpeScreen.tsx`, при этом старый wizard/journal остается в `apps/web/src/features/inventory/ppe`. Общие функции печати и статусов используются неодинаково, что увеличивает расхождение поведения и тестов.

## 5. Средние дефекты P2

- История карточки и строк загружается по 50 записей, объединяется без de-duplication и без полной пагинации.
- Mock history не всегда ограничивается конкретной карточкой/строкой.
- Mock сохранение может заменить введенный `IssuedAt` текущей датой и расходится с API.
- Частичная выдача одновременно учитывается как `выдано` и `частично`; KPI `не выдано` становится спорным.
- Количество больше нормы не требует подтверждения.
- Массовых действий `Выдано все` и `Не выдано` нет.
- PPE UI состоит из 22 CSS-файлов примерно на 8718 строк; итоговый inventory CSS build составляет около 451 KB. Каскад поздних override-слоев повышает риск визуальных регрессий.
- `InventoryPpeScreen.tsx` остается монолитом примерно на 1700 строк.

## 6. Что в текущей реализации сделано правильно

- сотрудник является центром активного сценария;
- нормы, личная карточка, лист подписи, история и print review разделены вкладками;
- section rows исключаются из действия выдачи;
- невыданные строки остаются в личной карточке;
- mapping сам по себе не создает факт выдачи;
- складской остаток не блокирует выдачу;
- `BrandModelArticle`, нормативное имя, пункт, период и printable quantity уже частично проходят через контракт;
- пустые размеры сотрудника не должны приводить к падению web preview;
- backend ведет события смены статуса строки;
- обычная API-модель допускает фактический `IssuedAt` для ряда выданных статусов.

## 7. Целевая минимальная модель

### NormSet

- `Id`, `Name`, `SourceFile`, `SourceHash`;
- `Revision`, `EffectiveFrom`, `EffectiveTo`;
- `Status: draft | published | archived`;
- `ApprovedBy`, `ApprovedAt`, `CreatedAt`, `UpdatedAt`.

### NormRow

- `Id`, `NormSetId`, `PositionId/PositionKey`;
- `RowType: item | condition_group | note`;
- nullable `ParentGroupId`;
- `SortOrder`;
- `NormItemName`, `NormPoint`;
- raw и normalized quantity/unit/period;
- `AppliesToAllPositionEmployees`;
- source sheet/row для аудита.

### NormCatalogMapping

- `NormRowId`, `ItemId`;
- `IsDefault`, `IsActive`;
- optional model/article defaults;
- автор, дата, комментарий.

Одна `NormRow` имеет несколько mappings.

### EmployeeNormOverride

- `EmployeeId`, `NormRowSnapshot` или `NormRowId`;
- `SortOrder`, причина, автор, дата;
- не изменяет общий набор должности.

### PpeIssueFact

- `Id`, `CardId`, `NormRowId`, `NormRevisionId`, `ItemId`;
- `IssuedAt`, отдельный `CreatedAt`;
- `IssueSequence` для стабильной сортировки одинаковых дат;
- quantity/unit/size/model/article/price;
- `IssueMethod: personal | dispenser`;
- status/comment.

### Return/WriteOff operation

- отдельная операция с датой, количеством, реквизитами акта и комментарием;
- не переиспользует `DueAt`.

## 8. Правильный порядок переработки

### Этап A. Stop-the-bleeding без миграции

1. Перестать фильтровать нормы первыми 50 items в `PpeOptions`.
2. Загружать карточку по `employeeId`, а не искать ее в первой странице списка.
3. Запретить перепривязку существующей карточки другому сотруднику.
4. Добавить защиту от stale async response.
5. Разделить функции сортировки personal card и signature sheet.
6. Унифицировать набор печатных статусов web/backend/tests.

### Этап B. Утверждаемая DB/API migration

1. Versioned NormSet/NormRow с `RowType`, `ParentGroupId`, `SortOrder`.
2. Persisted many-to-many NormCatalogMapping.
3. Stable `NormRowId` в факте выдачи.
4. `IssueMethod`, `SizeText`, `IssueSequence`.
5. Return/writeoff operations.
6. Unique active PPE card per employee.

### Этап C. Dry-run импорта актуального Excel

1. Импортировать 449 строк без записи в рабочую БД.
2. Сохранить source sheet/row/raw values.
3. Показать пользователю должности, группы и спорные количества.
4. Публиковать только после ручного подтверждения.

### Этап D. Конструктор норм и карточки

- добавить корневую позицию;
- добавить условную группу;
- добавить позицию в группу;
- переместить строку в группу или корень;
- менять порядок drag/controls;
- выбрать несколько допустимых catalog items;
- создать employee override;
- массовое `Выдано все` только через preview и подтверждение даты/способа/товара.

### Этап E. Печать

- personal card: `SortOrder`, все нормы и группы;
- signature sheet: `IssuedAt`, затем `IssueSequence`;
- нормативное основание берется из опубликованной revision;
- документ воспроизводится на историческую дату по snapshot/version.

## 9. Проверки текущего состояния

- `npm run typecheck --prefix apps\web`: passed.
- `npm run test:unit --prefix apps\web -- --run`: passed, 13 files / 128 tests.
- `npm run build --prefix apps\web`: passed.
- `dotnet build .\Patrol360.slnx --no-restore`: passed, 1 analyzer warning.
- `dotnet test .\Patrol360.slnx --no-build`: passed; 84 tests passed, 58 DB integration tests skipped.

DB-связка СИЗ, включая DOCX и persistence, этим запуском не подтверждена: соответствующие integration tests пропущены без отдельной тестовой БД.

## 10. Решение

Не начинать косметическую перерисовку текущего СИЗ до устранения P0-контрактов. Первый практический этап должен быть ограниченным `PPE stop-the-bleeding`: полнота options, получение карточки по сотруднику, защита от дублей/гонок, две независимые сортировки и единый статусный канон. После этого требуется отдельное подтверждение migration proposal для versioned norms и фактов выдачи.
