# Аудит модуля «Обходы»

Дата: 15.07.2026  
Объект: web, API, PostgreSQL persistence, worker и Android-клиент модуля «Обходы»  
Формат: read-only аудит, без изменения продуктового кода

## Резюме

Модуль функционально сформирован, а базовые серверные сценарии и DB-интеграция работают. Однако текущую ветку нельзя считать готовой к выпуску: есть подтверждённый регресс планирования, дашборд показывает отменённые заявки как ожидающие и «ближайшие», журнал результатов не даёт перейти дальше первой страницы, а часть реализации расходится с утверждённым baseline по версиям маршрутов, статусам и файлам.

Рекомендуемый порядок: сначала закрыть P0 и выпустить короткий стабилизационный патч, затем исправить API-контракты и модель версий маршрутов, после этого — UX/accessibility и эксплуатационное усиление.

## Что проверено

- Фактический web-интерфейс в API-режиме: дашборд, результаты, маршруты; DOM и интерактивные состояния.
- Контролируемый mock-сценарий: дашборд → назначения → расписание.
- Backend и persistence: назначения, заявки, результаты, маршруты, дашборд, вложения.
- Актуальные незакоммиченные изменения модуля.
- Агрегаты локальной PostgreSQL только через `SELECT`.
- Автоматические проверки:
  - .NET: 57 тестов пройдено, 56 пропущено без DB-флага;
  - DB integration: 60 из 60 пройдено на временных БД;
  - web typecheck и build: пройдены;
  - web unit: 115 пройдено, 1 упал;
  - mobile typecheck и lint: пройдены;
  - проверка UTF-8: 1309 файлов пройдено;
  - `git diff --check`: замечаний нет.

Ограничение доказательств: live-приложение исследовано по DOM и интерактивным состояниям, но CDP-снимок live-вкладки дважды завершился тайм-аутом. Визуальные снимки ниже сделаны с текущей сборки в контролируемом mock-режиме. Полный accessibility-аудит со screen reader и мобильными устройствами не выполнялся.

## Маршрут проверки интерфейса

1. Дашборд смены — **частично работоспособен**. KPI и списки загружаются, но отменённые заявки ошибочно попадают в «Ожидают назначения» и «Ближайшие назначения».
2. Назначения — **работоспособен с UX-тупиком**. При включённом по умолчанию «Только доступные» список сотрудников может стать пустым, хотя ниже видны назначения; важные поля в строках обрезаются.
3. Плановый обход — **нестабилен между режимами данных**. Хук всегда обращается к API даже в mock-режиме, из-за чего контролируемый сценарий получает «Требуется авторизация».
4. Результаты — **работоспособна только первая страница**. Старые записи нельзя открыть через пагинацию.
5. Маршруты и точки — **основная функция работает**, но выбор строк и изменение порядка недоступны с клавиатуры.

## Проблемные места

### P0 — блокеры выпуска

#### 1. Регресс истории результатов в плановом обходе

**Факт.** Unit-тест `opens schedule planning modal from an empty cell and sends notification payload` падает: блок «История результатов за день» отсутствует. Текущее изменение требует одновременного совпадения сотрудника и маршрута (`&&`) в [ScheduleScreen.tsx](../apps/web/src/features/patrol/ScheduleScreen.tsx#L102), хотя пустая ячейка может ещё не иметь обеих координат. Тест фиксирует ожидаемый сценарий в [ui.test.tsx](../apps/web/src/__tests__/ui.test.tsx#L750).

**Риск.** Диспетчер не видит замечание предыдущего обхода и не может создать повторное назначение «по результату».

**Исправление.** Ввести явную стратегию сопоставления: для заполненной ячейки — сотрудник + маршрут, для пустой — выбранная координата или маршрут/сотрудник с понятным fallback. Покрыть отдельными тестами пустую ячейку, ячейку сотрудника, ячейку маршрута и отсутствие совпадений.

#### 2. Отменённые заявки показаны как ожидающие и ближайшие

**Факт.** На 15.07.2026 дашборд показывает две ожидающие заявки с датами 30.06 и 02.07. В БД открытых назначений и заявок нет: назначения имеют только статусы «Завершено» (18) и «Отменено» (8), заявки — «Завершено» (18), «Закрыта» (6), «Отменено» (2). Функция [isOpenRequest](../apps/web/src/features/dashboard/DashboardScreen.tsx#L882) исключает только «Закрыта/closed», поэтому «Завершено» и «Отменено» считаются открытыми. Затем первые четыре записи без проверки будущей даты выводятся в [UpcomingAssignments](../apps/web/src/features/dashboard/DashboardScreen.tsx#L545).

**Риск.** Диспетчер получает неверную оперативную картину и может повторно отрабатывать уже закрытую работу.

**Исправление.** Использовать один доменный набор состояний `open/terminal`, а не строковый отрицательный фильтр. Для «Ближайших» дополнительно фильтровать `scheduledAt >= now`, сортировать по возрастанию времени и вынести правило в тестируемый селектор.

### P1 — высокий приоритет

#### 3. Журнал результатов фактически обрезан первой страницей

**Факт.** Репозиторий делает два запроса — за первой и второй страницами — но возвращает только первую; вторая используется лишь как признак `hasMore` ([resultsRepository.ts](../apps/web/src/repositories/resultsRepository.ts#L74)). Интерфейс вместо пагинации предлагает уточнить фильтр ([ResultsWorkspace.tsx](../apps/web/src/features/patrol/results/ResultsWorkspace.tsx#L365)).

**Риск.** Исторический результат невозможно найти, если оператор не угадает фильтр; каждый поиск при этом создаёт два HTTP-запроса.

**Исправление.** Вернуть envelope `{items, total, page, pageSize}` или cursor; добавить «Далее/Назад» либо «Загрузить ещё»; фильтрацию и сортировку выполнять на сервере. Критерий: любой результат доступен без изменения поискового запроса.

#### 4. Новые серверные фильтры не используются web-клиентом

**Факт.** UI вызывает назначения и заявки с пустыми фильтрами ([useAssignmentsWorkspace.ts](../apps/web/src/hooks/useAssignmentsWorkspace.ts#L137), [usePatrolWorkspaceData.ts](../apps/web/src/hooks/usePatrolWorkspaceData.ts#L161)). Оба репозитория затем выгружают все страницы циклом ([assignmentsRepository.ts](../apps/web/src/repositories/assignmentsRepository.ts#L100), [patrolRequestsRepository.ts](../apps/web/src/repositories/patrolRequestsRepository.ts#L98)).

**Риск.** Время открытия и нагрузка растут линейно со всей историей; фильтры в текущем изменении не решают масштабирование.

**Исправление.** Передавать активные фильтры, диапазон календаря и страницу в API; использовать React Query/SWR-подобный cache key; отменять устаревшие запросы. Добавить нагрузочный тест хотя бы на 100 тыс. назначений.

#### 5. Фильтр календарной даты режет данные по UTC, а не по бизнес-зоне

**Факт.** `DateOnly` превращается в полночь UTC в [EfPatrolStore.Assignments.cs](../libs/infrastructure/Persistence/Patrol/EfPatrolStore.Assignments.cs#L47). Baseline требует хранить время в UTC, но отображать и определять календарные границы по локальной зоне ([tz-normalization.md](tz-normalization.md#L317)).

**Риск.** Назначение в 00:00–04:59 по Екатеринбургу попадёт в предыдущую UTC-дату; фильтр «за день» и календарь будут расходиться.

**Исправление.** Передавать/конфигурировать IANA/Windows business timezone и преобразовывать локальные границы дня в UTC. Покрыть тестами UTC+5, ночную смену, переход года и DST-зону как переносимый случай.

#### 6. Создание назначения не защищено от конкурентного дубля на уровне API

**Факт.** Сначала выполняется поиск существующего назначения, затем `Add` и `SaveChanges` ([EfPatrolStore.Assignments.cs](../libs/infrastructure/Persistence/Patrol/EfPatrolStore.Assignments.cs#L146)). Между проверкой и сохранением два запроса могут пройти одновременно. `Create` в [AssignmentsController.cs](../apps/api/Controllers/AssignmentsController.cs#L35) не обрабатывает `DbUpdateException`; обработка 409 есть только для команд состояния.

**Риск.** При двойном клике/ретрае один запрос может получить 500 вместо идемпотентного результата или понятного 409.

**Исправление.** Добавить idempotency key или атомарный upsert по заявке, обработать unique violation и вернуть уже созданное назначение/409. Проверить параллельным интеграционным тестом.

#### 7. Версионирование маршрута не соответствует утверждённой модели

**Факт.** Назначение хранит `RouteVersionNo`, но не неизменяемую `route_version_id`; при изменении живого маршрута завершение блокируется сравнением номера версии. Baseline требует фиксировать конкретную версию и сохранять исторические точки ([tz-normalization.md](tz-normalization.md#L99)).

**Риск.** Сотрудник может выполнить обход офлайн, а после редактирования маршрута не сможет синхронизировать результат; исторические данные зависят от текущих точек.

**Исправление.** Добавить `route_versions`/`route_point_versions`, связывать назначение и результаты с immutable revision. Изменение маршрута должно влиять только на новые назначения.

#### 8. Расписание нарушает границу mock/API

**Факт.** [useSchedulePlanning.ts](../apps/web/src/hooks/useSchedulePlanning.ts#L36) всегда создаёт API-репозиторий и вызывает `getReferences()` при mount, хотя компонент уже получает каталоги из выбранного источника. В mock-сценарии это приводит к состоянию «Требуется авторизация».

**Риск.** Деморежим, тесты и fallback-режим нестабильны; 401 из дочернего запроса может сбросить общую сессию.

**Исправление.** Передавать `dataSourceMode`; в mock-режиме не создавать API-запрос. Централизовать реакцию на 401 и не очищать сессию из запросов, не относящихся к текущему источнику.

#### 9. Критичные операции маршрута недоступны с клавиатуры

**Факт.** Строка таблицы выбирается через `onClick` без focus/keyboard semantics, а кнопки порядка имеют только символы ↑/↓ без контекстного имени ([RoutePointTable.tsx](../apps/web/src/features/patrol/components/routes/RoutePointTable.tsx#L35)). В результатах есть вложенный `<main>`, неполная tab-semantics и одинаковое имя меню каждой строки ([ResultsWorkspace.tsx](../apps/web/src/features/patrol/results/ResultsWorkspace.tsx#L328)).

**Риск.** Клавиатурные и screen-reader пользователи не могут полноценно работать; семантика страницы неоднозначна.

**Исправление.** Сделать название точки ссылкой/кнопкой, добавить `aria-label="Переместить <точку> вверх"`; заменить внутренний `main` на `div/section`; добавить `role=tab`, `aria-selected`, связь tab/tabpanel; включить идентификатор маршрута/сотрудника в имя row-menu.

### P2 — средний приоритет

#### 10. Статусы хранятся и сравниваются как русские строки

Baseline требует `smallint + enum mapping`, но persistence и UI поддерживают наборы строк и синонимов. Ошибка `isOpenRequest` — уже проявившееся следствие дрейфа статусов.

**Исправление.** Ввести enum/code в API и БД, миграцию текущих значений и единый state machine. Отображаемые русские названия оставить только в presentation-слое.

#### 11. Вложения пишутся на диск до фиксации транзакции БД

**Факт.** [EfPatrolStore.Results.cs](../libs/infrastructure/Persistence/Patrol/EfPatrolStore.Results.cs#L325) вызывает `File.WriteAllBytes`; при последующем конфликте/ошибке БД останется orphan-файл. Это также расходится с baseline MinIO + scan pipeline.

**Исправление.** Перенести файлы в объектное хранилище, хранить `Pending`, подтверждать через outbox/worker и очищать незафиксированные объекты.

#### 12. KPI результатов требуют правил качества данных

В live-выборке встречаются длительности 0–2 минуты и 23 ч 20 мин, при этом UI считает среднее 3 ч 1 мин «по валидным длительностям». У маршрутов отображается 0 км. Без признака импорта, правил выбросов и расшифровки знаменателя KPI выглядит точным, но не является надёжным управленческим показателем.

**Исправление.** Зафиксировать формулы, показывать размер выборки и исключённые записи, валидировать `finishedAt >= startedAt`, маркировать импорт, вынести data-quality предупреждения.

#### 13. Поиск и persistence синхронны

`ToLower().Contains()` затрудняет использование индекса PostgreSQL, а EF-вызовы выполняются синхронно и без `CancellationToken`.

**Исправление.** Использовать `ILIKE`/нормализованное поле и trigram-индекс; перевести горячие endpoints на async и прокинуть cancellation.

## Что сделано хорошо

- DB-интеграционные тесты действительно поднимают временные PostgreSQL и покрывают жизненный цикл назначения, конфликты смен, обязательные точки и завершение.
- Есть optimistic lock для команд изменения назначения и читаемый 409 на concurrency conflict.
- Уникальность NFC ограничена маршрутом, что соответствует продуктовой договорённости.
- Текущий web build/typecheck, mobile typecheck/lint и проверка кодировок проходят.
- Архитектура остаётся modular monolith; переписывание на микросервисы для исправления найденных проблем не требуется.

## План исправления

### Этап 0. Стабилизация выпуска — 1–2 рабочих дня

1. Исправить сопоставление истории в календаре и вернуть web unit suite в 116/116.
2. Ввести единый terminal-status selector; исключить завершённые/отменённые заявки и прошлые даты из «Ближайших».
3. Отключить API-вызов расписания в mock-режиме.
4. Добавить smoke-тест: dashboard → schedule → result → повторная заявка.

**Критерий выхода:** все текущие тесты зелёные; на БД с нулём открытых заявок дашборд показывает 0 и не выводит прошлые даты.

### Этап 1. Контракты списка и календарное время — 3–5 дней

1. Ввести paged envelope для результатов, назначений и заявок.
2. Подключить UI-фильтры и диапазон календаря к серверным параметрам; убрать выгрузку всей истории.
3. Реализовать business timezone service и единое преобразование границ `DateOnly`.
4. Добавить идемпотентное создание назначения и параллельный DB-тест.

**Критерий выхода:** журнал листается до последней записи; один экран делает один запрос страницы; 100 тыс. строк не выгружаются в браузер; даты UTC+5 проходят контрактные тесты.

### Этап 2. Доменная целостность и offline — 5–8 дней

1. Мигрировать статусы на коды/enum и оформить state machine переходов.
2. Реализовать immutable версии маршрутов и точек; назначение фиксирует `route_version_id`.
3. Проверить сценарий: назначение скачано → маршрут изменён → обход завершён офлайн → синхронизация успешна против старой версии.
4. Перевести вложения на object storage + outbox/cleanup.

**Критерий выхода:** исторический результат не меняется после редактирования маршрута; повторная доставка команды безопасна; orphan-файлы автоматически удаляются.

### Этап 3. UX и accessibility — 3–5 дней

1. Убрать дубли page header/действий на странице результатов и вложенный `main`.
2. Добавить корректную tab/tabpanel и keyboard-навигацию таблиц.
3. Сделать фильтр «Только доступные» объяснимым и не оставлять пользователя в пустом списке без причины/действия.
4. Устранить обрезание ФИО, маршрута и времени в назначениях; добавить русские формы `точка/точки/точек`.
5. Проверить 1440×900, 1280×720, 200% zoom и screen reader smoke.

**Критерий выхода:** основные сценарии выполняются только клавиатурой; axe smoke не находит critical/serious; ключевые идентификаторы не скрыты без tooltip/раскрытия.

### Этап 4. Эксплуатация и качество данных — постоянно, первый срез 2–3 дня

1. Зафиксировать определения KPI и правила выбросов.
2. Добавить метрики latency/error/row count, tracing и алерты на ошибки синхронизации.
3. Сделать DB integration обязательным CI-gate для изменений persistence.
4. Добавить live smoke после deploy: авторизация, дашборд, создание/старт/завершение, результат и фото.

## Рекомендуемый контрольный набор тестов

- Unit: классификация каждого статуса; календарное сопоставление результата; русские формы числительных.
- API contract: paging envelope, серверные фильтры, `dateFrom/dateTo` в business timezone.
- DB integration: конкурентное создание, все переходы state machine, immutable route revision.
- Web E2E: пустое состояние, прошедшая/будущая заявка, пагинация результатов, 401 без глобального logout.
- Mobile/offline: повторная отправка, изменение маршрута после download, вложение и частичный retry.
- Accessibility: keyboard-only + axe для dashboard, schedule, results, routes.

## Решение по выпуску

**No-go для текущего состояния.** После Этапа 0 возможен ограниченный стабилизационный выпуск. Для промышленной эксплуатации с растущей историей и offline-мобильным обходом нужны как минимум Этапы 1 и 2.
