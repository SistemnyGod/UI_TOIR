# PERCo-Web: памятка по модулю интеграции

## Назначение модуля

Модуль `PERCo-Web` отвечает за подключение к системе контроля доступа PERCo, синхронизацию сотрудников и проходов, построение интервалов присутствия сотрудников на заводе и передачу этих данных в рабочие сценарии проекта.

Основная задача модуля: понимать, кто вошел на завод, кто вышел, кто сейчас находится на территории, какие проходы не сопоставлены с сотрудниками проекта и какие интервалы требуют ручной проверки.

## Таблицы базы данных

Основные таблицы PERCo:

```txt
perco_integration_settings
perco_integration_logs
perco_sync_state
perco_employee_links
perco_access_events
employee_presence_intervals
```

Назначение таблиц:

- `perco_integration_settings` - настройки подключения, периодичность синхронизации, найденные endpoints, статус последней проверки.
- `perco_integration_logs` - журнал операций: проверка подключения, sync сотрудников, sync проходов, ошибки.
- `perco_sync_state` - последняя успешная точка синхронизации по типам данных.
- `perco_employee_links` - связь сотрудников PERCo с сотрудниками проекта.
- `perco_access_events` - сырые нормализованные события входа/выхода.
- `employee_presence_intervals` - построенные интервалы присутствия сотрудника на заводе.

## API endpoints

Все endpoints находятся под:

```txt
/api/v1/integrations/perco
```

Список маршрутов:

```http
GET  /settings
PUT  /settings
POST /test-connection
GET  /secret-status
POST /sync-employees
POST /sync-events
GET  /unmatched-employees
POST /match-employee
GET  /logs
GET  /diagnostics
POST /presence-intervals/{intervalId}/close
```

Назначение:

- `GET /settings` - получить настройки без раскрытия пароля/токена.
- `PUT /settings` - сохранить настройки подключения.
- `POST /test-connection` - проверить PERCo и `/dev`.
- `GET /secret-status` - проверить, читается ли сохраненный секрет.
- `POST /sync-employees` - синхронизировать сотрудников PERCo.
- `POST /sync-events` - синхронизировать проходы.
- `GET /unmatched-employees` - получить несопоставленных сотрудников PERCo.
- `POST /match-employee` - вручную связать сотрудника PERCo с сотрудником проекта.
- `GET /logs` - журнал операций и ошибок.
- `GET /diagnostics` - диагностический экран проходов, интервалов и спорных данных.
- `POST /presence-intervals/{intervalId}/close` - ручное закрытие зависшего интервала.

## Права доступа

Для PERCo добавлены отдельные permissions:

```txt
integrations.perco.view
integrations.perco.manage
integrations.perco.sync
integrations.perco.match
integrations.perco.logs.view
```

Логика:

- `integrations.perco.view` - просмотр настроек и диагностики.
- `integrations.perco.manage` - изменение настроек, проверка подключения, ручное закрытие интервалов.
- `integrations.perco.sync` - запуск синхронизации сотрудников и проходов.
- `integrations.perco.match` - сопоставление сотрудников PERCo с сотрудниками проекта.
- `integrations.perco.logs.view` - просмотр журнала PERCo.

Администратор получает все права PERCo через seed.

## Как работает подключение

В настройках PERCo указываются:

- адрес сервера PERCo;
- логин;
- пароль или токен;
- часовой пояс предприятия;
- периодичность sync сотрудников;
- периодичность sync проходов;
- допуски до начала и после окончания смен;
- путь проверки `/dev`.

Кнопка `Проверить подключение` вызывает backend endpoint:

```http
POST /api/v1/integrations/perco/test-connection
```

Backend:

1. Читает настройки из `perco_integration_settings`.
2. Расшифровывает сохраненный пароль/токен.
3. Проверяет доступность PERCo.
4. Проверяет `/dev`.
5. Пытается обнаружить рабочие endpoints для сотрудников и проходов.
6. Записывает результат в `perco_integration_logs`.
7. Обновляет статус последней проверки в настройках.

Секреты не возвращаются на frontend. В ответах отображается только статус: есть пароль/токен или нет, читается ли он текущим key-ring.

## Как работает синхронизация сотрудников

Запуск:

```http
POST /api/v1/integrations/perco/sync-employees
```

Логика:

1. Backend обращается к найденному endpoint сотрудников PERCo.
2. Нормализует ФИО, внешний PERCo ID, табельный номер и подразделение.
3. Записывает или обновляет связи в `perco_employee_links`.
4. Не удаляет сотрудников проекта.
5. Не создает дубли по PERCo ID.
6. Несопоставленные сотрудники попадают во вкладку сопоставления.

Сотрудник PERCo и сотрудник проекта связываются через `perco_employee_links`. Это позволяет использовать PERCo как внешний источник проходов, не заменяя основной справочник сотрудников проекта.

## Как работает синхронизация проходов

Запуск:

```http
POST /api/v1/integrations/perco/sync-events
```

Логика:

1. Backend запрашивает проходы из PERCo с последней успешной точки sync.
2. Нормализует событие: сотрудник, время, устройство, направление, зоны входа/выхода.
3. Сохраняет событие в `perco_access_events`.
4. Отсекает дубли по `perco_event_id`.
5. Дополнительно отсекает естественные дубли по ключу: сотрудник + время + направление + устройство.
6. После загрузки перестраивает интервалы присутствия в `employee_presence_intervals`.

Если PERCo возвращает два события на один проход, например `Проход с подтверждением` и `Индикация: проход`, подтвержденный проход имеет приоритет. `Индикация: проход` используется как fallback, если подтвержденного события нет.

## Логика входа и выхода

Направление определяется по зонам PERCo:

```txt
Выход = Неконтролируемая зона, Вход = Завод -> IN
Выход = Завод, Вход = Неконтролируемая зона -> OUT
```

Расшифровка:

- `IN` - сотрудник вошел на завод.
- `OUT` - сотрудник вышел с завода.

Старые или неизвестные события могут оставаться в базе как исторический аудит, но для построения смен и текущего присутствия используются только корректно определенные `IN/OUT`.

## Интервалы присутствия

Интервалы присутствия строятся на основании последовательности событий `IN/OUT`.

Примеры:

```txt
IN 07:30 -> OUT 17:10 = закрытый рабочий интервал
IN 07:30 -> OUT 12:00 -> IN 12:40 -> OUT 17:10 = рабочая смена с обеденным выходом
IN 07:30 без OUT дольше допустимого лимита = спорный/зависший интервал
OUT без предыдущего IN = спорное событие
```

Короткий выход и возврат внутри смены считается обедом или технологическим перерывом. Такой выход не должен автоматически закрывать смену как завершенную.

Спорные случаи попадают в диагностику:

- нет выхода;
- конфликт последовательности;
- слишком длинный интервал;
- событие нельзя надежно восстановить автоматически.

## Связь с ЭМУ

ЭМУ использует PERCo-интервалы присутствия для расчета смен и простоев.

Что учитывается:

- фактический вход сотрудника на завод;
- фактический выход;
- обеденный или технологический выход/возврат;
- ручные корректировки смен.

Правила:

- PERCo помогает понять, был ли сотрудник на территории.
- Обеденный `OUT -> IN` не считается простоем ЭМУ.
- Если сотрудник вышел с завода во время активной работы, система может сформировать решение/уведомление для проверки.
- Ручная корректировка смены имеет приоритет над автоматическим PERCo-расчетом.

## Диагностика и спорные интервалы

Во вкладке `Проходы и смены` отображаются:

- сколько проходов загружено;
- сколько сотрудников сейчас на территории;
- сколько событий не сопоставлено;
- сколько часов на территории;
- сколько интервалов зависло или требует проверки.

Также есть список смен/интервалов:

- сотрудник;
- табельный номер;
- время входа;
- время выхода;
- длительность;
- статус: на территории, вышел, спорный.

Зависшие интервалы можно закрыть вручную, если PERCo не прислал корректный выход. Ручное закрытие должно использоваться только после сверки с журналом охраны или другим источником.

## Текущее состояние

PERCo можно временно считать завершенным по текущему этапу:

- модуль добавлен в интерфейс;
- backend API создан;
- БД и миграции добавлены;
- настройки подключения реализованы;
- проверка `/dev` реализована;
- синхронизация сотрудников реализована;
- синхронизация проходов реализована;
- диагностика входов/выходов реализована;
- спорные интервалы отображаются;
- ручное закрытие интервалов добавлено;
- связь с ЭМУ заложена.

При этом перед финальной приемкой важно проверить, что все PERCo-файлы входят в приемочный scope и не остались `untracked`.

## Что важно помнить

- PERCo не заменяет основной справочник сотрудников проекта.
- Сотрудники PERCo сопоставляются с сотрудниками проекта через `perco_employee_links`.
- Пароль/токен PERCo не хранится и не возвращается в открытом виде.
- API и worker должны иметь общий DataProtection key-ring, иначе worker не сможет расшифровать пароль, сохраненный API.
- Если PERCo UI показывает старые данные, нужно проверить, пересобраны ли `api`, `web`, `worker`.
- Если `/dev` доступен, но sync не работает, нужно смотреть найденные endpoints и журнал `perco_integration_logs`.
- Если в диагностике есть старые `UNKNOWN` события, они могут быть историческими. Для смен должны использоваться корректные `IN/OUT`.

## Что проверять после деплоя

После пересборки и перезапуска контейнеров проверить:

```txt
api
web
worker
```

Проверки в UI:

1. Открыть `http://192.168.2.194:5173/#perco-integration`.
2. Проверить, что модуль открывается без ошибок консоли.
3. На вкладке подключения нажать `Проверить подключение`.
4. Убедиться, что статус подключения актуальный.
5. Запустить синхронизацию сотрудников.
6. Проверить вкладку сопоставления.
7. Запустить синхронизацию проходов.
8. Проверить вкладку `Проходы и смены`.
9. Найти сотрудника с входом и выходом.
10. Проверить, что направление входа/выхода определено правильно.
11. Проверить сотрудника с выходом на обед и возвратом.
12. Убедиться, что обед не считается завершением смены.
13. Проверить, что ЭМУ использует PERCo-присутствие в истории и расчетах смен.

Проверки backend:

```txt
dotnet build --no-restore
dotnet test --no-restore
PATROL360_RUN_DB_INTEGRATION=true dotnet test --no-restore tests/Patrol360.Infrastructure.Tests/Patrol360.Infrastructure.Tests.csproj
```

Если Postgres/Docker недоступен, DB integration tests могут быть пропущены или упасть по внешней причине. Это нужно фиксировать отдельно от кода.
