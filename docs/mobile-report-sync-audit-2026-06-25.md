# Аудит отправки отчетов обхода из мобильного приложения

Дата: 2026-06-25  
Область: мобильное приложение Patrol360, mobile API, outbox, загрузка файлов, web-отображение результатов.

## Краткий вывод

Сценарий, когда первый неотправленный отчет уходит на сервер вместе со вторым отчетом, **возможен** и для offline-first архитектуры в целом является нормальным: мобильное приложение отправляет не только текущий отчет, а всю очередь pending/retry команд.

Это становится дефектом, если:

- пользователь не видит, что первый отчет сохранен локально и ждет отправки;
- пользователь не видит, что сейчас отправляется очередь из нескольких отчетов;
- отчет может быть создан повторно без защиты идемпотентности;
- фото/видео и точки имеют разный статус синхронизации, но UI показывает отчет как полностью отправленный;
- зависшая отправка остается без понятного статуса и ручного восстановления.

По коду сейчас есть сильная база: локальная SQLite, outbox, `clientOperationId`, single-flight sync, восстановление зависших `sending`, reconciliation с сервером и серверная идемпотентность по `ClientOperationId`. Главный риск сейчас не в том, что первый отчет отправился вместе со вторым, а в недостаточно понятной визуализации очереди и в нескольких краевых сценариях повторной отправки с новым `clientOperationId`.

## Ответы на проблемный сценарий

### 1. Возможно ли такое поведение

Да, возможно.

Если первый отчет остался в outbox со статусом `pending`, `retryLater` или был восстановлен из зависшего `sending`, то при отправке второго отчета приложение запускает общую синхронизацию. Эта синхронизация берет пачку команд из очереди и может отправить оба отчета за один проход.

### 2. Почему первый отчет отправился только вместе со вторым

Техническая причина простая: второе нажатие “Отправить отчет” вызвало foreground sync. Sync engine отправляет не “только текущую кнопку”, а всю очередь неотправленных команд. Поэтому старый отчет мог лежать локально, ждать сети/сервера/повтора, а второй отчет просто стал триггером отправки всей очереди.

### 3. Это баг или нормальная логика

Это нормальная логика, если:

- первый отчет был сохранен локально;
- он был в очереди;
- отправился один раз;
- сервер защитился от дублей;
- пользователь видел статус “ожидает отправки”.

Это баг, если:

- пользователь думал, что первый отчет потерян;
- не было понятного статуса очереди;
- повторная отправка могла создать второй отчет;
- файлы и точки синхронизировались частично, но UI показал успех;
- отчет завис без возможности понять ошибку и повторить отправку.

## Локальная модель сейчас

| Область | Текущее поведение |
| --- | --- |
| Черновик | Явного статуса `Draft` для отчета нет; до отправки состояние фактически живет в assignment/point results. |
| Готов к отправке | Вычисляется через readiness: обязательные точки должны быть закрыты. |
| Ожидает отправки | Assignment получает `completedLocal`, outbox-команда имеет `pending` или `retryLater`. |
| Отправляется | Outbox-команда получает `sending`. |
| Отправлено | Outbox-команда получает `accepted`/`duplicate`, assignment становится `completedServer`. |
| Ошибка | Outbox получает `rejected`/`conflict` либо `retryLater`; assignment может стать `syncError`/`needsDispatcherDecision`. |
| Фото/видео | Файлы имеют отдельный upload lifecycle; отчет не отправляется, если обязательные вложения в payload еще не uploaded. |

| Возможная причина | Как проявляется | Это нормально или баг | Риск |

| Первый отчет попал в очередь синхронизации | Sync отправляет его при следующем запуске | Нормально | Низкий |

## Риски дублей

| Объект | Текущая защита | Остаточный риск |
| --- | --- | --- |
| Outbox-команда | `ClientOperationId` является ключом на сервере | Низкий для повторов одной команды |
| Отчет assignment | Клиент не создает вторую complete-команду, если queued command найдена | Средний, если локальная очередь повреждена/очищена и создается новый `clientOperationId` для уже принятого assignment |
| Точки обхода | Сервер пересохраняет point results при complete | Средний: повторный complete с новым operation id может заменить набор результатов |
| Фото/видео | Уникальный `(mobileAccountId, clientFileId)` | Низкий |
| Комментарии | Идут внутри point result | Средний при повторном complete с другим payload |

Ключевой вывод: серверная идемпотентность по `clientOperationId` есть. Для полной защиты нужен второй уровень: идемпотентность результата по `assignmentId`/`clientReportId`, чтобы повторный complete с новым operation id не мог тихо пересоздать результат.

## Риски потери данных

| Риск | Причина | Текущая защита | Что усилить |
| --- | --- | --- | --- |
| Отчет не дошел из-за сети | Нет ответа/timeout | Outbox + retry | Ясный `PendingSync/SyncError` в UI |
| Сервер принял, телефон не понял | Потерян ответ | Reconciliation по `clientOperationId` | Логировать reconciliation result |

| Повторный complete с новым payload | Новый `clientOperationId` | Частично: assignment ownership/required checks | Не удалять/пересоздавать результат без явного режима исправления |

## Правильная модель статусов отчета

Предлагаемая модель для UI и внутренней диагностики:

| Статус | Смысл |
| --- | --- |
| `Draft` | Обход идет, отчет еще не готов к отправке. |
| `ReadyToSend` | Все обязательные точки закрыты, можно отправлять. |
| `PendingSync` | Отчет сохранен локально и ждет отправки. |
| `Syncing` | Отчет или его файлы сейчас отправляются. |
| `PartiallySynced` | Часть данных отправлена, но отчет еще не полностью принят. |
| `Synced` | Сервер подтвердил прием отчета. |
| `SyncError` | Ошибка отправки, можно повторить. |
| `Conflict` | Сервер отклонил из-за изменения/отмены/версии. |
| `NeedManualCheck` | Нужна проверка диспетчера или администратора. |

### Таблица переходов

| Старый статус | Действие | Новый статус | Что логировать |
| --- | --- | --- | --- |
| `Draft` | Все обязательные точки закрыты | `ReadyToSend` | assignmentId, completed points, skipped/manual counts |
| `ReadyToSend` | Нажали “Отправить” | `PendingSync` | timestamp, clientReportId/clientOperationId |
| `PendingSync` | Sync взял команду | `Syncing` | attempt, network state, files count |
| `Syncing` | Часть файлов uploaded, complete еще нет | `PartiallySynced` | uploaded files, pending files |
| `Syncing` | Сервер accepted | `Synced` | serverReportId/assignmentId, response status |
| `Syncing` | Timeout/network error | `PendingSync` или `SyncError` | error code, retryAt, attempt |
| `Syncing` | Сервер вернул duplicate | `Synced` | original operation/server id |
| `Syncing` | Сервер вернул rejected | `SyncError` | rejection reason |
| `Syncing` | Сервер вернул conflict | `Conflict` | conflict reason, server status |
| `Conflict` | Диспетчер разобрал | `NeedManualCheck` или terminal | resolution, operator |
| `PendingSync` | Пользователь нажал “Повторить” | `Syncing` | manual retry flag |

## Как должна работать отправка

### Нормальная сеть

1. Пользователь завершает обход.
2. Приложение сохраняет отчет локально.
3. Приложение создает `clientReportId`/`clientOperationId`.
4. Пользователь нажимает “Отправить”.
5. UI показывает `Syncing`.
6. Приложение загружает фото/видео.
7. Приложение отправляет `completePatrolAssignment`.
8. Сервер возвращает accepted и server id/lock version.
9. Приложение меняет статус на `Synced`.
10. Пользователь видит “Отчет отправлен”.

### Плохая сеть

1. Пользователь нажимает “Отправить”.
2. Отчет сохраняется локально.
3. Сеть/сервер не отвечают.
4. Отчет остается в `PendingSync`/`SyncError`.
5. Пользователь видит: “Отчет сохранен на устройстве и будет отправлен позже”.
6. При появлении сети приложение отправляет очередь.
7. Если второй отчет отправляется позже, приложение может отправить оба отчета одной очередью, но UI должен явно показать “Отправляется 2 отчета”.

## Что логировать

- время нажатия “Отправить”;
- `assignmentId`;
- `requestId`;
- `clientOperationId`;
- будущий `clientReportId`;
- outbox status до/после;
- attempt count;
- network state;
- server health state;
- upload каждого файла: `clientFileId`, size, content type, status;
- response status: `accepted`, `duplicate`, `conflict`, `rejected`;
- reconciliation result;
- terminal local status: `completedServer`, `cancelledServer`, `syncError`;
- текст ошибки для пользователя;
- технический stack/message для диагностики.

## Критерии готовности

Доработка считается готовой, если:

- отчет не теряется без интернета;
- отчет не дублируется при повторной отправке;
- пользователь видит статус отправки;
- пользователь может открыть список неотправленных отчетов;
- пользователь может повторить отправку вручную;
- сервер защищен от дублей;
- фото/видео и точки синхронизируются корректно;
- история синхронизации сохраняется;
- два отчета могут отправиться одной очередью без дублей;
- пользователь видит понятный статус “отчет отправился вместе с очередью”.

## Итог

Поведение “первый отчет отправился вместе со вторым” нужно считать штатным для очереди синхронизации, а не само по себе багом. Багом становится отсутствие ясного состояния для пользователя и недостаточная защита на уровне “один assignment = один принятый отчет”, если команда была создана повторно с новым `clientOperationId`.

Правильное направление: оставить outbox-логику, но сделать ее видимой, диагностируемой и двухуровнево идемпотентной: `clientOperationId` для команды и `clientReportId/assignmentId` для итогового отчета.
