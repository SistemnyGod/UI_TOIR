# Аудит мобильного приложения Patrol360

Дата: 23.07.2026  
Область: Android-клиент, локальное хранилище, подключение к mobile API, восстановление сессии и фоновая синхронизация.

## Итог

Локальный сервер и текущий контур работают штатно: восемь последовательных запросов к `mobile/health` получили HTTP 200 за 16–35 мс, контур сервера — `patrol360-local-enterprise`. Все контейнеры инфраструктуры находятся в состоянии `healthy`.

Проверки проекта также проходят: 58 тестов, TypeScript, lint и проверка кодировки завершились без ошибок.

При этом найдены клиентские риски, которые могут выглядеть для пользователя как потеря авторизации, недоступность сервера или зависание приложения.

## Найденные проблемы

### P1. Холодный запуск всегда переводит пользователя на экран офлайн-доступа

Подтверждено кодом:

- состояние `sessionUnlocked` хранится только в памяти и после запуска равно `false`;
- стартовый маршрут при наличии корректной сохранённой сессии выбирает `/(auth)/offline-login`;
- прямой переход к рабочему экрану предусмотрен типом, но фактически не используется.

Последствие: после выгрузки процесса Android или перезапуска приложения пользователь снова видит экран, похожий на авторизацию. Это происходит даже при исправном сервере и действующей сессии.

Рекомендация: формально определить политику разблокировки. Если повторная проверка устройства после каждого холодного запуска не требуется, восстанавливать рабочую сессию автоматически. Если требуется, переименовать и оформить экран как «Разблокировка приложения», не сообщая, что сервер недоступен.

### P1. Жёстко заданный локальный IP создаёт повторный отказ подключения

Адрес `192.168.2.194` одновременно зафиксирован:

- в конфигурации окружений;
- в разрешённом списке серверов;
- в функции быстрого выбора локального сервера;
- в Android Network Security Config для HTTP.

Последствие: при смене адреса сервера DHCP, переносе сервера или использовании другой Wi-Fi-сети приложение не позволит сохранить новый адрес, а Android может заблокировать HTTP-соединение с ним. Для исправления потребуется новая сборка.

Рекомендация: использовать стабильный локальный DNS/статический IP и HTTPS. До этого — передавать адрес разрешённого сервера параметром сборки, не дублируя его в нескольких файлах.

Статус после аудита: локальный адрес подтверждён и сохранён основным; добавлен параметр сборки `PATROL360_PUBLIC_API_URL` для резервного белого IP. Конкретный внешний адрес пока не задан.

### P1. Ошибка чтения SecureStore может оставить бесконечный индикатор загрузки

В стартовом маршруте и экране офлайн-доступа цепочки чтения SecureStore/локального профиля не имеют финального обработчика ошибки.

Последствие: при временной недоступности защищённого хранилища приложение не покажет понятную ошибку и останется на индикаторе загрузки. В `SessionGuard` аналогичная ошибка трактуется как отсутствие сессии и переводит пользователя на обычный вход.

Рекомендация: ввести единое состояние `loading / ready / recoverableError`, повтор чтения и отдельное сообщение о недоступности локальной сессии без удаления токенов.

### P2. Проверка целостности всей SQLite выполняется при открытии каждой транзакции

Каждая защищённая транзакция открывает новое соединение, а общий метод открытия выполняет `PRAGMA integrity_check`.

Последствие: по мере роста базы каждая запись, синхронизация или сохранение уведомлений может становиться заметно медленнее, увеличивать расход батареи и создавать впечатление зависания.

Рекомендация: проверять ключ при каждом соединении лёгким запросом, а полный `integrity_check` выполнять при запуске, после миграции или по отдельной диагностической команде.

### P2. Не все конкурирующие операции БД используют повтор при `SQLITE_BUSY`

Для основных репозиториев добавлен повтор транзакции, но сохранение уведомлений и несколько операций outbox вызывают защищённую транзакцию напрямую. Одновременно приложение запускает обновление данных, отправку очереди и обработку уведомлений.

Последствие: при совпадении фонового обновления, возврата приложения на экран и отправки отчёта возможен редкий `database is locked`. Тайм-аут 5 секунд снижает вероятность, но не гарантирует восстановление.

Рекомендация: применить единый retry/serialization слой ко всем пишущим операциям либо разделить одну общую очередь записи.

### P2. Health-проверка выполняется перед каждым API-запросом

`mobileRequest` сначала запрашивает `/mobile/health`, затем выполняет полезный запрос. Синхронизация дополнительно вызывает отдельную проверку сервера до отправки очереди.

Последствие: удваивается число сетевых запросов и появляется дополнительная точка отказа. Кратковременная проблема только health-маршрута блокирует рабочий API; при недоступном сервере ожидание может складываться из нескольких тайм-аутов.

Рекомендация: использовать health для ручной диагностики, выбора контура и периодического контроля, но не как обязательный шлюз перед каждым запросом.

### P2. Экран офлайн-доступа сообщает о недоступном сервере даже при обычном запуске

Текст экрана утверждает, что сервер временно недоступен, хотя маршрут открывается при любой корректной сохранённой сессии после холодного старта.

Последствие: пользователь считает, что подключение снова сломалось, хотя соединение может быть исправно.

Рекомендация: разделить состояния «разблокировка сохранённой сессии» и «сервер действительно недоступен».

### P3. Сетевые ошибки теряют техническую причину

Тайм-аут, DNS, отказ соединения и Android Network Security преобразуются в одно сообщение «Сервер недоступен».

Последствие: пользователю сообщение понятно, но диагностике трудно отличить неправильный IP, блокировку HTTP, тайм-аут и отсутствие маршрута.

Рекомендация: сохранять безопасный технический код причины в локальном диагностическом журнале, не показывая сетевые детали обычному пользователю.

## Исправлено 23.07.2026

- Холодный запуск с действующей сохранённой сессией больше не ведёт на экран офлайн-авторизации: приложение возобновляет рабочий контур.
- Ошибки SecureStore/локального профиля больше не оставляют бесконечный индикатор и не удаляют данные автоматически; добавлена повторная попытка.
- Экран офлайн-доступа переименован по смыслу в разблокировку приложения и больше не заявляет о недоступности сервера без проверки.
- Полный `integrity_check` исключён из каждой транзакции; добавлен retry для уведомлений и ключевых outbox-операций при блокировке SQLite.

Остаются для следующей итерации: сокращение обязательных health-запросов перед каждым API-вызовом, детализация сетевых причин в диагностике и проверка поведения на реальном Android-устройстве.
## Что уже защищено корректно

- Неизвестный HTTP 401 не удаляет локальную сессию автоматически.
- Повторный вход требуется только для явных серверных кодов отзыва, смены устройства, владельца или истечения refresh-сессии.
- Refresh-сессия сервера действует 180 дней, access token — 8 часов.
- Сервер поддерживает 60-секундное безопасное повторение refresh-запроса.
- Локальные отчёты и очередь отправки сохраняются при временной недоступности сети.
- Контур клиента проверяется по health, заголовкам и ответу авторизации.
- Старый фиктивный production-домен больше не является значением по умолчанию в исходном коде.
- Ошибка SQLCipher `file is not a database`, вызванная незашифрованным соединением транзакции, устранена ранее и покрыта тестом политики шифрования.

## Выполненные проверки

- `npm run verify`: 58/58 тестов, encoding, TypeScript и lint — успешно.
- 8 последовательных health-запросов к локальному API — успешно.
- Проверка Docker: proxy, API, worker, PostgreSQL, Redis, RabbitMQ и MinIO работают; критические контейнеры healthy.
- Проверка ADB: Android SDK найден, но подключённых устройств или эмуляторов нет.

## Ограничение аудита

Инструментальная проверка на реальном APK не выполнялась: устройство не подключено, а сборка APK по текущему заданию не выполнялась. Поэтому поведение при убийстве процесса Android, смене Wi-Fi, блокировке экрана и работе фоновых задач подтверждено анализом кода, но ещё требует отдельного прогона на телефоне.

## Рекомендуемый порядок исправления

1. Исправить восстановление/разблокировку сессии и бесконечные загрузки.
2. Убрать жёсткую зависимость от одного LAN-IP.
3. Отделить полный `integrity_check` от каждой транзакции.
4. Унифицировать retry/очередь записей SQLite.
5. Сократить обязательные health-запросы.
6. Добавить диагностические коды сетевых причин.


## Follow-up fix — 23.07.2026

- Normal JSON requests, token refresh and file uploads now reuse a short-lived health probe per server/contour (30 seconds on success, 3 seconds on failure).
- Concurrent requests to the same address share one health probe; a failed request invalidates the address cache so failover can recover immediately.
- Explicit user-initiated connection checks remain uncached and always validate the current contour.
- Network failures now retain a typed `timeout`/`network` cause for diagnostics while preserving the existing user-facing message.
- Added three focused cache tests; `npm run verify` passes with 64 tests.

APK was not built and no Android device was connected for runtime validation.
