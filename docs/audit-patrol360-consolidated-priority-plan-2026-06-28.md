# Сводный аудит Patrol360 и приоритетный план, 2026-06-28

## 1. Краткий вывод

Главная проблема сейчас не в отсутствии направления, а в большом количестве параллельных незакрытых хвостов. Проект уже имеет рабочую базу: backend-сервисы частично разрезаны, СИЗ-печать стабилизирована первым проходом, mobile outbox имеет защиту от дублей, web-проверки проходят. Но эксплуатационно проект еще нельзя считать закрытым, потому что остаются P0/P1 риски в интерфейсе бухгалтерии, сценариях под запись, мобильном runtime/APK, кодировке, истории движений и API-gate.

Рабочий порядок должен быть таким:

1. Сначала закрыть дефекты, которые мешают пользователю провести операцию или понять состояние.
2. Потом закрыть сценарии учета: выдача, под запись, возврат, списание, история.
3. Потом провести mobile APK/runtime smoke.
4. После этого делать структурный рефакторинг и OpenAPI/security hardening.
5. Большие идеи и миграции делать только после стабилизации текущих экранов.

## 4. P1 - следующий рабочий слой

### 4.1. Бухгалтерия -> Выдача

| Задача | Решение | Не делать |
| --- | --- | --- |
| Сделать карточки номенклатуры активными | Клик по карточке добавляет/выбирает предмет. Отдельные кнопки `Добавить` убрать | Не оставлять две конкурирующие зоны клика |
| Исправить поиск | Один компактный search field, без растяжения и наложений | Не делать несколько поисковиков на одном табе |
| Исправить период в истории/черновиках | Компактный выбор периода: `с` / `по` в одной группе или preset `Сегодня/Неделя/Месяц` | Не плодить отдельные несвязанные date inputs |
| Черновик выдачи | Слева предмет, центр количество/единица/цена/сумма, справа параметры/удалить | Не прятать ключевые действия под строку |
| Проведение выдачи | После проведения строка уходит в историю движения сотрудника | Не считать это складским списанием |

### 4.2. Бухгалтерия -> Под запись

| Задача | Решение |
| --- | --- |
| Удобная модалка выдачи под запись | Модалка: сотрудник -> предметы -> количество -> комментарий -> провести |
| Поиск предметов | Быстрый поиск по названию, артикулу, группе |
| Группы/колонки предметов | Справочник групп: `Рации`, `Инструменты`, `Ключи`, `Прочее`; каждая группа может быть отдельной колонкой/фильтром |
| Возврат | Показывать только предметы, которые реально находятся на руках у сотрудника |
| Списание | Доступно только для предметов на руках или уже возвращенных с дефектом |
| Неисправность | Добавить статус `Неисправно/Поломка`, комментарий и историю |

### 4.3. Бухгалтерия -> Возврат и списание

| Задача | Решение |
| --- | --- |
| Выбор сотрудника | Сначала выбрать сотрудника, затем увидеть список выданных предметов |
| Список предметов | Показывать предмет, дату выдачи, количество, статус, комментарий |
| Возврат | Фиксировать движение `возвращено`, кто сделал, когда, комментарий |
| Списание | Фиксировать движение `списано`, причина, акт/комментарий |
| Нет складского остатка | Считать агрегаты движения: всего выдано, на руках, возвращено, списано, неисправно |

## 6. Что исключить из ближайшего плана

| Исключить | Почему |
| --- | --- |
| Большой rewrite проекта | Рабочая архитектура уже есть; нужен controlled refactor |
| Микросервисы | Сейчас добавят сложность без пользы |
| Перенос mobile в `apps/mobile` до APK smoke | Есть риск сломать обновление поверх старого APK и SQLite/outbox |
| Прямой перенос старого СИЗ-проекта | Нужны правила, а не старый UI/код |
| Полный складской учет остатков в текущем сценарии выдачи | Пользователь прямо уточнил, что склада нет; нужен учет движений |
| Автоматическое списание по складу | Не соответствует текущей модели: фиксируем выдачу/возврат/списание как историю |
| SignalR/live updates | Полезно позже, но сначала стабилизировать текущие экраны и API |
| Materialized views/read models | Только после реальных замеров производительности |

## 7. Рекомендуемый порядок ближайших проходов

### Проход 1: бухгалтерия, выдача

1. Убрать кнопки `Добавить` из карточек предметов.
2. Сделать карточку предмета кликабельной целиком.
3. Исправить строку черновика: сумма, цена, количество, actions.
4. Убрать остатки/наличие/склад из UI.
5. Проверить проведение выдачи и запись в историю движения.

### Проход 2: под запись

1. Сделать модалку выдачи под запись.
2. Добавить поиск предметов и сотрудника.
3. Добавить группы предметов и фильтры.
4. Добавить возврат, списание, неисправность.
5. Проверить историю движения по сотруднику.

### Проход 3: СИЗ

1. Перерисовать карточки журнала СИЗ.
2. Упростить правый инспектор.
3. Проверить данные сотрудника в форме и DOCX.
4. Добить периодичность в UI.
5. Повторить DOCX tests.

### Проход 4: mobile release gate

1. `npm run typecheck` и `npm run lint` в mobile.
2. Собрать APK.
3. Обновить поверх старого APK.
4. Проверить offline report, media, reconnect, sync queue.
5. Проверить 401/authRequired без удаления outbox.

### Проход 5: patrol web

1. `#results` строки, фильтры, modal/media.
2. `#assign` события/время/активные назначения.
3. Browser smoke.

### Проход 6: структура и API

1. Разрезать оставшиеся крупные UI/CSS-файлы.
2. Добавить structure tests.
3. OpenAPI/contract drift.
4. Auth/security policies.
5. Полный DB/API gate.

## 9. Самые важные 10 задач

1. Исправить реальные mojibake-строки в user-facing экранах и audit/docs, которые используются дальше.
2. Добить `#inventory-issue`: карточки кликабельны, нет кнопок добавления, сумма вводится, период компактный.
3. Убрать склад/остатки из текущей логики выдачи и оставить учет движения.

## 10. Рабочее замечание

В рабочем дереве уже есть много dirty/untracked изменений по web/backend/mobile/docs. При следующих проходах нельзя чистить или откатывать дерево целиком. Каждый проход должен менять только свой модуль и после этого запускать локальный gate.
