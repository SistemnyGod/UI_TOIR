# Mobile Offline Recovery

Дата: 2026-06-25

Документ описывает устойчивую отправку отчетов обхода из мобильного приложения Patrol360.

## Базовая модель

- Все действия сначала сохраняются локально в SQLite.
- Отправка идет через `outbox_commands`.
- Каждый отчет и команда имеют `clientOperationId`.
- Повторная отправка того же `clientOperationId` не должна создавать дубль на сервере.
- Фото и видео загружаются до финального успешного завершения отчета.
- При `401/403` после неуспешного refresh токены очищаются, но SQLite/outbox не удаляются.

## Статусы отчета

| Статус | Значение |
| --- | --- |
| `Draft` | Данные еще редактируются. |
| `ReadyToSend` | Все обязательные точки заполнены. |
| `PendingSync` | Отчет сохранен и ждет отправки. |
| `Syncing` | Отчет отправляется. |
| `PartiallySynced` | Часть данных уже отправлена, например файлы еще в очереди. |
| `Synced` | Сервер подтвердил отчет. |
| `SyncError` | Временная или окончательная ошибка отправки. |
| `Conflict` | Серверное состояние конфликтует с локальным. |
| `NeedManualCheck` | Нужна ручная проверка диспетчером. |

## Ключевые сценарии

### Сервер принял отчет, телефон не получил ответ

1. Команда остается в outbox как `retryLater` или зависшая `sending`.
2. Recovery возвращает зависшую команду в повтор.
3. Повторная отправка приходит на сервер с тем же `clientOperationId`.
4. Сервер возвращает `duplicate`/уже созданный результат.
5. Телефон закрывает локальный отчет как `completedServer`.

### Offline отчет готов, затем сервер отменил заявку

Цель: задача не должна висеть у сотрудника вечно.

1. Мобильное приложение не удаляет локальный отчет.
2. При sync сервер либо принимает поздний результат, либо фиксирует отмененный/неполный результат в истории.
3. Мобильная задача переходит в терминальный статус и исчезает из активной работы.
4. Пользователь видит понятное сообщение: отчет доставлен, отменен сервером или требует повторной отправки.

### Фото/видео не загрузились

1. Файл остается в локальной очереди со статусом `failed` или `retryLater`.
2. Отчет не считается полностью успешным, пока нужные вложения не загружены.
3. Экран "Не отправлено" показывает файл, ошибку и кнопку повтора.

### Точки готовы, файлы еще нет

Статус отчета должен быть `PartiallySynced` или `PendingSync`, не `Synced`.

### Пользователь вышел из аккаунта с неотправленными данными

Выход или смена пользователя блокируются, пока есть pending/sending/retryLater outbox-команды. Если сессия истекла сама, данные остаются в SQLite, а экран показывает `Нужно войти снова`.

### Старый APK

Legacy-команда `takePatrolRequest` сохраняется. Она означает "принять и сразу начать обход". Новые команды `accept/start/pause/resume/release/handoff` добавляются совместимо и не ломают старый APK.

## UX-правила

- Пользователь должен видеть, что отчет сохранен на устройстве.
- Повторное нажатие "Отправить" не создает новую команду, а открывает статус существующей отправки.
- Экран "Не отправлено" показывает отчеты, файлы, попытки, последнюю ошибку и время последнего обновления.
- Если отправка двух отчетов произошла одной синхронизацией, это нормальная логика очереди, но UI должен показать оба отчета и их финальный статус.

## Серверные гарантии

- `clientOperationId` обязателен.
- Повторный `completePatrolAssignment` с тем же payload возвращает `duplicate`.
- Повторный `completePatrolAssignment` с тем же `clientOperationId`, но другим payload возвращает `conflict`.
- `ok`, `issue`, `skipped` закрывают обязательные точки.
- `pending` и `deferred` по обязательным точкам отклоняются.
- `confirmationType = manual` сохраняется для web-отчета как "Заполнено вручную без сканирования".
