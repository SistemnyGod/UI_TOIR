1. Назначение проекта

Patrol360 - веб-система для управления обходами территории, маршрутами,
точками контроля NFC/QR, заявками, назначениями сотрудников, мобильными
аккаунтами и результатами обходов.

В проект также интегрируется модуль "Бухгалтерия" / Web-инвентарь:
номенклатура, складские остатки, выдача, возвраты, списания, СИЗ,
учет "под запись", история, отчеты, настройки справочников и импорт
legacy-данных из старой Inventory-системы.

Отдельно добавлен модуль ЭМУ / "Учет работ": дашборд ЭМУ, учет текущих
работ, план работ, избранные сотрудники, справочники участков и причин,
история выполненных работ, аудит и перенос незавершенных работ через worker.

7. Reverse proxy и HTTPS

Файл:
- infra/docker/Caddyfile

Caddy:
- Отключен автоматический HTTPS:
  auto_https off
- HTTP :80 редиректит на HTTPS.
- HTTPS :443 использует локальный сертификат.
- /api/* -> api:8080.
- /health/* -> api:8080.
- все остальное -> web:80.

Локальные URL:
- https://localhost
- https://192.168.2.194
- https://192.168.2.194:5173
- API через proxy:
  - https://192.168.2.194/api/...
  - https://192.168.2.194/health/ready

11. Модули "Обход"

Маршруты и точки:
- Создание маршрута.
- Создание точек внутри маршрута.
- Перемещение точек и сохранение порядка.
- Типы контроля:
  - NFC
  - QR
  - Ручной
- NFC/QR-код хранится как код точки.
- Для дублей NFC внутри маршрута нужен UI warning.

Назначения:
- Выбор сотрудника.
- Выбор маршрута.
- Плановое время.
- Смена.
- Комментарий/текст уведомления.
- Создание заявки на обход.
- Backend создает связанное назначение.
- notifyEmployee=true сейчас сохраняет подготовленное уведомление;
  мобильная push-доставка относится к следующему этапу.

Плановый обход:
- Недельный и месячный режим.
- Дата-якорь.
- Фильтр смены.
- Создание плановой ячейки через заявку/назначение.

Мобильные аккаунты:
- Создание аккаунта.
- Редактирование.
- Блокировка/разблокировка.
- Сброс пароля.
- Удаление.
- Привязка 1-5 сотрудников.

Дашборд:
- Реальные counts из API:
  - активные обходы.
  - задержки.
  - точки.
  - сотрудники онлайн.
  - покрытие смен.

14. Печать и экспорт

Текущий целевой подход:
- Server-side file generation.
- Для v1 допускается синхронная генерация файлов.
- export_jobs используется как аудит выгрузок.
- Для предпросмотра основным форматом является PDF.
- Пользователь открывает документ в preview-модалке, проверяет содержимое
  и затем печатает или скачивает файл.

Форматы:
- PDF.
- DOCX.
- XLSX.

Рекомендуемые технологии:
- PDF: QuestPDF или аналогичный .NET PDF generator.
- DOCX: DocumentFormat.OpenXml.
- XLSX: ClosedXML.
- Frontend preview: Blob URL + iframe/object в React-модалке.
- Browser print: печать открытого PDF preview.

Приоритетные печатные формы:
- СИЗ-карточка.
- СИЗ-лист подписи.
- Акт "Под запись".
- Маршрутный лист обхода.
- Результат обхода.
- Заявка/назначение обхода.
- Отчет ЭМУ по выполненным работам.
- Табель/история работ ЭМУ.

Важно:
- В текущих csproj не найдены отдельные NuGet-зависимости ClosedXML,
  DocumentFormat.OpenXml или QuestPDF. Если понадобятся полноценные красивые
  файлы промышленного качества, эти пакеты нужно добавить явно.

Минимальный API-паттерн для печати:
- GET .../print?format=pdf - получить PDF для предпросмотра и печати.
- GET .../print?format=docx - скачать DOCX.
- GET .../export?format=xlsx - скачать Excel-отчет.
- POST .../export - создать audit-запись export_jobs, если нужен журнал выгрузок.

Минимальный frontend-паттерн:
- Кнопка "Предпросмотр" загружает PDF Blob.
- PDF открывается в модальном окне или drawer.
- Кнопка "Печать" печатает PDF preview.
- Кнопка "Скачать PDF/DOCX/XLSX" скачивает файл с именем из Content-Disposition.
